## 0. CTO 前言：这不是一次"换皮"

在进入具体方案之前，我需要先明确这次优化的**真正问题**和**战略价值**。

### 0.1 当前问题诊断

| 维度 | 现状 | 问题 |
|------|------|------|
| **信息架构** | 官网（Marketing）与产品（Product）混在同一个 Shell 和导航里 | 用户无法在 3 秒内判断"这是一个什么产品"还是"这是一个我要使用的工具" |
| **首屏叙事** | 首屏是 Vision Coverage Panorama——10 个赛道网格 | 对外感知像"项目管理看板"，而非"一个有愿景的产品" |
| **视觉品质** | 基础暗色主题、功能式卡片布局、无动效、无品牌视觉记忆点 | 与竞品 / 参考站（chekusu.com 等）的视觉差距明显 |
| **产品入口** | 产品功能（Ask、Tasks、Agents、Approvals 等）作为导航平级展示 | 新用户看到一堆功能链接但不知道从哪开始；与"IM 聊天窗口"的核心产品形态严重脱节 |
| **转化路径** | CTA 分散（Start Building / Book a Demo / Join Waitlist） | 缺少主转化路径的收敛：visitor → 理解愿景 → 进入产品 → 使用 |
| **开发者入口** | Developers / Docs 在导航中但与产品页混在一起 | 开发者和终端用户的入口路径交叉，体验互相干扰 |

### 0.2 这次优化的核心目标

1. **建立清晰的双区架构**：官网区（Marketing Zone）讲愿景和路线图，产品区（Product Zone）承载真实使用。
2. **让首屏成为"30 秒内理解 GaiaLynk 是什么"的唯一入口**——不是 10 个卡片，而是一个有力量的叙事。
3. **将产品的核心交互形态锚定为 IM 聊天窗口**——所有产品功能围绕这个中心展开，而不是分散在若干独立页面。
4. **用视觉品质建立品牌信任**——在 Agentic 基础设施赛道中，第一印象直接影响开发者和企业的信任判断。

### 0.3 回撤成本评估

这次优化涉及前端架构调整，但**不触及后端 API 和主线核心服务**，属于**可逆 / 中等代价**决策。如果某个方向验证失败，可以按模块回滚。因此建议**快速执行、分阶段验证**。

## 1. 架构重构：双区模型落地

### 1.1 核心结构

```
gaialynk-a2a.vercel.app（或后续自定义域名）
│
├── /[locale]/                     ← 官网区（Marketing Zone）
│   ├── /                          ← 首页：愿景 + 路线图 + CTA
│   ├── /roadmap                   ← 完整路线图（可交互时间线）
│   ├── /use-cases                 ← 场景/用例展示
│   ├── /developers                ← 开发者社区入口（文档 / SDK / API Reference / 快速开始）
│   │   ├── /developers/quickstart ← 快速开始指南
│   │   ├── /developers/sdk        ← SDK & API Reference
│   │   └── /developers/protocol   ← 协议文档
│   ├── /node-collaboration        ← 节点协作说明页（Coming Soon，Phase 2+ 上线）
│   ├── /pricing                   ← 定价 / 免费额度说明
│   ├── /about                     ← 关于我们 / 团队
│   ├── /demo                      ← Book a Demo（极简表单，可嵌入 Typeform/Tally）
│   ├── /privacy                   ← 隐私政策
│   └── /cookies                   ← Cookie 政策
│
├── /[locale]/app/                 ← 产品区（Product Zone）
│   ├── /                          ← 产品主界面（IM 聊天窗口 + 侧边导航）
│   │                                 ※ 浏览免登录，开始使用时触发登录
│   ├── /login                     ← 登录 / 注册
│   ├── /onboarding/provider       ← Provider 引导：接入已有 Agent 到平台
│   ├── /onboarding/consumer       ← Consumer 引导：学会调用平台上的 Agent
│   └── (内部路由由聊天界面内部状态管理)
│       ├── 会话列表
│       ├── 聊天窗口（Ask / 任务执行）
│       ├── Agent 目录（未登录也可浏览）
│       ├── 审批队列
│       ├── 历史记录
│       ├── 连接器管理
│       └── 设置
│
└── /api/                          ← API 层（不变）
```

### 1.2 关键设计原则

| 原则 | 说明 |
|------|------|
| **官网区零登录** | 官网区所有内容无需登录即可访问，降低了解门槛 |
| **产品区浏览免登录** | 进入 `/app` 无需登录，可自由浏览 Agent 目录、查看产品形态；**开始使用时**（发起对话、调用 Agent、创建任务等写操作）触发登录 |
| **两类用户** | **Provider**（Agent 提供者，同时也是使用者）和 **Consumer**（仅使用者）。产品区和 Onboarding 流程需同时服务这两类角色 |
| **导航分离** | 官网区和产品区使用完全不同的 Shell / Layout / 导航组件 |
| **共享品牌** | 同一品牌视觉系统（色彩、字体、Logo），但 Layout 不同 |
| **官网到产品的桥接** | 官网区右上角始终有"进入产品 / Open App"按钮作为核心 CTA |

### 1.3 与 `Agent-IM-Product-Plan.md` §12.1A 的对齐

本方案完全遵循 §12.1A 已定义的双区模型：
- Phase 0/1：同域不同路由（`/` 官网 + `/app` 产品区）
- Phase 2+：可演进为子域（`www.xxx.com` + `app.xxx.com`）
- 禁止项全部遵守：不把控制台做成 marketing 二级页面

## 2. 官网区首页重设计

### 2.1 设计哲学

综合参考两个优秀站点：

**chekusu.com 的成功要素（官网叙事层）**：
- **强烈的 Hero 区域**：一句话让人知道你是谁、做什么
- **哲学/愿景叙事**：不只展示功能，而是讲清"为什么这件事重要"
- **产品矩阵/路线图**：让人知道"你要走向哪里"
- **社会证明/愿景引用**：增加可信感

**alma.now 的成功要素（产品展示层）**：
- **极简聚焦**：导航只有 Features / Providers / Download / Docs / Discord / Feedback，六个入口，没有多余
- **特性展示 = 截图 + 一句话**：每个功能用真实界面截图/Mockup + 短标题 + 一段话，比纯文字描述有说服力 10 倍
- **Chat Interface 作为核心展示**：作为聊天产品，它把聊天界面直接放在首页展示——GaiaLynk 同样应该把 IM 聊天窗口的界面截图/动画放在首页
- **Provider 生态展示**：以 Logo 网格展示支持的 AI Provider——GaiaLynk 可以类比展示"支持的 Agent 框架"或"已接入的 Agent 生态"
- **清晰的 CTA**：Download 作为唯一核心 CTA，干净利落——GaiaLynk 的 "Open App" 应该同样突出

但**不照搬** chekusu.com 或 alma.now——GaiaLynk 的定位是「可信 Agent 准入与协作层」，叙事调性应更偏**基础设施级 Agent 准入治理**，而非"酷炫科技公司"或"优雅桌面工具"。我们要做的是**chekusu 的愿景叙事力 + alma 的产品展示力 + 我们独有的 Agent 准入信任体系**。

## 3. 产品区重设计：以 IM 聊天窗口为核心

### 3.1 设计哲学

> 我们的产品定位是"Agent IM"——即 Agent 即时协作平台。那么产品的核心交互形态就**必须**是 IM 聊天窗口。所有功能应围绕"会话"这个中心展开，而不是分散为独立页面。

**参照物**：Slack / Discord / 微信 / Telegram 的 IM 布局——左侧列表、中间聊天、右侧详情。

### 3.3 侧边栏结构

侧边栏分为两个区域：

**上半区：会话列表**
- 新建对话按钮（触发路由选择或直接开始对话）
- 最近对话列表（按时间倒排）
- 每条对话显示：标题 / 摘要、参与的 Agent(s)、最后活跃时间、未读指示

**下半区：功能入口**
- 📋 **任务中心**：临时任务 + 订阅任务统一管理
- 🤖 **Agent 目录**：浏览、搜索、筛选可调用的 Agent
- ✅ **审批队列**：待审核的高风险操作（有权限时显示）
- 📊 **历史记录**：全部执行历史与审计回放
- 🔗 **连接器**：本地连接器状态与权限管理
- ⚙️ **设置**：账号、空间、安全、计费

### 3.4 核心交互：聊天窗口

聊天窗口是产品的**绝对核心**，所有用户操作最终都收敛到"在对话中完成"。

| 场景 | 在聊天窗口中的体现 |
|------|-------------------|
| **即时 Ask** | 用户在输入框输入需求 → 平台路由 Agent → Agent 在对话中返回结果 |
| **多 Agent 协作** | 多个 Agent 在同一会话中分工回复，用户可 @ 指定 Agent |
| **风险确认** | 高风险操作以**内嵌卡片**形式出现在对话流中，用户在卡片上操作确认/拒绝 |
| **审批** | 审批请求以**消息卡片**推送到对话中或审批队列 |
| **订阅任务** | 订阅任务的执行结果以**定时消息**推送到对应对话 |
| **执行进度** | Agent 执行中显示进度条 / 实时流式输出 |
| **收据与审计** | 每条关键操作后，对话中附带"查看收据"/"查看执行链路"的可展开面板 |

### 3.5 右侧上下文面板

右侧面板根据当前选中对象**动态切换内容**：

| 当前焦点 | 面板内容 |
|----------|---------|
| 一条对话 | 参与者列表、会话拓扑（T1-T5）、授权范围、会话级别策略 |
| 一个 Agent | Agent 详情、身份验证状态、能力声明、信誉评分、历史成功率、风险等级、费用估算 |
| 一条审批 | 审批详情、触发原因、调用链摘要、影响范围、审批历史 |
| 一条收据 | 收据详情、签名验证、关联审计事件 |

面板可折叠（移动端默认收起）。

### 3.6 两类用户与鉴权策略

**平台用户分为两类**：

| 用户类型 | 定义 | 典型角色 |
|----------|------|---------|
| **Provider** | Agent 提供者，同时也是平台使用者。拥有 Agent 并希望接入平台供他人调用 | 开发者、AI 服务商、企业内部 Agent 运维方 |
| **Consumer** | 仅使用平台上 Agent 的用户 | 普通用户、企业员工、业务人员 |

**鉴权策略**：

| 区域 / 操作 | 是否需要登录 | 说明 |
|-------------|:----------:|------|
| 官网区所有页面 | ✗ | 全部公开，零门槛 |
| 产品区浏览（Agent 目录、产品形态展示） | ✗ | 让用户先看到价值，降低使用门槛 |
| 产品区写操作（发起对话、调用 Agent、创建任务、审批等） | ✓ | 触发时弹出登录/注册，支持邮箱 + OAuth（GitHub / Google） |
| 产品区 Provider 功能（接入 Agent、查看运维面板） | ✓ | 需登录且完成 Provider 身份验证 |

### 3.7 新用户 Onboarding（双路径）

产品区的 Onboarding 需要**同时服务两类用户**，通过选择分流：

**入口**：用户首次触发登录后，进入角色选择页 → 分流到对应引导路径。

**路径一：Consumer 引导（"我想调用 Agent"）**

1. 欢迎页：一张图说清"在这里你可以让 Agent 帮你完成任务"
2. **浏览 Agent 目录**：引导用户看到平台上已有的 Agent，了解能做什么
3. **发起第一次对话**：选一个推荐 Agent，输入需求（预置建议话术）
4. **体验首次结果**：平台路由到 Agent，返回结果 + 收据
5. 完成引导，进入主界面

目标：**3 分钟内获得首次可见结果（TTFR）**。

**路径二：Provider 引导（"我想接入我的 Agent"）**

1. 欢迎页：一张图说清"把你的 Agent 接入 GaiaLynk，让全网用户都能调用"
2. **填写 Agent 基本信息**：名称、能力声明、A2A 端点地址
3. **连通性检查**：平台自动向 Agent 发送健康检查请求
4. **首次测试调用**：Provider 自己作为用户调用一次刚接入的 Agent，验证端到端链路
5. **提交上架审核**（或直接可用，取决于准入策略）
6. 完成引导，进入主界面（Provider 同时获得运维面板入口）

目标：**10 分钟内完成 Agent 接入并通过首次测试调用**。

> **注**：Provider 引导的核心步骤（接入 Agent 的快速开始）也应在官网开发者生态区（`/developers`）有所展示，作为吸引 Provider 注册的关键内容。第五屏"开发者生态入口 + 社区"中的"快速开始 / 5 分钟接入首个 Agent"即为此入口。

## 4. 视觉设计升级

### 4.1 执行策略

建议官网开发团队使用 **impeccable.style** 进行视觉重构。具体 skill 使用路径：

| 步骤 | 使用的 skill | 目的 |
|------|-------------|------|
| 1 | `get_guidelines(topic=landing-page)` | 获取 landing page 设计规范 |
| 2 | `get_guidelines(topic=web-app)` | 获取 Web App（产品区）设计规范 |
| 3 | `get_style_guide_tags` → `get_style_guide(tags)` | 获取适合 Agentic 基础设施赛道的风格指南 |
| 4 | `critique` | 对当前页面做设计审查 |
| 5 | `colorize` + `typeset` | 建立统一色彩和字体系统 |
| 6 | `animate` | 添加品牌动效和交互微动效 |
| 7 | `arrange` + `clarify` | 优化布局和信息层级 |

### 4.2 视觉方向建议

| 维度 | 当前 | 目标 |
|------|------|------|
| **色彩** | 基础暗色主题 + 淡蓝高亮 | 深空色主题 + 品牌主色（建议冷调蓝绿渐变） + 高对比度 accent |
| **字体** | 系统默认 | 引入品牌字体（如 Inter + 展示用字体），中文使用高品质字体 |
| **动效** | 无 | Hero 背景网格动画、卡片入场 stagger、scroll-triggered reveal、hover 微交互 |
| **间距** | 偏紧凑 | 更大的呼吸空间，特别是 Hero 区域和核心价值主张区域 |
| **图标/插图** | 无 | 为核心概念设计品牌图标（Trust、Evidence、Policy 等） |
| **组件风格** | 基础 border + bg-card | 毛玻璃效果、渐变边框、品牌阴影、更有质感的卡片 |

### 4.3 产品区视觉

产品区（IM 界面）与官网区共享品牌色系，但视觉密度更高：
- 侧边栏：深色背景、紧凑列表
- 聊天区：适当留白但信息密度高
- 右侧面板：信息清晰分区、数据可视化
- 整体参考：**alma.now**（聊天界面的优雅感） + **Slack**（IM 布局成熟度） + **Linear**（信息密度 + 美感平衡）+ **Vercel Dashboard**（工程师审美）

### 4.4 官网特性展示方式（借鉴 alma.now）

alma.now 的特性展示模式非常有效：每个功能 = **真实界面截图 + 短标题 + 一段描述**。GaiaLynk 官网应采用类似模式：

| 功能区块 | 标题 | 截图/Mockup 内容 | 描述 |
|----------|------|-----------------|------|
| **智能对话** | "Beautiful Chat Interface" 的 GaiaLynk 版本 | IM 聊天窗口：用户与多个经过验证的 Agent 的真实对话 | 在一个对话中与多个可信 Agent 协作... |
| **Agent 准入** | Verified Agents, Trust as Policy | Agent 信誉面板 + 风险确认卡片 + 审批流程界面 | 每个 Agent 经过身份验证和信誉评估，恶意 Agent 被挡在门外... |
| **Agent 市场** | Agent Marketplace | Agent 目录界面截图 | 浏览专业 Agent，一键启用... |
| **自动化任务** | Recurring Automation | 订阅任务管理界面 | 一次设定，持续执行... |
| **执行链路** | Evidence by Default | 审计时间线 + 收据验证界面 | 完整可追溯的执行记录... |

每个区块建议使用 alma.now 风格的**卡片 + 浮窗式截图**，而不是纯文字描述。

## 6. CTO 发散思考：额外优化点

以下是产品规划文档中未明确覆盖、但对官网 + 产品体验有显著影响的优化方向。

### 6.1 品牌叙事收敛

**问题**：当前首页试图展示"10 个赛道的全景覆盖"，但对新访客来说信息过载。

**建议**：首页只讲**一个核心故事**——"帮用户挡住恶意 Agent，让 Agent 调用有信任保障"。10 个赛道的全景覆盖移到 `/roadmap` 页面，首页聚焦 3 个核心价值主张 + 1 个工作流程演示。

### 6.2 演示优先于文字

**问题**：当前页面全是文字描述和状态标签，缺乏可交互的演示。

**建议**：
- 首页 How It Works 使用**交互式流程动画**
- `/use-cases` 页面嵌入**可操作的 demo 片段**（如一个简化版的 Ask 交互）
- 路线图使用**可交互时间线**而非静态文本

### 6.3 社会证明与生态

**建议**新增：
- 首页底部增加 "Powered by A2A Protocol" 标识
- 如果有早期用户/合作方，增加 testimonial 区域
- GitHub Star 数、社区开发者数等指标展示
- 技术博客 / 更新日志入口（Building in Public 策略）

### 6.4 SEO 与性能

- 确保所有页面 SSR / SSG，满足 SEO 需求
- Hero 动效使用 CSS animation / lightweight canvas，不引入重型 3D 库
- 图片使用 Next.js Image 组件 + WebP
- Core Web Vitals 达标：LCP < 2.5s、CLS < 0.1、INP < 200ms

### 6.5 移动端适配

- 官网区：响应式设计，移动端 Hero 简化、导航折叠
- 产品区 IM 界面：移动端侧边栏变为底部 Tab / 抽屉，聊天窗口全屏
- 优先保证移动端"发一条消息、看一个结果"的核心流程

### 6.6 国际化增强

- 当前三语支持（EN / 繁中 / 简中）保留并加强
- Hero 区域的核心文案需三语同等品质（不能英文优先中文凑合）
- 路线图和 Use Cases 确保三语完整覆盖

### 6.7 分析与转化追踪

- 统一 CTA 事件模型（当前已有，需扩展到新页面）
- 新增：首屏滚动深度追踪、Hero CTA 点击热力图
- Consumer 漏斗：Open App 点击 → 浏览 Agent 目录 → 触发登录 → 首次对话 → 首次结果
- Provider 漏斗：Start Building 点击 → 阅读接入文档 → 触发登录 → 填写 Agent 信息 → 首次测试调用成功

## 10. CTO 总结与建议

### 10.1 执行优先级判断

**最应该先做的 3 件事**：
1. **双 Layout 分离 + 官网首页 Hero 重构**——这是用户第一印象，直接决定转化
2. **产品区 IM Layout 骨架**——这是产品核心形态的锚定，越早落地团队越有方向感
3. **色彩/字体统一系统**——这是所有后续视觉工作的基础

**最不应该先做的 3 件事**：
1. 不要先做完整的 /pricing 页面——定价模型还没有最终确定
2. 不要先做高级动效——先保证结构和信息正确，再打磨视觉
3. 不要先做移动端原生优化——先保证桌面端体验闭环

### 10.2 风险提示

| 风险 | 缓解 |
|------|------|
| 双 Layout 增加前端复杂度 | 使用 Next.js Route Groups + Layout 嵌套，架构清晰 |
| IM 布局对后端 API 有依赖 | Phase A 先做骨架和 Mock 数据，Phase B 再接真实 API |
| 视觉重构周期拉长 | 用 impeccable.style 加速，先定系统再做页面 |
| 产品区与官网区品牌断裂 | 共享设计 Token（色彩、字体、间距），只变 Layout |

### 10.3 最终判断

这次优化是**必要且紧迫的**。当前官网的信息架构和视觉品质与我们的产品定位（Agentic 基础设施的可信 Agent 准入与协作层）存在明显落差。竞品和参考站的视觉标准在持续提升，我们不能用"Phase 0 先跑通再说"来回避这个问题——因为官网是**所有用户接触我们的第一个触点**。

建议 Founder 审阅后，按 Phase A → B → C → D 的节奏推进，官网团队即刻启动 Phase A。
