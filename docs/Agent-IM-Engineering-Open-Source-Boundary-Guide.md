# Agent IM 开源边界原则与现有代码审核指南（主线工程）

日期：2026-03-15  
面向对象：主线开发工程师（代码与仓库边界守护人）

---

## 1. 目标

在坚持 Open Core 的前提下，确保：

1. 社区可接入、可验证、可扩展  
2. 商业护城河不被提前泄漏  
3. 现有仓库中开源内容与边界一致

---

## 2. 必须开源（Must Open Source）

以下内容属于生态入口，必须保持开放、可审阅、可复用：

- 协议规范与扩展文档（A2A 扩展、Node-Hub 协议、收据格式）
- SDK / Adapter / 示例实现（最小可运行）
- 自建节点最小核心能力（可连接、可同步、可最小会话）
- 极简控制台（用于验证流程，不含商业后台能力）
- 开发者文档与快速接入指南

判定原则：**能降低接入门槛、提升网络可连性的能力，优先开源。**

---

## 3. 不应开源（Should Stay Closed）

以下内容属于商业与运营护城河，不应在社区版公开完整实现：

- 托管运行时生产级调度、容量管理、运营自动化
- 高级 Trust Engine 策略包与企业策略编排
- 计费、分成、结算、反作弊与风控规则
- 高级目录推荐、信誉网络治理、运营后台
- 官方 Connector 的运维与策略控制能力
- 企业合规报表产品化能力、SLA 支撑系统、白标管理能力

判定原则：**可规模化运营和商业化变现的核心能力，保持闭源。**

---

## 4. 边界判定四问（提交 PR 前自检）

1. 这段代码是“生态接入基础能力”还是“商业运营能力”？  
2. 公开后，是否会显著降低我们未来商业护城河？  
3. 公开后，是否会暴露可被直接复制的运营策略？  
4. 该能力是否可拆成“开源接口 + 闭源实现”？

若 2、3 任一为“是”，默认进入闭源或分层实现。

---

## 5. 现有开源代码审核范围（当前仓库）

请按以下路径逐项审核并标记：

- `packages/server`
- `packages/console`
- `scripts`
- `docs`

每项输出标签：

- `OPEN-OK`：可保留在开源仓库
- `MOVE-CLOSED`：应迁移到闭源层或替换为抽象接口
- `SPLIT-NEEDED`：应拆分为开源接口 + 闭源实现
- `RISK-REVIEW`：存在边界争议，需 CTO 决策

---

## 6. 审核清单（必须逐条过）

### 6.1 协议与接口层

- 是否存在硬编码商业策略（不应）
- 是否暴露内部运营控制字段（不应）
- 是否提供稳定、可文档化、可测试的公共契约（应）

### 6.2 信任与风控层

- 基础风险决策与审计收据流程可开源（应）
- 企业级策略规则、权重、阈值、治理策略是否泄露（不应）
- 是否把“可验证能力”与“商业规则”混在同模块（应拆分）

### 6.3 节点与网络层

- Node-Hub 接入协议与最小节点能力是否完整（应）
- 主网运营策略与计费控制逻辑是否混入（不应）
- 是否保留版本兼容与协议演进约束（应）

### 6.4 控制台与前端层

- 极简演示与调试能力可开源（应）
- 商业后台能力（运营、计费、分成、企业治理）不应出现在 OSS（不应）
- 是否出现“看起来是 demo，实际包含商业逻辑”的混合实现（应拆分）

### 6.5 文档与示例层

- 文档是否清晰区分社区版与托管/企业版能力（必须）
- 是否存在误导性承诺（必须修正）
- 示例是否含内部密钥、内部流程、敏感脚本（禁止）

---

## 7. 审核输出模板（提交给 CTO/Founder）

请按下列结构提交一次完整报告：

1. 审核范围（路径与 commit 范围）  
2. 发现列表（按 `OPEN-OK / MOVE-CLOSED / SPLIT-NEEDED / RISK-REVIEW`）  
3. 高风险项 Top 5（附处理建议）  
4. 本周可完成整改项  
5. 需 CTO 决策项（不可逆边界变更）  

---

## 8. 本周必须完成的动作

1. 对 `packages/server` 做第一轮边界审计并打标签。  
2. 对 `packages/console` 做“演示层 vs 商业后台层”拆分建议。  
3. 在 `docs` 中新增“OSS vs Cloud 能力矩阵”草稿。  
4. 提交 1 份边界争议清单给 CTO 决策。  

---

## 9. 一句话原则

**开源“可接入标准与最小能力”，闭源“可运营规模化能力与商业策略”。**
