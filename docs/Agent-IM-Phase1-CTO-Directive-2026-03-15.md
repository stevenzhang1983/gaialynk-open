## 2. CTO 新口径（立即生效）

### 2.1 关于你提出的问题：官网区是否应完整体现产品愿景？

结论：**是，必须完整体现。**

但必须同时满足两条硬约束：

1. **愿景完整表达**：官网区应覆盖 `Product Plan` 的全局叙事，不因当前未交付而“看起来像只有一小块产品”。  
2. **能力状态透明表达**：对未上线能力必须明确标注状态（`Now` / `In Progress` / `Coming Soon` / `Research`），禁止把未来能力写成已可用承诺。

一句话：**官网区要讲全景愿景，但承诺口径必须与真实交付严格一致。**

## 3. 接下来 4 周开发指令（W+1 ～ W+4）

## 3.1 主线团队指令

### M1. 运行期可靠性（P0）

- 建立固定发布节奏下的 `preflight + post-release smoke + rollback drill`。
- 输出每周一次的失败复盘（含根因、回滚时长、预防动作）。

### M2. 节点可用化运营（P0）

- 将“可连”提升为“稳定协作”：离线/重试/不匹配异常纳入统一错误分级。
- 固化 Node-Hub 兼容承诺与版本窗口，新增每周兼容快照。

### M3. 一键部署支撑（P1）

- 冻结模板实例化相关接口与错误码。
- 对官网区“Start Building”链路提供稳定、脱敏、低耦合接口，不接受临时插单接口。

## 3.2 官网/入口团队指令

### W1. 官网区“愿景全景化”改造（P0）

- 基于 `Product Plan` 建立“愿景覆盖矩阵”（赛道 -> 页面模块 -> 状态标签 -> CTA）。
- 首页/能力页需覆盖核心赛道全景；未上线能力统一使用 `Coming Soon` 或 `Research` 标识。

### W2. 承诺一致性门禁（P0）

- 发布前新增“页面承诺一致性检查”硬门禁：页面能力声明必须可映射到主线已交付能力或明确状态标签。
- 任何无法映射的声明一律下线或改写。

### W3. 增长实验连续化（P1）

- 每周至少 1 个实验卡片（假设/阈值/护栏/决策）。
- 优先缩小 `zh-Hant` 与 `zh-Hans` 关键漏斗 gap，不追求无效流量带来的虚高 CTR。

---

## 4. 双团队协同机制（继续强化）

1. 固定每周联调窗口（不可临时取消）。  
2. 问题单统一分级：阻塞 / 高优先 / 可延后。  
3. 任何跨团队需求必须附“阶段一致性说明”和“契约影响声明”。  
4. 周五统一提交 Founder 决策包：继续放大 / 调整后执行 / 暂缓。  

---

## 5. 禁止项（本轮）

- 禁止把 Phase 2 编排/计费重能力提前塞入当前迭代。  
- 禁止官网区以营销叙事伪装“已可用能力”。  
- 禁止以赶进度为由削弱契约门禁、收据审计链路或反滥用基线。

**主线负责把可信能力做成稳定产品，官网负责把完整愿景讲清且不超承诺；两条线共同对“真实可验证结果”负责。**
