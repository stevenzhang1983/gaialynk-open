# Agent IM 主线工程审核结论与 Phase 1 执行规划

日期：2026-03-15  
面向对象：主线开发工程师（平台后端 / 协议 / 基础控制台）

---

## 1. 审核结论（可直接执行）

当前结论：**继续推进（Adjust & Execute）**，不回滚方向，不扩大范围。

### 1.1 已确认完成（认可）

- Phase 0 主闭环已经跑通：`发现 -> 调用 -> 审计 -> 收据 -> 风险控制`
- 核心能力已具备可运行、可测试、可演示条件：
  - 会话 + 目录 + A2A 映射
  - Trust Policy v0（allow / need_confirmation / deny）
  - 高风险 Review Queue
  - 审计与收据验签
  - Node-Hub 最小闭环（MVP）
- 工程节奏健康：具备 typecheck、测试、PG 集成验证与 CI 门禁

### 1.2 必须立刻修正（收敛）

- 不再扩 Phase 0 范围，进入 Phase 1 增强项
- 不允许“看起来完整”的功能挤占“可验证价值”的资源
- 信任体系优先做“可解释与可治理”，不提前复杂化为重信誉系统

---

## 2. Phase 1 目标与边界

## 2.1 目标（本阶段必须回答的问题）

1. 用户是否因为“可信调用”而持续使用网络协作，而不是只看 demo。  
2. 节点是否“可连且愿连”，并形成真实协作增量。  
3. 可信调用能力是否在产品层可感知（不仅存在于日志层）。

## 2.2 非目标（本阶段不做）

- 不做完整编排 DSL
- 不做完整商业化账单体系
- 不做大规模 IM 官方桥接矩阵
- 不做硬件大范围能力扩展（仅保留接口与阶段兼容）

---

## 3. 接下来 12 周执行安排（主线工程）

## M1（W1-W3）：可信链路硬化

### 目标

把“可跑通”升级为“可审计、可治理、可长期演进”。

### 交付

- 收据签名标准升级方案落地（建议 JWS/Ed25519）
- 审计 append-only 持久化与查询能力增强
- 人审流程升级（队列状态、幂等、防重复确认）
- 风险决策输出结构标准化（便于前端展示与对外解释）

### DoD

- 高风险路径端到端集成测试通过
- 收据验签链路有稳定测试覆盖
- 审计事件覆盖率与收据覆盖率可观测

## M2（W4-W6）：体验与发现补强（主线接口优先）

### 目标

支持上层体验增强，不在主线层制造技术债。

### 交付

- 线程 / @ / 在线状态所需后端接口与数据结构
- 能力标签与基础推荐所需目录扩展字段
- 首次会话成功率相关观测指标补齐

### DoD

- 新字段/新接口有向后兼容策略
- 关键写路径与查询路径性能基线可接受

## M3（W7-W9）：节点可用化

### 目标

从“节点可接入”走向“节点协作可用”。

### 交付

- 目录同步 + 跨节点中继最小闭环（受控范围）
- 节点兼容性承诺（版本策略与最低兼容）
- 限流与免费额度所需计数接口（先统计后计费）

### DoD

- 跨节点核心场景有稳定 demo 与自动化测试
- 节点异常状态（离线/超时/重试）可观测

## M4（W10-W12）：增长与产品线协同

### 目标

为官网入口线、开发者入口与后续轻量商业化提供稳定底座。

### 交付

- 面向官网/入口线的只读数据接口（安全脱敏）
- 开发者快速接入 API 体验优化
- 指标口径文档化（产品、工程、增长一致）

### DoD

- 对外文档与接口行为一致
- 关键指标可被官网/入口使用且不泄露敏感数据

---

## 4. 风险控制（主线工程必须遵守）

- **范围风险**：所有新增需求必须映射到本阶段目标问题，否则延后
- **架构风险**：不可逆变更（协议/核心数据模型）必须 ADR
- **可信风险**：任何削弱审计/收据完整性的改动禁止合入
- **性能风险**：新增链路必须给出超时、重试、降级策略
