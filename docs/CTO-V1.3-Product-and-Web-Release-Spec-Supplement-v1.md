## 1. 新用户首次体验（不含 §16 衔接）

### 1.1 目标与原则

- **5 分钟内完成一次「输入需求 → 得到可用结果」**（与主规格 §1.1 B.3.2 一致）。
- **零前置**：首启不强制连接器、不强制双 SaaS。

### 1.2 首启流程（建议 4 步内结束或可跳过）

| 步骤 | 行为 | 产品要点 |
|------|------|----------|
| S0 | 注册/登录完成 | 落地到 **产品区首屏**（见 1.3），而非长表单。 |
| S1 | （可选）一句话目标 | 单行输入：「你今天想完成什么？」可 **跳过**。若填写，用于 **预填首条消息** + **轻量推荐 Agent**（不阻塞进入对话）。 |
| S2 | 选起点 | 二选一：**「开始对话」** / **「浏览 Agent」**；老用户可勾选「不再显示」。 |
| S3 | 进入主路径 | 打开 **新会话** 或 **目录**，首条系统消息 **1 行**说明：可打字、可选 Agent、连接器「用到再连」。 |

### 1.3 首屏信息架构（登录后默认）

- **主区**：当前会话或「新建对话」占位 + 输入框（始终可用）。
- **次区（可折叠）**：「推荐 Agent」3～5 张卡片（可解释一行理由：类目/任务匹配）。
- **弱提示**：连接器入口为 **次要 CTA**（文案：「需要日历/笔记/本机文件时再连接」）。

### 1.4 空状态（无历史、无连接器、无订阅）

| 场景 | 展示 | 主 CTA | 次 CTA |
|------|------|--------|--------|
| 无对话 | 插画 + 一句价值主张 | 「新建对话」 | 「浏览 Agent」 |
| 无连接器 | 不打扰；仅在用户触发需连接器能力时 **内联** 引导 | 「去连接」 | 「先不用」 |
| 无订阅任务 | 在「任务/自动化」Tab 空状态说明 B 类能力；未上线则标 **In Progress** | 「创建任务」（若已交付） | 回对话 |

### 1.5 验收（产品向）

- 新用户 **未绑任何 SaaS、未装 Connector** 下，完成 **≥1 条有价值回复**（由 Epic 定义「有价值」：非空、非纯错误、符合任务类型）。
- 首启 **≤4 屏**（含跳过），无强制 OAuth。

## 2. 通知与异步结果投递

> 支撑主规格 §5.1.8 **B 类定时召回**与长耗时 A 类任务；避免「任务跑完用户无感知」。

### 2.1 通知类型矩阵

| 类型 | 触发 | 默认渠道 | 用户可关？ |
|------|------|----------|------------|
| **任务完成** | B 类 run 成功/失败；A 类长任务后台完成 | App 内 **通知中心/活动** + （可选）邮件 | 邮件可关；App 内「关键任务」不可全关 |
| **需确认/人审** | Trust Policy `need_confirmation` / `need_human_review` | App 内 **强提醒** + 邮件（可配） | 安全类不可关；非关键邮件可关 |
| **连接器授权** | 即将过期、已撤销、scope 变更 | App 内 + 邮件 | 邮件可关 |
| **配额** | 80% / 100% 用量 | App 内 + 邮件 | 邮件可关 |

### 2.2 渠道优先级（V1.3 建议落地顺序）

1. **产品内「通知中心」**（必须）：统一列表、未读角标、点进跳转到 **会话 / 任务 / run 详情**。
2. **浏览器通知（Web Push）**（可选 V1.3）：仅用户显式开启；用于「任务完成」「待审批」。
3. **邮件**（建议 V1.3 最小版）：模板至少：**任务完成摘要**、**待审批**、**配额告警**。

### 2.3 与审计/收据的关联

- 每条通知携带 **`notification_id` ↔ `run_id` / `conversation_id` / `receipt_id`（若有）** 的可解析链接（权限内可见）。
- **待审批**通知 deep link 直达 **会话内审批卡片**。

### 2.4 用户偏好（设置页）

- **勿扰时段**（仅影响非关键通知）。
- **渠道开关**：邮件开/关；Web Push 开/关（若实现）。
- **任务类通知粒度**：仅失败 / 成功+失败。

### 2.5 验收

- B 类任务在 **用户未打开聊天窗** 时，**通知中心** 可见完成态；点击可跳到该 run 时间线。
- 关闭浏览器后再次打开，**未读状态** 与服务器一致。

## 3. Agent 发现与推荐 UX

> 支撑主规格 §5 **双轨多 Agent** 与 §4 **分级/评测** 的 **消费者可理解** 呈现。

### 3.1 目录页 IA

- **顶栏**：搜索框 + 筛选（类目、风险等级标签、是否「平台评测通过」、是否支持 `scheduled`）。
- **主体**：卡片网格（移动端单列）；每卡信息层级见 3.2。
- **空搜索**：展示「热门 / 新手友好 / 低延迟」等运营位（与主规格 §4 分级一致）。

### 3.2 Agent 卡片（列表态）字段顺序

1. 名称 + **信任/分级徽章**（与主规格 §4 一致：`未验证` / `消费者可用` / `高敏加强` 等）。
2. **一句话价值**（≤80 字）。
3. **能力标签**（3～5 个）。
4. **容量信号**（诚实）：如「排队预计」「低并发」——来自主规格 §5.1.3 声明或平台默认。
5. **主按钮**：「加入会话」或「以此发起对话」。

### 3.3 Agent 详情页

- Tab：**概览** | **能力与限制** | **隐私与数据** | **评价（若有）** | **开发者信息**。
- **概览**：适用场景、不支持什么、典型输入输出示例。
- **能力与限制**：`supports_scheduled`、记忆层级声明、超时建议。
- **隐私与数据**：会传哪些字段到 Agent；是否经云代理；连接器依赖说明。
- **固定底部 CTA**：「用此 Agent 开聊」；若需连接器，**内联**说明缺什么授权。

### 3.4 意图路由交互（官方动态拓扑）

1. 用户发送自然语言消息后，若触发路由：在输入框上方出现 **「推荐方案」条**（可收起）。
2. 展示 **推荐链路**（步骤 1→2→3）+ **每步 Agent 名称** + **可替换下拉**（同类目候选）。
3. 用户选 **「确认执行」** 或 **「只要单 Agent」**（降级）。
4. 执行中：**步骤进度条** + 每步 **可展开** 看状态/收据入口。
5. 失败：**哪一步失败** + 「重试本步 / 换 Agent / 人工反馈」。

### 3.5 开发者拓扑包（轨 B）

- 目录中 **独立 Tab「工作流 / 拓扑包」**；卡片显示：包含 Agent 数、预估步数、适用场景、安装次数。
- **安装**：进入「我的已安装拓扑」；发起时选 **输入参数表单**（由包定义）。
- **升级**：显式版本号；升级前 **diff**（能力/Agent 变更摘要）。

### 3.6 验收

- 从搜索到「加入会话」≤3 次点击（不含登录）。
- 路由确认后，用户始终能回答：**当前在跑哪一步、为什么卡住**。

### 3.7 排序与推荐策略（产品规则）

> **详细优化计划、决策表与 Epic 建议**见 **§17.4**（目录搜索排序、信任等级对排序的影响、运营位与冷启动倾斜）。本节仅作 **UX 与 §3.1 空搜索运营位** 的索引，避免与 §17 双写。

## 4. 用户反馈与举报

> 落实主规格 §4 **线上声誉闭环** 的 **用户侧入口** 与 **治理可追溯**。

### 4.1 反馈入口（轻量）

- 每条 Agent 回复下 **「有用 / 没用」**；点「没用」展开 **可选原因**（事实错误、有害、太慢、未按要求、其他）+ **可选文本**。
- **不强制**长文；提交后 Toast：**已记录，将用于改进与风控**。

### 4.2 举报入口（重）

- Agent 卡片/详情 **「举报」**；会话内 **更多菜单 → 举报此 Agent 行为**。
- 表单：**类型**（欺诈、违法、隐私侵犯、恶意调用、版权、其他）+ **证据**（自动附带 `conversation_id`、`run_id`、最近 N 条消息摘要的引用，用户可删减敏感内容）。
- 提交后：**工单号** + 状态页链接（登录可见）。

### 4.3 处理与用户可见性

- 状态：**已收到 → 处理中 → 已处理**（结论枚举：下架/警告/无违规/需补充材料）。
- **不公开**举报人身份给被举报方。

### 4.4 与 Trust Policy / 声誉的联动（产品规则）

- 一定量「没用」+ 同类原因 → 触发 **自动复测**（主规格 §4）队列；结果回写 **降权/限权** 时，产品内 Agent 卡显示 **「近期体验波动」** 提示。
- 举报 **成立** → 目录状态变更 + 可选：历史用户通知「你曾使用的 Agent X 已下架」。

### 4.5 验收

- 100% 举报带 **可追溯引用**（消息/run 维度）。
- 用户可在 **账户 → 我的反馈/举报** 查看状态。

## 5. 对话生命周期管理

### 5.1 对话列表

- **默认排序**：最近活跃在上。
- **分组**：今天 / 7 天内 / 更早（可折叠）。
- **操作**：置顶、归档、删除（二次确认）、**标星**。

### 5.2 搜索

- **全局搜索**（顶栏）：按 **对话标题、参与者名、关键词**（服务端索引最近 M 条消息，M 由 Epic 定）。
- 结果：**高亮片段** + 跳转 **定位到该条消息**。

### 5.3 会话状态机（产品语义）

- `active`：正常收发。
- `archived`：只读 + 可恢复；不出现在默认列表（「已归档」入口）。
- `closed`：仅保留审计查看；UI 提示「已结束，可 fork 新会话」。

> **消息正文、审计切片、收据与 B 类 run 历史的保留期限、免费/付费差异**等产品规则见 **§17.3**，须与法务/合规定稿后写入帮助中心。

### 5.4 多会话并行

- 侧边栏 **固定「当前空间」会话列表**；**未读**按会话角标。
- **Draft**：切换会话时输入框内容 **按会话隔离** 自动保存（本地 + 短期服务端可选）。

### 5.5 导出（对齐 `Agent-IM-Product-Plan.md` §6.3 精神）

- 会话菜单 **「导出」**：Markdown / 纯文本；**默认脱敏**选项（隐藏系统上下文、连接器 token 引用）。
- 导出文件 **水印**：`user_id` 哈希短码 + 时间 + `conversation_id`（供内部纠纷关联）。

### 5.6 验收

- 归档会话在默认列表不可见；搜索可选「包含归档」。
- 导出内容与 **审计可见范围** 一致（权限不足则禁用）。

## 6. 错误状态与降级体验

### 6.1 分类与用户文案原则

| 类别 | 用户可见标题 | 说明策略 |
|------|--------------|----------|
| **平台故障** | 「GaiaLynk 暂时不可用」 | 状态页链接、重试、预计恢复（若有）。 |
| **Agent 不可用** | 「该 Agent 暂时无法响应」 | **原因枚举**（超时、503、维护）；**换 Agent** / **重试**。 |
| **队列饱和** | 「当前排队较长」 | **预估等待**或队列区间；「通知我」或「换低负载 Agent」。 |
| **策略拦截** | 「需要你的确认」 | Trust **摘要** + `reason_codes` **人话翻译** + CTA。 |
| **连接器** | 「需要连接或重新授权」 | 直达授权流；说明 scope。 |

### 6.2 部分成功（多步拓扑）

- **汇总卡片**：成功步数 / 总步数；失败步 **可单独重试**。
- **禁止**只显示「未知错误」。

### 6.3 网络断线

- **连接状态条**：「已离线，消息将在恢复后发送」；**发送队列** 可见（可取消）。
- 恢复后 **自动重试**（带退避）；失败转 **草稿**。

### 6.4 验收

- 所有错误 UI **必须**带 **下一步动作**（重试/替换/联系支持/帮助中心）。
- 可用性抽测：用户 **能区分**「平台挂了」和「Agent 挂了」。

## 7. 移动端策略（V1.3）

### 7.1 定位声明（写入规格）

- V1.3：**Responsive Web 为主**；**不承诺**原生 App。
- **PWA**：可选；若做，仅 **安装图标 + 基础离线壳**，核心能力仍在线。

### 7.2 断点与布局

- **&lt;768px**：单栏；侧栏 **抽屉**；目录全屏页；审批 **全屏模态**。
- **输入区**：固定底部；键盘弹起时 **输入框不被遮挡**（前端验收）。

### 7.3 桌面 Connector 在移动端

- 明确：**「本机目录/命令类能力需在桌面安装 Connector」**；移动端 **隐藏**不可用动作或灰显 + 说明。

### 7.4 验收

- iOS Safari + Android Chrome **核心路径**（对话、目录、审批）无横向滚动硬伤。
- 性能数字见 **§10**。

## 8. 用户账户与设置

### 8.1 账户页结构

- **个人资料**：头像、显示名、邮箱（只读或可验证修改）。
- **安全**：密码/SSO、活跃会话列表、**注销所有会话**。
- **通知**：见本文 **§2**。
- **连接器**：已授权列表、撤销、过期时间、**最近一次成功调用**。
- **数据与隐私**：导出我的数据、**删除账号**（见 8.3）。

### 8.2 用量/配额仪表

- **本月**：对话数、Agent 调用数、连接器出站调用数、（若有）订阅 run 数、（若有）上传存储。

### 8.4 验收

## 13. OAuth / 集成异常处理

### 13.1 Token 过期

- **主动刷新**；失败时：**会话内**系统消息「连接已失效」+ **一键重连**。
- **禁止静默失败**：依赖该集成的发送须 **前置拦截** 并说明。

### 13.2 用户撤销权限

- 下次调用前 **探测**；UI 同 13.1。

### 13.3 Scope 不足

- 提示 **缺哪项权限** + **重新授权（增量 scope）**（若厂商支持）。

### 13.4 区域/账号不一致

- 展示 **已连接账号掩码**（如邮箱后四位）；「切换账号」= 先撤销再连。

### 13.5 验收

- 集成测试覆盖：**过期、撤销、scope 不足** 三条路径 **均有**用户可见文案。

## 17. 产品语义与策略补缺（缺口 1–4）

> **来源**：CTO 在 V1.3 主规格 + 补充 v1.1 审阅后识别的 **中等重要** 缺口；**不替代** §15 的 P0/P1 Epic，而是与其 **正交**：§15 偏 **协作/身份/实时/Trust 呈现/门户/数据回路**，本节偏 **编排语义、跨 Agent 授权、留存策略、发现排序**。  
> **用途**：Epic/PRD 拆分时 **必须书面闭合** 下列决策点，避免工程各自假设导致返工。

---

### 17.1 缺口 1｜编排运行时产品层语义（官方动态拓扑 + 拓扑包执行）

#### 17.1.1 问题陈述

主规格 §5、补充 §3.4 已定义 **UX 形态**（推荐方案条、步骤进度、失败重试），但未定义 **「一步何时算完成」「步骤间数据如何传递」「部分成功如何交付」** 等 **产品行为约束**。没有这些约束，前后端与编排引擎会对 **进度条、收据、用户可见终态** 产生不一致实现。

#### 17.1.2 优化目标

用户在任意多步执行中始终能回答：**当前第几步、这一步凭什么算完成/失败、前面步骤的结果是否已安全交付、失败后重试会不会重复副作用**。

#### 17.1.3 须书面定稿的产品决策（检查清单）

| 序号 | 决策点 | 选项示例（Epic 中须择一或组合并写明） | 说明 |
|------|--------|----------------------------------------|------|
| **D1** | **单步「完成」判定** | A) Agent/A2A 返回终态 `completed`；B) 另加 **输出契约校验**（schema/非空字段）；C) 低置信度时标为「需用户确认才算完成」 | 与主规格 §4 质量轨、评测包可挂钩；**默认建议**：至少 B 用于消费者向关键链路 |
| **D2** | **步骤间数据流** | 上一步 **结构化输出** → 映射为下一步 **输入模板**（字段级映射表）；是否允许用户在步骤间隙 **编辑中间稿** | 若允许编辑，须定义 **版本号 / 确认动作** 写入审计 |
| **D3** | **部分成功（multi-step partial success）** | A) **已交付步骤结果即时展示**，后续步骤失败不回收已展示内容；B) **事务式**：任一步失败则整链标记失败，已生成内容进「草稿」区；C) 按 **风险等级** 分支（低敏 A、高敏 B） | 须与补充 §6.2「汇总卡片」一致；**帮助中心**须用同一套用户话术 |
| **D4** | **单步超时** | 超时后：**自动重试本步**（次数上限）/ **暂停等待用户选换 Agent** / **标记失败** | 与主规格 §5.1.3 `timeout`、队列行为一致 |
| **D5** | **重试语义** | 重试本步是否 **新 `run_id`**、是否 **幂等键**、是否 **重复计费/计配额** | 须与 §5.1.8、§9 计量对齐 |
| **D6** | **用户取消** | 取消时：**正在执行的步骤** 是否发中断信号给 Agent；已排队步骤 **是否丢弃** | 与对话生命周期 §5.3 `closed` 的交互须无矛盾 |
| **D7** | **拓扑包 vs 官方动态拓扑** | 轨 B 是否 **强制** 与轨 A 共用同一套步骤语义；差异是否仅在于 **触发源**（安装包 vs 实时推荐） | 建议 **共用运行时语义**，避免双实现 |

#### 17.1.4 建议交付物

| 交付物 | 说明 |
|--------|------|
| **《编排运行时产品语义》1 页决策表** | 填满上表 D1–D7，CTO/产品签字版 |
| **用户可见状态机图** | `pending → running → completed | failed | awaiting_user` 每步；与 UI 进度条字段一一对应 |
| **与收据/审计的字段映射** | 每步至少关联：`run_id`、步骤序号、`step_agent_id`、终态、可选 **输出摘要哈希或引用** |
| **帮助中心 FAQ** | 「为什么停在第二步？」「部分完成了结果在哪？」 |

#### 17.1.5 验收要点

- 任意 **官方动态拓扑** 演示脚本中，**失败注入** 后用户看到的文案与 **D3** 选定策略一致。  
- **重试** 后审计中可区分 **首次 run 与重试 run**（或幂等合并规则一致且可查）。  
- 与补充 §3.6 验收兼容：**用户能说明当前步与卡住原因**。

#### 17.1.6 建议 Epic / 文档名

- **`V1.3-Orchestration-Product-Semantics`**（产品子文档 + UX 联调）；实现可与 **`V1.3-Orchestration-Runtime`** 拆分，但 **语义文档先行**。

### 17.2 缺口 2｜Agent 代用户行动与「Agent 拉 Agent」的授权、可见性与审计

#### 17.2.1 问题陈述

主规格 §2 **第四章** 描述 **Agent 在策略允许下邀请其他 Agent**；Invocation Context 与 Trust 默认以 **人类用户** 为责任主体。若不在产品层定义 **授权继承链** 与 **数据可见边界**，会出现：谁为邀请负责、被拉入的 Agent 能看到多少上下文、用户如何知晓与撤销。

#### 17.2.2 优化目标

- **单一可追溯责任链**：任意 **非人类发起的入群/调用** 均能解析到：**发起实体**（哪个 Agent）、**授权来源**（哪个人类用户或哪条 Space 策略）、**策略命中摘要**。  
- **用户可理解**：成员列表与系统消息能回答 **「谁把谁拉进来的」**。  
- **可撤销**：用户可限制 **自己的助理 Agent** 的代理范围（与 §15 P0-1 RBAC 衔接）。

#### 17.2.3 须书面定稿的产品决策（检查清单）

| 序号 | 决策点 | 选项示例 | 说明 |
|------|--------|----------|------|
| **A1** | **邀请/拉 Agent 的授权模式** | **显式委托**：仅当用户曾对该 Agent 授予「可邀请其他 Agent」类权限；**Space 策略授权**：管理员预设允许的角色可发起 | 与 P0-1 角色矩阵合并一张表，避免重复冲突 |
| **A2** | **授权继承** | 被邀请 Agent 的 **`actor` 在审计中记为**：代理 Agent + **delegating_user_id**；Trust 决策的 **subject** 仍以人类或 Space 策略为准 | 与主规格收据字段设计对齐 |
| **A3** | **上下文可见性（数据边界）** | 被拉入 Agent **默认**：仅可见 **邀请之后** 的消息子集，或 **经策略标注为可共享** 的摘要；**升级可见** 须二次授权或人审 | 与企业数据边界叙事一致 |
| **A4** | **跨组织（供应商 Agent）** | **额外门槛**：仅公开目录 + Trust 允许 + 可选 **采购方管理员开关**；**默认禁止** 自动带入内部附件全文 | 与主规格 §2 第四章「可公开」一致 |
| **A5** | **用户侧呈现** | 系统消息模板：「**{助理名}** 已邀请 **{Agent 名}** 加入会话（依据你的授权/依据 Space 规则）」；成员列表 **徽章**：代理邀请 | 文案须进 i18n |
| **A6** | **撤销与过期** | 用户关闭「允许助理邀请」后：**已入群 Agent** 是立即移除、仅禁止新邀请、还是保留至会话结束——**三选一写死** | 与连接器撤销 UX 类比，降低认知成本 |

#### 17.2.4 建议交付物

| 交付物 | 说明 |
|--------|------|
| **《Agent 发起动作 RBAC 扩展》** | 附于 P0-1 或独立附录；含 A1–A6 决策结果 |
| **审计事件类型清单增量** | 如 `agent.invited_agent`、`agent.invitation_denied_policy` |
| **Trust Policy 输入增量** | 是否增加 `invocation_initiator_type: user \| agent` 及对应规则 |
| **帮助中心短文** | 「助理能替我拉别的 Agent 吗？」 |

#### 17.2.5 验收要点

- **演示脚本**：用户 A 的助理邀请 Agent B → 用户 B（同会话）在 UI 与 **导出审计摘要** 中能看到 **可理解** 的归因。  
- **负例**：未授权代理邀请 → **策略拦截** + 与补充 §6.1 **策略拦截** 文案一致。

#### 17.2.6 建议 Epic 名

- **`V1.3-Agent-Initiated-Actions-AuthZ`**（可与 **`V1.3-Space-Collab-RBAC-UX`** 子任务合并，但 **验收脚本独立**）。

### 17.3 缺口 3｜数据保留与会话存储生命周期策略

#### 17.3.1 问题陈述

补充 §5 定义会话 **状态机** 与 §8.3 **删除账号**，但未定义平台对 **消息正文、审计、收据、B 类 run 历史** 的 **默认保留期限** 及 **免费/付费差异**，工程难以统一归档、导出与合规话术。

#### 17.3.2 优化目标

- **对内**：表结构与归档任务有 **单一产品口径**。  
- **对外**：帮助中心、隐私政策、用量页 **表述一致**，避免「用户以为永久可查」与实际不符。

#### 17.3.3 须书面定稿的产品决策（检查清单）

以下 **具体月数/天数为占位**，由 **产品 + 法务** 在 Epic 中 **写死**；本节要求 **类目齐全**。

| 数据类目 | 须定义内容 | 备注 |
|----------|------------|------|
| **会话消息正文** | `active` / `archived` / `closed` 下分别保留多久；用户删除单条 vs 删除整会话 | 与 §5.3 状态机挂钩 |
| **系统消息与策略快照** | 是否与消息正文同周期；是否 **更长保留** 以利纠纷 | 常长于正文，须法务确认 |
| **审计日志（不可篡改取向）** | 用户可见 **切片** 保留 vs 平台完整保留；**导出范围** | 可与 Agent-IM 规划中的合规叙事对齐 |
| **收据 / 执行记录** | 与 `local_action_receipts`、外部连接器收据 **是否同周期** | 须满足主规格「可追溯单次真实操作」 |
| **B 类 scheduled run 历史** | run 记录、输出摘要、失败原因 **保留多久**；用户侧「历史可追溯」边界 | 与 §5.1.8、补充 §2 通知 deep link 一致 |
| **已下架 Agent 相关引用** | 历史会话中 **是否展示占位符**；元数据保留策略 | 与 §15 P1-2 Agent 生命周期衔接 |
| **免费档 vs 付费档** | 保留期是否不同；**超限提示** 是否含「将按策略清理」 | 与 §9 配额、主规格 §6 定价未定项兼容（策略可先定、数字后补） |
| **删除账号（§8.3）** | 各类目 **删除 / 匿名化 / 法定留存例外** 清单 | 法务主笔，产品对齐勾选 |

| 交付物 | 说明 |
|--------|------|
| **《数据保留与删除矩阵》** | 行：上表类目；列：保留期、用户可见、导出、删除账号、法律依据占位 |

| **实现任务**：归档作业、软删除、定时清理 | 依赖矩阵定稿 |

### 17.4 缺口 4｜目录搜索排序与推荐策略（含信任与运营位）

#### 17.4.1 问题陈述

补充 §3.1–§3.2 定义 **IA 与卡片字段**，但未定义 **搜索结果排序、默认列表排序、信任等级对排序的影响、新 Agent 冷启动、运营位选入规则**，前后端易各自实现一套逻辑，损害 **可信叙事诚实**（主规格 §4）。

#### 17.4.2 优化目标

- **可解释**：排序变化 **对用户/开发者** 有 **规则级说明**（非黑盒算法细节）。  
- **与信任一致**：高敏/未验证 **不得** 因运营需要被 **错误前置** 为「新手首选」。  
- **可验收**：同一查询在 **staging** 可复现排序 **优先级规则**（确定性测试或快照）。

#### 17.4.3 须书面定稿的产品决策（检查清单）

| 序号 | 决策点 | 选项示例 | 说明 |
|------|--------|----------|------|
| **R1** | **关键词搜索默认排序** | 相关性 primary + **信任等级/评测状态** 为 tie-breaker；或 **显式 Tab**：「最相关 / 最安全 / 最新」 | **建议** 提供 **安全优先** 切换，默认策略写清 |
| **R2** | **信任等级对排序的影响** | `未验证` **不得** 进入默认「新手友好」运营位；`消费者可用` **可** 参与；`高敏加强` **仅** 在显式筛选或专业场景出现 | 与主规格 §4 分级命名对齐 |
| **R3** | **新上架冷启动** | **有限期**「新上架」曝光位（如 7/14 天）、** capped 展示次数**、仍须满足最低评测/安全门槛 | 避免无门槛刷榜 |
| **R4** | **空搜索 / 首页运营位**（§3.1） | 「热门」「新手友好」「低延迟」**各自入池条件**：最低评测分、最近错误率、上架时长、**排除列表** | 每条运营位 **一段产品规则** |
| **R5** | **意图路由推荐（§3.4）与目录排序** | 是否 **共用** 信任信号；路由推荐 **是否允许** 未验证 Agent **仅在用户显式确认后出现** | 与 §17.1 编排语义衔接 |
| **R6** | **降级与失败** | 推荐服务失败时 **回退排序**（如纯字母序 + 信任过滤） | 避免空白或乱序 |
