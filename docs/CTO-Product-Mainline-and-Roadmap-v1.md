## 0. 为什么需要这份文档

### 0.1 问题

当前的产品规划有两套"地图"：

- **八大赛道**（内部工程维度）：IM 体验、发现网络、信任合规、编排语义、可观测、生态商业化、前沿差异化、硬件融合——它们是并行的技术流。
- **十项全景覆盖**（官网展示）：Developer Co-Building、Enterprise Governance、Node Collaboration、Managed Cloud Orchestration、Ask Main Path、Failure Recovery + HITL、Recurring Task Lifecycle、Connector Governance、Autonomous Business Ops、Marketplace Ecosystem——它们是平铺的能力清单。

**两套地图都不是"产品主线"。** 它们对内部团队有用，但对外部用户、投资人和社区开发者来说，看到的是一堆并行的东西，无法回答："你们**先**做什么？**然后**做什么？每一步用户获得什么**新价值**？"

### 0.2 解决什么

本文档的任务是：

1. **从用户价值视角**，把八大赛道和十项全景覆盖**合成为一条逐步扩张的产品主线**。
2. 每个阶段有**清晰的用户价值主张**（一句话说清楚"这个版本用户能做什么"）。
3. 映射回八大赛道和十项全景覆盖，确保**没有遗漏**。
4. 输出一份**可直接用于官网 Roadmap 页面**的路线图。

## 1. 产品主线：七个里程碑

> 核心逻辑：从"帮用户挡住恶意 Agent、建立可信调用闭环"开始，逐步扩展到"多 Agent 协作 + 行为管控"→"自动化"→"开放生态 + 准入治理"→"网络效应"→"企业级"→"物理世界"。每一步都建立在前一步之上，**可信 Agent 准入**始终是贯穿全线的底层能力。

```
M1 ─── M2 ─── M3 ─── M4 ─── M5 ─── M6 ─── M7
Agent   协作    自动化   开放    网络    企业    物理
准入    风控    闭环    生态    效应    治理    世界
闭环
```

### M1：可信调用闭环 — "说出需求，经过验证的 Agent 为你执行"

> **用户价值**：普通用户说出需求，平台将请求路由到经过身份验证、能力校验、行为追踪的 Agent；恶意或不可信的 Agent 被体系性挡在门外；调用全程可审计、可追溯，附带可验证收据。

**这是一切的起点。如果"平台能帮用户挡住恶意 Agent，让用户放心调用"这件事做不到，后面的一切都不成立。**

> **关于"可信"的核心定义**：本项目中的"可信调用"指的是**调用对象（Agent）的可信度**——它的身份是否经过验证、它的能力声明是否真实、它的历史行为是否稳定、它是否在策略管控之下。**我们不承诺 Agent 输出结果的正确性或准确性**——这取决于 Agent 自身的能力。我们承诺的是：用户调用的 Agent 来路清楚、经过准入、行为受监控、出了问题可追溯。类比：App Store 验证上架 App 不是恶意软件，但不保证每个 App 里的内容都是对的。我们做的是前者。

| 能力 | 来自哪个赛道 | 全景覆盖对应 |
|------|-------------|-------------|
| 最小会话系统（Conversation CRUD、消息流、人+Agent） | 赛道一：IM 体验 | Ask Main Path |
| 最小 Agent 目录（注册、查询、一键加入会话） | 赛道二：发现网络 | Developer Co-Building |
| A2A Gateway（与外部 Agent 通信、Task ↔ 消息映射） | 赛道一 + 二 | Developer Co-Building |
| 最小审计（事件日志、基础收据签发与校验） | 赛道三：信任合规 | — |
| 可信身份与来源标识 | 赛道三：信任合规 | — |
| 结构化通道（防注入基础） | 赛道三：信任合规 | — |
| 高风险动作默认不自动执行 | 赛道三：信任合规 | — |
| 会话记忆存储 + 上下文注入（最小版） | 赛道四：编排语义 | — |
| 时间线 API、基础延迟/错误统计 | 赛道五：可观测 | — |
| 开发者文档、SDK/示例 | 赛道六：生态 | Developer Co-Building |
| 多租户数据隔离 | 赛道七：前沿 | — |

**用户感知**：
- 我输入需求，平台帮我把请求交给了一个经过验证的 Agent，而不是一个来路不明的服务
- 我能看到这个 Agent 的身份、能力声明和信誉，知道它不是恶意的
- 高风险操作不会被自动执行，我有控制权
- 调用过程有收据，出了问题可以追溯
- 开发者可以按文档接入自己的 Agent

**验收问题**：用户是否因为"平台帮我筛选了可信的 Agent"而选择在这个网络里使用 Agent，而不是继续自己找来路不明的工具？

### M2：协作与风控 — "多 Agent 协作，恶意行为看得见、拦得住"

## 5. CTO 附注：主线之外的战略考量

### 5.1 为什么里程碑之间不是简单的先后关系

仔细看会发现，M3（自动化）在 Phase 1 就开始萌芽，在 Phase 2 才完善；M5（网络效应）在 Phase 2 起步，在 Phase 3 成熟。这不是规划混乱，而是**有意为之**：

- **早期种子**：在前一个 Phase 种下下一阶段的 MVP 种子（如 Phase 1 种下订阅任务 MVP）
- **用户验证**：用 MVP 数据验证假设后，下一个 Phase 再全力投入
- **避免断层**：如果每个 Phase 只做一个里程碑，用户在等待期会感到"平台停滞"

### 5.2 路线图的调整原则

这份路线图**不是圣经**。以下情况需要调整：

| 触发条件 | 调整方式 |
|----------|---------|
| 某个里程碑的用户验证结果为负面 | 缩减该方向投入，资源转向验证结果更好的方向 |
| 竞品在某个方向快速推进 | 评估是否需要提前该方向的里程碑 |
| 硬件团队提前就位 | M7 从 Research 提前到 Planned 或更早 |
| 企业客户需求强烈 | M6 可与 M4/M5 并行推进而非串行 |
| A2A 协议大幅变动 | 评估适配成本，必要时调整 M1/M2 的实现细节 |

### 5.3 对外叙事口径

路线图的对外口径应统一为：

> **GaiaLynk 正在建造 Agent 互联网的可信准入与协作层。**
> 
> 在 GaiaLynk 的网络里，每一个 Agent 都经过身份验证、能力校验和行为追踪——恶意 Agent 被挡在门外，而不是让用户自己承担风险。
> 
> 我们从"帮用户找到可信的 Agent 并安全调用"出发，逐步扩展到多 Agent 协作、自动化、开放生态、节点网络、企业治理，直至让 Agent 进入现实空间。
> 
> 每一步都建立在前一步的基础上——更多用户、更多可信 Agent、更强的准入体系、更大的网络。
