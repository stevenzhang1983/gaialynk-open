## 1. 总目标与边界

**范围**：下列能力在 **Web 主应用** 与 **官网** 上均为 **V1.3 完整交付**（非「部分上线 / 仅文案」）：同一套叙事、同一套对象模型。**工程与验收重心**为 **§2 总用户旅程第一至第五章**——须尽可能 **真实可用**，而非仅可演示（详见 **§3「完整实现」含义**）。**M5 子网互联**与 **M7 硬件 / 物理世界** 的工程交付 **不纳入 V1.3**（路线图与叙事可保留远期表述）。**仅定价策略未定项**见 **§6 定价**。

**受众**：以 **普通消费者** 为主；开发者 / 组织为次级但路径清晰。

**原则**：

- **可信** = **治理轨**（准入、策略、审计、收据）+ **质量轨**（可复现评测、线上声誉、分级曝光）。
- **多 Agent** = **官方动态拓扑推荐** + **开发者上架拓扑包** 双轨；**上架 Agent 的并发、实例池、调度与记忆隔离** 见 **§5.1**。
- **连接器** = 见 **§1.1**；V1.3 须从「仅记账演示」升级为 **真实能力出口**（本地有限控制与/或真实 API），与总旅程第一至第五章的 **真实可用** 要求一致。

### 1.1 连接器：现有代码现状与 V1.3 需求（定稿）

#### A. 现有实现（代码与产品现状）—— Phase 0 / 演示级

**A.1 主网（`packages/server`）**

| 能力 | 实现位置与行为 |
|------|----------------|
| **授权模型** | `connector_authorizations` 表（迁移见 `packages/server/src/infra/db/migrations/0004_mainline_phase_bcd.sql`）；内存 Map 回退见 `packages/server/src/modules/connectors/connector.store.ts`。字段含：`user_id`、可选 `device_id`、`connector`（字符串命名空间）、`scope_level`（`directory` \| `application` \| `action`）、`scope_value`、`expires_at`、`status`。 |
| **Scope 语义** | `packages/server/src/app.ts` 内 `isActionWithinScope`：`action` 级精确匹配；`application` 级为 `scope_value` 或其 `prefix.`；`directory` 级为 `action.startsWith(scope_value)`。 |
| **动作命名约束** | `isActionForConnector`：`action` 必须以 `` `${connector}.` `` 为前缀，否则拒绝执行。 |
| **HTTP API** | `GET/POST /api/v1/connectors/authorizations`；`POST .../authorizations/:id/revoke`；`POST .../authorizations/revoke-by-device`；`POST /api/v1/connectors/local-actions/execute`；`GET /api/v1/connectors/local-action-receipts/:id`（需 `actor_id`）。鉴权与 `ACTOR_CONTEXT` 对齐见 `app.ts` 各路由。 |
| **执行语义** | `executeLocalActionAsync` **不调用** 本机进程、**不发起** 外网 HTTP、**不读写** 用户磁盘；直接构造 `LocalActionReceipt`，`result` 恒为 **`completed`**，`env_signature` 为对 payload 的 **HMAC-SHA256**（密钥为 store 内常量 `phase0-local-action-receipt-secret`）。 |
| **高风险** | `risk_level` 为 `high` / `critical` 时，请求体须 `confirmed: true`，否则 403 并记审计事件 `connector.local_action.blocked_overreach`。 |
| **审计** | 拒绝/成功路径见 `emitAuditEventAsync`（如 `connector.local_action.completed`、`blocked_scope`、`blocked_revoked` 等）。 |

**A.2 网站（`packages/website`）**

| 能力 | 实现位置与行为 |
|------|----------------|
| **主网代理** | `packages/website/src/app/api/mainline/connectors/**` 将浏览器请求转发至配置的主网 Base URL。 |
| **产品 UI** | `packages/website/src/app/[locale]/(product)/app/connectors-governance/page.tsx`：`ConnectorPanel` + `ConnectorsGovernanceDemo`——登录用户可 **创建授权、列表、撤销、填写 action 字符串执行、查看收据**；与聊天主路径 **无自动联动**。 |
| **侧栏** | `packages/website/src/components/product/sidebar.tsx` 等将「连接器」作为独立治理入口。 |
| **治理元数据** | `packages/website/src/content/entry-page-governance.ts` 中 `connectors-governance` 带 **Coming Soon** 类状态标记（与已有 Demo 并存，对外口径需 V1.3 统一）。 |

**A.3 结论（现状边界）**

- 现有栈实现的是 **授权生命周期 + 作用域校验 + 审计事件 + 可验证收据（针对「已登记的执行」）** 的 **契约与账本**，**不是** 真实「本地执行器」或「SaaS 客户端」。  
- **V1.3 须在保留/演进上述治理概念的前提下，新增「真实副作用」执行层**；不得仅依赖当前 `executeLocalActionAsync` 的模拟完成。

#### B. V1.3 需求（并行交付：浏览器能力 + 云代理 SaaS + 轻量桌面 Connector）

**B.1 总原则**

- **同一套用户心智**：连接器 = 用户显式授权下的 **能力延伸**；**撤销、过期、scope、高风险确认、审计、收据** 与主网策略一致并 **可追溯到单次真实操作**。  
- **密钥与 client_secret**：**禁止** 长期密钥裸露在浏览器；SaaS 类走 **服务端（或受控边缘）云代理**，浏览器仅完成 **用户授权流** 或持有 **短期、受限令牌**（按方案定稿）。  
- **V1.3 拍板**：**并行** 交付 **轻量桌面 Connector**（非「下一小版本可选」）；与 **浏览器内文件能力**、**两个真实 SaaS 集成（默认选型见 B.3.1）** 同属 V1.3 范围。

**B.2 路径一：Web App 内（标准浏览器）**

| 需求项 | 说明 |
|--------|------|
| **用户选取文件** | 通过 `<input type="file">`、拖放等，将 **用户选中的文件** 纳入会话或连接器任务；内容上传或解析路径须 **可审计**（含谁、何时、哪类操作）。 |
| **可选：已授权目录（浏览器能力允许时）** | 在支持的浏览器上，使用 **File System Access API** 等，使用户 **绑定一块明确目录** 为工作区；读写范围 **不得** 超出用户授予的句柄；降级策略须在 PRD 写明（不支持时回退到「仅选文件」）。 |
| **与对话/Agent 联动** | 至少一条路径：**从聊天或连接器管理** 触发「基于用户授权文件的」真实处理（例如上传至对象存储再由 Agent 处理、或浏览器内解析后传结构化结果——具体由 Epic 定），且 **非** 仅调用现有模拟 `local-actions/execute`。 |

**B.3 路径二：云代理 SaaS（真实 HTTP API）**

| 需求项 | 说明 |
|--------|------|
| **集成数量** | V1.3 **至少 2 个** 第三方 SaaS，完成 **真实 OAuth（或等价用户授权）** 与 **每集成至少一类可核验 API 调用**。**默认选型** 见下文 **B.3.1**；若因合规、区域市场或厂商政策需替换，须在 Epic 中 **书面记载替换对象与理由**，并保持「两类能力不重叠」原则（时间协调类 + 文档/知识库类）。 |
| **与连接器模型对齐** | 连接类型、scope、令牌存储、刷新、撤销 与现有 `connector` / `scope_*` 概念 **映射或扩展**（新增表字段或并行 `integration` 实体须在技术设计书中定义，避免两套无法合并的孤岛）。 |
| **收据** | 每次对外 API 调用（或幂等批次）产生 **可与审计对齐** 的记录（沿用或扩展 `local_action_receipts` / 新 `external_action_receipts` 由设计决定），**禁止** 无记录的「静默代发」。 |

**B.3.1 CTO 建议：V1.3 优先落地的两个 SaaS（面向广义消费者）**

> 定位对齐：**普通消费者**与小型团队均高频使用、OAuth/API 生态成熟、且与「对话里交代办事项」自然契合；两集成 **类目不同**，避免重复证明同一种集成模式。

| 优先级 | SaaS | 建议首期 API 能力（可验收） | 选型理由 |
|--------|------|-----------------------------|----------|
| **1** | **Google Calendar** | 只读或可写其一：**列出近期事件 / 空闲时段**，或 **创建一条带标题与时间的日历事件**（具体只读/读写组合以最小合规 scope 为准）。 | **时间协调**是跨人群刚需（上班、带娃、自媒体排期、小团队会议）；与总旅程中 **排班、会议、发布节奏** 同一抽象；全球消费者渗透率与文档完整度高，利于 V1.3 **第一条**云连接器跑通 OAuth + 云代理 + 收据全链。 |
| **2** | **Notion** | **按用户授权列出若干页面/数据库条目**，或 **在指定父页下创建一页简单笔记**（以 Notion Integration 能力与 scope 最小集为准）。 | **文档与结构化笔记**覆盖个人知识库、轻量 PRD/清单、营销 brief 等 **非日历** 场景；与 **CEO/CMO 文案、研发纪要** 叙事一致；与 Calendar **不同数据面**，证明平台连接器模型 **可复用到多厂商**。 |

**替换与区域说明**

- 若产品阶段侧重 **中国大陆** 且上述厂商接入成本过高，可在 Epic 中将 **Calendar 类** 替换为 **飞书日历 / 钉钉日程** 等等价能力，将 **文档类** 替换为 **飞书云文档 / 钉钉文档** 等，但须仍满足 **两类目各一、真实 OAuth/授权、可核验 API**。
- 若法务或品牌限制 **单一厂商绑定**，可将 Notion 换为 **Microsoft OneDrive 个人版** 或 **Dropbox** 等「个人云盘」类，仍属 **文档/文件元数据** 类目；Calendar 侧可换 **Microsoft Outlook Calendar**，但会改变「双厂商」组合，须在 Epic 注明。

**B.3.2 消费者「上手即用」与 SaaS 集成（产品设计约束）**

> 问题：面向 **普通消费者** 时，**真实 SaaS 集成** 是否必然导致「要先做一大堆配置」？  
> 结论：**可以做到接近「上手即用」**，前提是 **不把连接器当作注册后的强制门槛**；**安全相关步骤**（登录、OAuth 同意、权限说明）**允许存在**，且应设计得 **短、可跳过、可事后补**。

| 原则 | 要求 |
|------|------|
| **核心价值零前置** | 用户 **首次进入** 即可在 **不连接任何 SaaS、不安装 Connector** 的情况下，完成 **对话 → 调用平台内 Agent → 得到结果**（与 §2 第一章一致）。**禁止** 将「必须绑定 Google / Notion」作为注册或首启 **硬门槛**。 |
| **渐进式Reveal（用到再连）** | Calendar / Notion 仅在 **用户意图需要**（如「帮我看下明天有没有空」「记到笔记里」）或用户 **主动进入「连接服务」** 时，再触发 **最短路径** 授权；授权流目标控制在 **少量明确步骤**（跳转厂商登录 + 勾选权限 + 回到产品），**不**要求用户事先在厂商控制台自建应用或填写 Client ID。 |
| **多集成非串联** | **不要求** 同一用户首日即完成 **两个** SaaS 绑定；第二个集成应在 **相关场景** 或 **设置页可选** 中出现，避免「配置清单」体感。 |
| **与「安全设定除外」对齐** | 用户理解的 **安全动作**（登录本产品、看清连的是什么账号、同意范围、可撤销）**保留**；这些 **不算**「繁琐前置配置」，但文案须 **短句、一页说清**，避免技术术语堆砌。 |
| **轻量 Connector** | **安装与配对**可与主路径 **解耦**：用户可先纯 Web 使用；仅在需要 **工作区目录** 等能力时再引导下载，且 **配对码 / 首次信任** 视为 **安全步骤** 而非「配置工程」。 |
| **验收（体验向）** | V1.3 可用性评审须包含：**新用户 5 分钟内**在不绑 SaaS 的情况下完成 **一条有价值对话**；绑定 **第一个** SaaS 的 **中位步骤数** 与 **放弃率** 在 Epic 中设 **基线目标**（具体数字由产品定）。 |

**B.4 路径三：轻量桌面 Connector（与 Web 并行发布）**

| 需求项 | 说明 |
|--------|------|
| **定位** | 官方分发的 **小体积常驻组件**（托盘/菜单栏级），为 Web App 提供 **浏览器无法单独完成** 的本机能力；**主交互仍在浏览器**。 |
| **目录挂载** | 用户 **显式选择** 至少一个 **工作区根目录**；Connector 仅在该根目录树内执行约定操作（列表、读、写、监视变更——以 Epic 列清）；**禁止** 默认全盘访问。 |
| **命令执行（受限）** | V1.3 允许 **0～1 条** 经 **白名单** 与 **版本锁定** 的本机命令或等价能力（若无则目录读写即可交付）；任何扩展须 **人在环** 或 **首次明确同意**。 |
| **与主网通信** | **双向认证**（配对码、设备令牌或 mTLS 等——技术设计书定稿）；仅接受 **声明过的 Web Origin** 或本地回环上的受控客户端；所有本机侧执行结果 **回传主网** 以落 **审计 + 收据**。 |
| **生命周期** | 安装、更新、卸载路径清晰；**更新包签名** 与 **渠道单一可信** 为发布 **硬门槛**。 |
| **威胁模型与文档** | V1.3 发布包须附带 **简短威胁模型**（本机恶意软件滥用 localhost、用户误授权、更新劫持等）与 **缓解措施** 说明（对内安全评审 + 对外可摘要）。 |

**B.5 与现有 API 的演进关系（须在设计书中落地）**

- **选项（择一或组合）**：(1) 将真实执行结果仍挂到 **`authorization_id` + 新执行子类型**；(2) 引入 **`execution_id` / `connector_runtime_id`** 与收据关联；(3) 模拟 execute 路径 **废弃或仅用于兼容测试环境**。  
- **明确废弃语义**：V1.3 发布后，**面向终端用户的「执行」不得** 再依赖「无外部副作用的 `completed` 伪造」作为唯一实现。

#### C. V1.3 验收（连接器专项）

1. **浏览器**：完成 **选文件 → 真实处理 → 会话或 UI 可感知结果** 的端到端路径至少 **1 条**（可含目录授权降级说明）。  
2. **SaaS**：对 **B.3.1 所列两项**（或 Epic 批准的等效替换）分别完成 **授权 → 云代理真实调用 → 结果写回产品内可展示**；每项均可独立验收。  
3. **轻量 Connector**：完成 **安装 → 绑定工作区 → 经 Web 触发本机侧真实文件操作 → 主网审计/收据可查询** 至少 **1 条** 端到端路径。  
4. **治理**：上述路径均与 **撤销授权、过期、scope 越权拒绝** 之一 **可演示负例**；帮助中心与 **Coming Soon** 类入口 **与真实能力一致**，无虚假宣传。

## 2. 总用户旅程（单线叙事：CompanyA）

以下将原先多条并列用户旅程收敛为 **一条总旅程**：主角为 **CompanyA**（从事智能硬件研发设计、生产与销售的虚构公司），在 **Web 端、类 IM 聊天窗、可信调用 Agent** 的产品形态下，按 **协作复杂度递进** 展开。你也可以把 **CompanyA** 想成你自己的工作室或小团队甚至是你自己的一人公司——环节变少，阶段逻辑不变。

### 第一章｜各自起步：一句话需求 → 单个可信 Agent

各部门先在 **自己的对话** 里习惯「说清楚要什么 → 平台把请求交给已上架、已核验的 Agent → 拿结果」。此时不要求跨部门同群，重点是 **敢用、能用、有收据 / 可追溯**。

| 环节 | 谁 | 典型一句话 | 单个 Agent 能做什么 |
|------|----|------------|---------------------|
| 对外营销 | **CMO** | 「这周要发三条短视频，讲新品亮点」 | **视频向 Agent**：按 brief 生成可发布的短视频素材或粗剪方案 |
| 战略与产品 | **CEO** | 「写一版三个月产品愿景和 PRD 骨架」 | **助理类 Agent**：输出愿景叙述、PRD 结构、待填假设清单 |
| 研发协同 | **CTO**（可先单人用） | 「把本周软硬件接口变更整理成研发看板说明」 | **研发助理 Agent**：从文档 / 纪要风格需求生成接口变更说明、风险列表（后续再拉团队进群） |
| 采购 | **采购部** | 「帮我把这几类物料的网上公开报价整理成对比表」 | **价格调查 Agent**：爬梳公开信息、汇总报价维度（非最终合同价） |
| 生产 | **生产管理** | 「按现行产线规则生成下周排班初稿」 | **排班 Agent**：根据默认规则生成可执行排班表 |

**本章收束**：同一家公司、五种运营动作，五种「单 Agent 可信调用」。

### 第二章｜能力加深：多 Agent / 多步拓扑（仍以部门为主语）

部门内从「单次问答」变成 **固定链路**。实现方式可以是：**平台根据自然语言从目录自动推荐多 Agent 链路**，或用户安装 **开发者上架的拓扑包**。

| 环节 | 演进 |
|------|------|
| **CMO / 自媒体一站式** | 从「一个视频 Agent」→ **多 Agent 链路**：选题 / 脚本 → 视频生成 → 发布清单 → 评论 / 互动摘要（以实际上架能力为准） |
| **CEO** | 从「一版 PRD 骨架」→ **多步深化**：市场假设 → 功能优先级 → 里程碑与风险 |
| **CTO / 研发** | 从「单份说明」→ **软硬件协同链**：结构约束摘要 → 嵌入式接口提示 → 联调检查表 |
| **采购** | 从「公开报价表」→ **多源、多轮澄清**：规格维度补全、交期 / MOQ 等字段结构化 |
| **生产** | 从「规则排班」→ **多约束迭代**：产线班次、技能认证、设备保养窗口等逐步纳入 |

**本章收束**：多 Agent 拓扑落在 **营销、研发、采购、生产** 的真实工作流上；对齐「官方动态拓扑 + 开发者拓扑包」双轨供给。

### 第三章｜人进群：部门内与跨部门的真实协作

| 场景 | 旅程画面 |
|------|----------|
| **生产排班** | 生产管理把 **班组负责人** 拉进群；各人提交不可用时段 / 特殊需求；**同一排班 Agent** 收集约束后重算排班表，负责人确认定稿 |
| **研发协同** | **CTO** 把 **嵌入式、结构、测试** 等同事拉进「新品联调」群；人类讨论同时可 @ **自己的助理 Agent** 拉纪要、接口草案、测试点 |
| **CEO ↔ 高管** | **CEO** 邀请 **CTO、CMO** 进「产品方向」群；各高管可再拉自己的助理 Agent，形成 **人 + Agent** 混编 |
| **采购（内部）** | 采购与财务 / 法务同事进群，对报价整理结果做内部评审 |

**本章收束**：组织、权限与人在环进入同一条叙事。

### 第四章｜Agent 拉 Agent：供应链与「可发现」网络

- **采购**：**采购方 Agent**（或经授权的编排）在策略允许下，邀请平台上 **公开可发现** 的 **供应商 Agent** 入群；供应商 Agent 提供 **可公开** 的规格、交期区间、公开价层级等信息。
- **CMO**：可类比调用已上架的 **分析 / 合规类** 等 Agent（若生态已有），体现同一机制跨部门复用。

**本章收束**：跨组织协作在 **平台治理下的可发现连接** 中发生。

### 第五章｜关键裁决回到人

- **采购**：内部意向明确后，向 **供应商侧人类** 发起来访 / 邀请，完成 **订单条款核对、审批与放行**。
- **CMO**：对外发布前，**CMO 或法务** 对最终稿确认（高风险动作不自动发布）。
- **生产**：排班 **定稿** 由指定负责人确认。
- **研发 / CEO**：对客户或外部的 **规格与承诺类文档** 由 **人类签发**，Agent 仅出草案。

**本章收束**：可中断、可归责——Agent 加速准备，**盖章在人**。

### 第六章｜公司与网络同频生长（路线图延伸）

> **V1.3 工程范围说明**：本章为 **愿景与路线图延伸**，用于官网/叙事与长期对齐；**子网互联（M5 相关工程）** 与 **硬件 / 物理世界（M7）** **不作为 V1.3 必须交付**。第一至第五章见 **§3「完整实现」含义**。

- **网络效应（M5）**：更多供应商与行业 Agent 可发现，可选拓扑变多；组织可与合作伙伴 **子网互联**（与开发者第三类路径叙事衔接）。
- **企业治理（M6）**：公司级策略、审计视图、合规导出。
- **物理世界（M7）**：产线设备、仓配、售后终端以 **执行型 Agent / 设备身份** 接入（叙事与路线图可 **预留一致表述**）。

## 5. 多 Agent 双轨（主应用 + 官网 + 协议 / 目录）

| 轨 | V1.3 完整实现要点 |
|----|-------------------|
| **A. 官方动态拓扑** | 意图解析 → 从 **已上架单 Agent** 集合生成 **推荐链路** → 用户可确认 / 修改 → **编排运行时**执行 → 失败 **fallback**。 |
| **B. 开发者拓扑包** | 目录中 **一等公民**：拓扑包实体（DSL / 配置 + 引用的 Agent ID + 版本）；**安装、升级、评测、展示**与计费钩子与单 Agent 区分。 |

**官网**：应用场景与开发者文档 **同时** 解释 A / B。  
**主应用**：两条路径均可 **创建会话 / 任务**。

### 5.1 上架 Agent：并发、容量、调度与记忆隔离（产品方案）

本节收敛 **多人同调同一上架 Agent**、**OpenClaw 类部署**、**平台与上架者分工**、**营销表述**、**记忆隔离**、**逻辑 Agent / 实例池 / 租约** 等讨论，作为 V1.3+ **目录与调用链** 的产品与契约基准（实现可分期，**语义须先统一**）。

#### 5.1.1 多人同时调用时会发生什么（技术现实）

并发行为 **不由平台单方面决定**，而由 **Agent 部署端架构** 决定。常见结果如下：

| 部署形态 | 多人同时发起调用时 |
|----------|---------------------|
| **单进程 / 单会话** | 后续请求 **排队**、**超时**，或 **503**；若实现有缺陷，可能出现 **状态错乱**。 |
| **每请求独立 worker、无共享可变状态** | 可 **并行**；吞吐随部署方 **横向扩展**；成本在部署方。 |
| **共享全局变量、单例记忆、单文件锁等** | 易出现 **竞态、串话、错误写入**——属于实现事故高发区。 |

**平台侧责任边界**：平台提供 **可观测、可排队、可限流、会话隔离键透传**，并在 **上架契约** 中固化 **容量与语义**（见 **5.1.3**）；**不**承诺替任意第三方部署「变快」或「变无限并发」。

#### 5.1.2 OpenClaw 类 Agent 是否支持并发

**无统一答案**，须按 **具体部署** 认定：

- **OpenClaw 类**（高权限、长驻、工具链重）常偏 **交互式、有状态**，许多实例默认 **单用户或低并发** 心智，**不等于**开箱即 **多租户高并发 API**。
- **是否支持并发** 应作为 **上架者声明 + 平台评测/压测** 的事实，**不是**平台默认保证。

**工程归纳**：若后端为 **无状态 HTTP/A2A**，每请求携带 **`conversation_id` / 平台 `user_id` / `run_id`**，且进程内 **不** 使用「当前用户」类全局可变状态，则 **更易** 并发。若为 **单线程事件循环 + 全局 planner 状态** 等，则 **天然不适合** 多路并发，除非 **多实例 + 负载均衡** 且状态外置。

#### 5.1.3 并发能力弱时：平台设计原则与上架者最小负担

**原则**：平台 **默认不** 对外承诺「每个上架 Agent 均为无限并发 SaaS」；以 **契约 + 默认平台行为** 保护消费者与上架者。

**对平台（产品设计）**：

| 能力 | 要求 |
|------|------|
| **上架声明** | 须可填或系统默认：`max_concurrent`（或等价「副本数」）、`queue_behavior`（排队 / 快速失败）、建议 `timeout`、是否 **无状态可扩缩**。**未声明** 时采用 **保守默认**（例如并发 **1**、超出则 **排队或 429**）。 |
| **隔离键** | 每次调用 **必须** 携带平台 **会话 ID、用户 ID、运行 ID**（字段名以协议为准），避免 Agent 端将多人混为同一上下文，并支撑 **审计 / 收据**。 |
| **队列与 UX** | 平台侧 **公平队列**（或明确失败策略）；用户侧展示 **等待、重试、更换 Agent**；**禁止** 无限挂起无超时。 |
| **多实例路由（可选、可分期）** | 同一 **逻辑 Agent** 背后 **N 个同构 endpoint**，由平台 **调度**；减轻小开发者 **第一天上 K8s** 的压力——**诚实低并发 + 队列** 优于 **虚假高可用**。 |

**对上架者（最小负担路径）**：

- **起步**：**单副本** + 声明「同时仅服务 **1** 路，其余由平台排队或拒绝」——**诚实优于崩溃**。  
- **进阶**：**无状态** 请求处理 + **外置会话存储**，键为 **平台下发的 user/session/run**。  
- **扩容**：按预估增加 **同构副本**（见 **5.1.6**），由平台或发布者维护 **实例池**，消费者仍见 **一张卡片**。

#### 5.1.4 「高效整合调用」与营销表述

| 可做卖点（须可验证） | 慎用/避免 |
|----------------------|-----------|
| **会话级隔离**、**公平排队与超时**、**与 A2A/目录的统一发现**、**收据与可观测** | 泛泛承诺「调度所有 Agent 都最快」——瓶颈常在 **对方机器**。 |
| 若未来落地 **冷启动缓解、重试策略、连接池、区域中继** 等，再升级为 **性能向卖点** | 无实测数据前的 **绝对化性能宣传**。 |

**推荐主叙事**：**可信调用 + 容量可预期 + 排队/失败可理解 + 不串话**，优于「全网最快」类表述。

#### 5.1.5 多用户、不同时段调用与记忆系统

| 风险 | 说明 |
|------|------|
| **全局一条记忆** | 用户 A 的偏好/秘密进入 **共享检索面**，用户 B 可能被 **误召回**——严重事故。 |
| **按部署者单一库混写** | 多用户写入同一长期记忆空间 → **污染、串味**。 |

**合理模型（平台规范 + 评测）**：

- **长期记忆** 必须按 **`(platform_user_id, conversation_id)`** 或 **更细粒度** 分区；**默认禁止** 跨用户召回。  
- **每轮请求** 仅注入 **本会话**（及 **经用户明确授权** 的跨会话策略）。  
- **Agent Card / 上架文档** 须声明 **记忆范围**：无 / 仅会话 / 按用户隔离的持久化等。  
- **平台**：通过 **评测与抽查** 验证 **无跨用户记忆泄漏**；**上架者**：实现 **隔离存储**，不得以「共享 Agent」为借口混库。

#### 5.1.6 逻辑 Agent、实例池与两种调度模式

**目录呈现（消费者视角）**：

- **对外**：一个 **逻辑 Agent**（**一张卡片**、一套评测与叙事）。  
- **对内**：**N 个同构实例**（endpoint/副本），由 **平台或发布者** 注册与维护；**避免** 消费者面对 `foo-copy-3` 等分裂条目（除非明确为「不同版本/不同能力」）。

**容量与扩展**：

- **容量以实例（副本）为单位**；上架者可从 **1 副本** 起，按负载 **增加同构实例**。  
- **记忆隔离** 为 **硬要求**，与是否「独占某一副本」**无矛盾也互不替代**：隔离靠 **数据模型与存储 key**，不能只靠「换机器」。

**两种调度模式（并存，上架时可选或平台按声明默认）**：

| 模式 | 含义 | 适用 |
|------|------|------|
| **池化 + 无独占** | 任意空闲副本处理请求；靠 **`conversation_id` + 用户隔离记忆** 保证不串话 | **无状态 HTTP/A2A**，**利用率高**，**推荐为默认** |
| **会话租约（亲和或独占）** | 会话活跃期内 **锁定或优先绑定** 某一副本；**租约释放** 后副本回池 | **有状态重、不宜迁移** 的部署；须配 **空闲超时、显式结束会话、最长租约** |

**租约释放条件（须同时满足，不能只依赖「用户点结束」）**：

- 用户 **显式结束会话**；或  
- **空闲超时**（无消息超过阈值）；或  
- **最长租约**到期。  

**说明**：关浏览器标签 **不等于** 必然触发结束会话；须 **服务端会话状态 + TTL** 协同。

**记忆与实例释放的关系**：

- **会话级进程内缓存**：租约结束或会话结束时 **应丢弃**，避免下一请求误用。  
- **跨会话长期记忆**：必须落在 **外置存储**，键含 **`platform_user_id`**（及策略允许的 conversation 维度）；**不能**依赖「实例释放」保证隐私——下一请求可能打到 **另一副本**，仍须读到 **同一用户隔离数据**。

#### 5.1.7 三条铁律与验收要点

1. **容量诚实**：上架须 **声明或接受保守默认**；平台 **排队/超时/429** 行为与文案一致。  
2. **调用可隔离**：每调 **必带** 平台用户/会话/运行标识；**禁止** 默认可跨用户记忆。  
3. **调度可选但语义清晰**：**无状态默认池化**；**有状态** 须显式选 **租约模式** 并配置 **释放条件**。

**V1.3 相关验收（与目录/调用 Epic 挂钩，可分期）**：至少 **一种调度路径**（池化或租约）端到端可演示；上架表单或 Agent Card **可展示** `max_concurrent` / 队列策略 / 记忆声明中的 **必填项或默认值**；帮助中心与开发者文档与本节 **无矛盾**。

#### 5.1.8 临时会话（A）与定时召回（B）：并存与不事故原则

消费者常见两类需求，须在 **调用语义、记忆边界、实例占用** 上 **显式区分**，避免「定时任务串进别人会话」「会话结束仍被自动唤起」「长期霸占实例」等重大事故。

| 类型 | 典型场景 | 平台侧核心标识（须落协议/日志） | 生命周期与实例占用 |
|------|----------|----------------------------------|---------------------|
| **A｜临时会话型** | 聊完即走；会话结束即 **不再** 代表用户调用该 Agent | **`conversation_id`** + **`user_id`** + **`run_id`**（每轮或每子任务）；`invocation_source = session` | 与 **§5.1.6** 一致：会话结束 / 空闲超时 / 最长租约 **释放租约**；**池化** 模式下无独占则仅结束 **本会话上下文**。会话结束后 **默认不再** 向该 Agent 发请求，除非用户 **新开会话**。 |
| **B｜定时召回型** | 如「每早新闻汇总」；**不依赖** 用户当时是否打开聊天窗 | **`subscription_id`（或 `task_id`）** + **`user_id`** + **每次触发的 `run_id`**；`invocation_source = scheduled` | **禁止** 为定时任务持有 **跨天跨夜的实例租约**。每次调度 **仅** 占用实例 **至该次 run 完成（+ 超时）**，完成后 **归还池**；与 **临时会话** 解耦。 |

**记忆边界（防串味、防泄漏）**

| 类型 | 会话内上下文 | 跨次 / 持久偏好 |
|------|----------------|-----------------|
| **A** | **默认** 仅写入 **本会话** 可见的记忆分区（`user_id + conversation_id`）；会话归档结束后 **按策略冻结或淘汰**（产品规则须在帮助中心说明）。 | 若用户开启「记住我的偏好」，写入 **用户级** 分区，**不得** 默认等同于「把本次闲聊写入所有未来定时任务」。 |
| **B** | 每次 run 可有 **短时 run 上下文**；展示上建议 **归入订阅任务线程** 或 **带系统标签的独立时间线**，与 **用户当时正在聊的 A 会话** 在 UI 上 **可区分**。 | **订阅级 / 用户级** 配置（栏目、语气、时区、推送渠道等）存放在 **`user_id (+ subscription_id)`** 键空间，**禁止** 与 **他人** 或 **无关 conversation** 共享检索。 |

**防重大事故的硬规则**

1. **来源不可混**：调度器下发 B 类调用时 **必须** 带 `invocation_source=scheduled` 与 **唯一 `run_id`**，**禁止** 复用他人 `conversation_id` 或静默挂载到随机活跃会话。  
2. **用户可控**：B 类须 **可暂停、可删除、可改频**；暂停后 **立即** 停止触发（含边缘时钟漂移下的 **幂等与去重**）。  
3. **容量与公平**：B 类与 A 类 **共用** 同一 Agent 实例池与 `max_concurrent` 时，须有 **公平策略**（如单用户订阅并发上限、全局每 Agent 调度速率上限），避免定时洪峰挤死在线聊天。  
4. **对上架者透明**：Agent Card 或上架契约中可声明 **是否接受 scheduled 调用**、**建议最长 run 时长**；若 Agent **仅支持会话型**，平台 **不得** 对其实强行排定时（或仅允许「创建草稿到用户草稿箱」等弱形态）。  
5. **审计与收据**：A/B 均在审计中 **可区分来源**，便于追责与客服排查。

**与 §5.1.6 的衔接**：**临时会话** 可优先 **租约**（有状态 Agent）或 **池化**（无状态）；**定时召回** **默认池化短占**，**不** 将会话租约模型套在「隔夜订阅」上。

#### 5.1.9 V1.3 平台落地清单 vs 开发者协议优化（平台多扛、上架少改）

本节约定：**§5.1～§5.1.8** 中 **能由主网 / 网关 / 调度 / 控制台完成的，一律平台实现**；开发者侧以 **「接标准头/标准 JSON 上下文 + 诚实声明」** 为 **最小合规**，避免要求每个上架者自建队列、自写路由或自解析复杂协议。

##### （一）V1.3 平台开发须实现的能力（主线团队）

| 域 | 交付项 | 说明 |
|----|--------|------|
| **目录与上架模型** | **逻辑 Agent** 与 **实例（endpoint）列表** 的数据模型与 API | 消费者见 **一张卡片**；后台维护 **N 个同构 URL**、健康检查、下线实例；支持 **默认 `max_concurrent`**（未填则按策略兜底，见 §5.1.3）。 |
| | 上架表单 / Provider 控制台字段 | **`max_concurrent`、`queue_behavior`、`timeout`、调度模式（池化/租约）、`supports_scheduled`、记忆声明枚举、`suggested_max_run_duration`** 等；**未填项用平台默认值**，减少开发者决策负担。 |
| **调用网关** | **统一注入上下文** | 每次 A2A/HTTP 转发至 Agent 时，由平台 **自动附加**（Header 或规范 JSON 字段，以 **`Agent-IM-Invocation-Context` 协议版本** 定稿为准）：`gaia_user_id`、`conversation_id`（A 类）、`run_id`、`invocation_source`（`session` \| `scheduled`）、`subscription_id` / `task_id`（B 类）、`trace_id`。开发者 **不应** 自行从 Cookie 猜用户身份。 |
| | **排队、超时、429、重试策略** | 平台侧 **全局实现**；达到并发上限时 **排队或快速失败** 由声明与策略决定；**不向开发者暴露「请自建队列」** 作为默认路径。 |
| **调度** | **池化路由** | 在无租约模式下，从 **可用实例池** 选取 endpoint；失败 **切换副本**（若有多实例）。 |
| | **租约（可选 V1.3 分期）** | 若 V1.3 仅交付池化，租约可标 **Phase 1.1**；若同期交付，须含 **空闲超时、最长租约、显式结束** 释放逻辑（§5.1.6）。 |
| **定时任务（B 类）** | 与订阅/任务系统联动 | 调度触发时 **强制** 生成 **唯一 `run_id`**，**禁止** 绑定错误 `conversation_id`；**暂停/删除/改频** 即时生效；**幂等键** 防重复执行；与 **§5.1.8** 公平策略、速率上限一致。 |
| **公平与容量** | A/B **共用容量** 时的策略 | 如：单用户订阅并发上限、每 Agent 每分钟调度上限、聊天 **不** 被定时任务 **饿死** 的保底策略（具体算法 Epic 定稿）。 |
| **可观测** | 审计与收据字段 | 审计事件 **可区分** `invocation_source`；收据或执行记录可关联 **`run_id` / `subscription_id`**，便于客服与开发者排障。 |
| **开发者体验** | **协议文档 + OpenAPI 片段 + 多语言 SDK 中间件** | 提供 **复制即用** 的「从请求读出 `Invocation-Context`」示例（如 Hono/Express/FastAPI 中间件）；**Agent Card** 可由控制台 **表单生成 JSON**（`/.well-known/agent.json` 托管可选：平台代管或校验外链）。 |
| **合规与测试** | **上架前可选探测** | 对声明 **无状态** 的 Agent 提供 **轻量连通性 + 头透传** 自检；**记忆隔离** 抽检纳入 **§4 质量轨**（不替代开发者责任，但降低「上了才发现串话」的概率）。 |

**V1.3 分期说明**：若订阅系统（B 类）工程过大，可 **V1.3 先交付 A 类全链 + 平台注入 + 队列 + 逻辑 Agent/多实例**；B 类以 **「调度接口已预留 + 文档」** 或 **最小 MVP（单 cron、单 Agent）** 交付，但 **须在 Epic 中写清**，且 **不得在用户侧假装已支持完整订阅**。

##### （二）开发者协议与上架流程优化（尽量减小 OpenClaw 类负担）

**原则**：**默认少填、默认安全、复杂逻辑平台做**；开发者只做 **「能响应带上下文的请求 + 按 key 隔离状态」**。

| 优化项 | 平台承担 | 开发者最小义务 |
|--------|----------|----------------|
| **身份与会话** | 网关 **注入** `user_id` / `conversation_id` / `run_id` / `invocation_source` 等 | 从 **标准头或标准 body 扩展** 读取；**禁止** 依赖未文档化的侧信道。 |
| **并发与排队** | 平台 **排队、限流、超时** | 在 Card/表单中 **声明** `max_concurrent` 或接受默认 **1**；**无需** 自建消息队列。 |
| **多副本** | 平台 **注册多个 endpoint**、健康检查、路由 | 开发者 **部署多份同构服务** 并 **提交 URL 列表**（或一条 **自动发现** webhook，若 V1.3 实现）；**无需** 改代码区分「第几号副本」。 |
| **定时任务** | 平台 **调度器**、幂等、`run_id`、与用户订阅绑定 | 若 **不支持**：表单勾选 **`supports_scheduled: false`**，平台 **不** 对该 Agent 排 B 类；若支持：实现 **单次 run 在超时内完成**、**不假设** 长期占实例。 |
| **Agent Card** | 控制台 **表单 → 生成 Card 草稿** + 字段校验 | **核对** 能力与端点；可选 **仅维护表单** 由平台 **发布 Card**。 |
| **SDK** | 官方维护 **中间件** 解析上下文对象 | 业务代码只接 **`ctx.gaiaUserId` / `ctx.conversationId`** 等 typed 字段。 |
| **记忆** | 平台文档与评测 **明确违规后果**；可选提供 **托管记忆 API**（若 V1.3 或后续里程碑落地） | 自管记忆时 **必须** 使用平台下发的 **键规范**（`user_id` + `conversation_id` 等）；**不得** 全局单库混用户。 |

**协议构成建议（V1.3 定稿目标）**

1. **一份《Invocation Context》规范**（版本号）：固定字段名、类型、在 A2A 与直连 HTTP 中的承载位置（Header 名、JSON path）。  
2. **Agent Card 扩展块 `gaialynk.routing`**（名称可调整）：`max_concurrent`、`pooling_preferred`、`supports_scheduled`、`memory_tier` 等，**与控制台表单双向同步**。  
3. **最小合规路径（文档醒目标出）**：`max_concurrent=1` + 每请求读上下文 + 会话键隔离 → 即可上架 **MVP**；扩容路径为 **加同构实例 URL**，**无需** 先改代码为多线程。

**明确不要求开发者（除非另购/另约）**

- 自建 **全局队列服务**、**自实现跨用户公平**、**自解析平台内部 trace**。  
- 在 V1.3 首版 **实现租约回调**（若平台侧租约未交付）。  
- 为通过上架 **编写自定义 TLS 客户端补丁**（使用平台标准 HTTPS + 标准头即可）。
