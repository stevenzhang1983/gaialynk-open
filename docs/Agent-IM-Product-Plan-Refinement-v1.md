# Agent IM 产品规划细化（Refinement v1）

> 基于 `Agent-IM-Product-Plan.md` 的深化版本  
> 目标：把“初心回归（普通用户零门槛拿结果）”落成可执行的产品细节、界面方案与治理机制。

---

## 1. 本文要解决的问题

当前主规划已经明确了方向，但在执行层还缺少一层“产品操作系统”级别的细化：

1. 哪些关键细节尚未被规划，可能在上线后成为体验断点或治理风险。  
2. Agent 与 Agent 的 A2A 通讯是否要可视化、为谁可视化、可视化到什么粒度。  
3. 人对 A2A 通讯如何审核（触发条件、证据面板、审批动作、追责链路）。  
4. Web 为主 + 本地轻连接器路线下，界面信息架构怎么设计。  
5. 如何用更完整的用户故事保证“临时需求 + 日常自动化 + 复合任务”三类场景并行成立。

---

## 2. 产品原则（执行版）

1. **结果优先**：每个新功能必须回答“普通用户是否更快拿到可用结果”。  
2. **默认可解释**：关键路由、关键调用、关键拦截都要能解释“为什么”。  
3. **默认可治理**：高风险能力必须可审、可断、可追溯、可回滚。  
4. **分层复杂度**：普通用户看“结果与动作”，专业用户看“流程与证据”，开发者看“契约与指标”。  
5. **渐进暴露**：首屏不暴露协议复杂度，深层视图再开放 A2A 细节。

### 2.1 平台边界（新增）

- 平台核心职责是：**可信调用、编排、路由、观测、治理与结算**。  
- 平台不直接提供“每个用户独特需求”的最终方案逻辑，这部分由平台上的 Agent 能力供给者承担。  
- 产品设计应优先保证“通用调用体验 + 通用治理机制”，避免平台侧过早承接垂直业务实现。

## 3. 还没规划充分的关键细节清单

### 3.1 供给侧（发布者上架）遗漏项

> 供给侧范围扩展：不只专业开发者，凡是有能力**独立部署 Agent 并连接平台供他人调用**的用户，均可成为发布者。

- **能力契约版本化（必须）**：每个 Agent 模板声明 `major.minor`；`major` 升级默认不影响既有订阅实例，必须先创建新版本并允许用户迁移。  
- **可调用健康检查（必须）**：上架前完成 `连通性、平均延迟、错误率、限流行为` 四项基测，未达标不公开上架。  
- **最小质量评测（必须）**：每个类目至少 10 条标准样例，达成最低通过率才可上架；支持周期复测和自动降权。  
- **失败语义标准化（必须）**：统一错误码到用户文案层（例如：超时、权限不足、依赖故障、输入不完整），禁止裸露技术报错。  
- **可解释定价（应做）**：统一展示“预估费用区间 + 计费维度 + 触发条件 + 免费额度消耗”。  
- **下架与替换策略（必须）**：已订阅用户必须收到替代方案（同类模板推荐、参数迁移、历史保留策略）。  
- **信誉与排名机制（应做）**：排名不只看调用量，加入成功率、复用率、投诉率、撤销率等质量信号。  
- **开发者运维面板（应做）**：最小提供调用量、成功率、失败类型、平均耗时、被收藏/复用等指标。

### 3.5 可信调用与真实评价闭环遗漏项（新增）

- **发布者身份分级（必须）**：匿名发布者、实名发布者、认证发布者分层，影响默认可见性与调用上限。  
- **能力声明可验证（必须）**：发布者声明能力后，平台需通过标准测试样例做最小验真，不可只信自报。  
- **线上表现持续校验（必须）**：按 7/30 天窗口持续计算成功率、超时率、撤销率、投诉率，动态影响推荐权重。  
- **评价防刷机制（必须）**：评分仅作为弱信号，需结合真实调用事件、留存与复用信号；异常评分触发风控。  
- **调用后反馈结构化（应做）**：用户反馈至少分“结果质量、速度、稳定性、是否符合预期”四维。  
- **争议处理链路（应做）**：用户对结果不满可发起争议，平台支持证据回放、仲裁、补偿或拒付策略。  
- **风险分级调用策略（必须）**：高风险调用默认更严格（确认、人审、限制范围），低风险调用保证流畅性。  
- **降权与下架机制（必须）**：连续异常、投诉高发、策略违规的 Agent 自动降权，必要时下架并通知受影响用户。

### 3.2 消费侧（普通用户）遗漏项

- **输入增强（必须）**：自然语言输入外，补齐 `附件、目标格式、截止时间、预算上限` 四个结构化参数。  
- **路由可控（应做）**：用户可选“自动选择 Agent / 手动指定 Agent / 禁止某类 Agent”。  
- **结果可信展示（必须）**：结果页至少包含“调用路径摘要、耗时、费用、关键依据、风险提示、一键复跑”。  
- **失败回退体验（必须）**：失败后给出三类动作：`重试同路径`、`切换替代 Agent`、`降级到简化结果`。  
- **订阅任务生命周期（必须）**：创建、暂停、恢复、修改参数、删除、查看历史、导出。  
- **通知策略（应做）**：支持 App 内、邮件、IM 推送；可配置“仅异常通知/每次都通知”。  
- **用户资产沉淀（应做）**：沉淀“常用提示词片段、常用模板参数、个人偏好配置”。  
- **冷启动体验（必须）**：新用户首日必须在 3 分钟内完成首次可见结果（Time to First Result）。

### 3.3 治理与合规遗漏项

- **责任归因模型（必须）**：每次执行写入“模板版本、输入来源、策略命中、输出责任提示”，便于追责。  
- **敏感领域门槛（必须）**：医疗、法律、投资等高敏类目默认展示“非专业建议声明 + 风险确认”。  
- **版权与数据边界（必须）**：明确用户输入默认不用于再训练；输出版权归属与二次分发规则公开透明。  
- **未成年人保护（应做）**：情绪、健康、消费等场景补充年龄提示与内容安全分级。  
- **滥用防控（必须）**：针对刷调用、撞库、提示词绕过、恶意自动化建立实时限流与黑名单。  
- **申诉与纠错（应做）**：用户可对错误结果发起申诉，平台给出处理时限与回执。  
- **高风险拦截治理（应做）**：高风险动作触发确认后，超过时限自动失效，避免“滞后审批”误放行。

### 3.4 本地能力路线遗漏项

- **权限颗粒度模型（必须）**：按 `目录、应用、动作` 三层授权，默认最小权限。  
- **授权时效模型（必须）**：支持一次性授权、会话期授权、长期授权，长期授权可随时撤销。  
- **高风险动作二次确认（必须）**：涉及删除、覆盖、发送、系统设置修改时必须二次确认。  
- **本地执行证据（必须）**：动作回传结构化收据（时间、动作、参数摘要、结果、执行环境）。  
- **离线与断网降级（应做）**：连接器离线时提供“排队执行 / 取消执行 / 转云端替代”的可选策略。  
- **多设备一致性（应做）**：同一账号多设备授权策略独立管理，避免跨设备权限意外继承。  
- **隐私模式（应做）**：支持“本地数据只本地处理、不上传原文”模式，并在 UI 中显式提示。

## 4. A2A 通讯是否需要可视化：结论与方案

## 4.1 结论

**需要可视化**，但必须分层，不同角色看不同深度。

- 普通用户：看“做了什么、为什么、是否安全”。  
- 重度个人用户：看“谁调用了谁、何时、风险如何、是否经审批”。  
- 开发者：看“请求/响应结构、错误点、性能与成本”。

## 4.2 三层可视化模型

### L1 结果层（默认）
- 用自然语言展示执行路径摘要：  
  - “已调用：行业分析 Agent -> 报告生成 Agent”  
  - “1 次高风险动作已等待你的确认”  
- 适合普通用户，避免协议细节污染主流程。

### L2 流程层（进阶）
- 时间线 + 简图：谁发起、谁接收、每步状态（成功/失败/待审）。  
- 支持查看每步输入摘要、输出摘要、耗时与成本。  
- 适合重度个人用户与内容创作者。

### L3 证据层（专家）
- 原始 A2A 包结构、reason codes、策略命中、签名与收据引用。  
- 支持导出审计包。  
- 适合开发者、风控、法务审计。

## 5. 人对 A2A 通讯如何审核（HITL 审核体系）

## 5.1 审核触发规则

- 按风险等级触发：`high`/`critical` 默认审。  
 - 按能力类别触发：支付、发信、外呼、文件批量改写、本地终端控制。  
- 按数据敏感度触发：含隐私、凭证、合同、财务数据。  
- 按异常行为触发：短时高频调用、策略违规高发、历史信誉下降。

## 5.2 审核队列设计

每条待审任务必须至少展示：

- 触发原因（规则命中）。  
- 调用链摘要（发起方 -> 被调用方）。  
- 拟执行动作（用户可读）。  
- 影响范围（数据范围/设备范围）。  
- 风险提示与建议动作。  
- 审批时限（SLA）。

## 5.3 审核动作模型

- `approve`：放行，可附加约束（只读、限时、限范围）。  
- `reject`：拒绝并返回原因。  
- `ask_more_info`：要求补充上下文后再审。  
- `delegate`：转交更高权限审批人。  

所有审核动作都要生成收据并进入审计链。

## 5.4 审核界面关键组件

- **左侧**：任务列表（按风险和时效排序）。  
- **中部**：任务详情与调用链可视化。  
- **右侧**：证据面板（策略命中、历史行为、关联记录）。  
- **底部**：审批动作条（审批、拒绝、附加限制条件）。

## 6. Web 主体 + 本地轻连接器的信息架构与界面草案

## 6.1 顶层信息架构（IA）

1. **Home / Ask**：输入需求、查看即时结果。  
2. **Tasks**：任务中心（临时任务 + 订阅任务）。  
3. **Agents**：可调用的专业 Agent 市场。  
4. **Approvals**：待审队列（按权限可见）。  
5. **History**：执行历史、可视化回放与导出。  
6. **Connector**：本地连接器状态、权限与授权记录。  
7. **Settings**：账号、空间、安全与计费。

## 6.2 关键页面交互定义

### Ask 页（核心转化页）
- 输入框支持：文本 + 附件 + 目标输出格式（可选）。  
- 提交后展示“执行中卡片”：路由到哪些 Agent、预计时长。  
- 结果区提供：成品输出、可解释路径、再次优化按钮。

### Tasks 页
- 双 Tab：`临时任务` / `订阅任务`。  
- 订阅任务卡片包含：下次执行时间、最近一次结果、状态（运行/暂停/异常）。  
- 支持一键暂停/恢复，异常自动提示修复动作。

### Agents 页
- 分类浏览 + 搜索 + 风险标签 + 成功率。  
- Agent 详情页展示：能力边界、输入要求、费用、示例结果、风险等级。  
- “快速试用”入口直接回到 Ask 页并带预填模板。

### Approvals 页
- 仅对有权限的用户展示。  
- 支持批量审低风险、单条审高风险。  
- 审批后自动生成说明并通知发起者。

### Connector 页
- 连接状态：未安装 / 已安装未授权 / 已授权。  
- 权限矩阵：文件、应用、终端命令等分组。  
- 审计回放入口：查看本地动作执行证据。

## 7. 领域模型细化（供产品与研发对齐）

## 7.1 核心对象

- `PublicAgentTemplate`：开发者上架能力模板。  
- `UserTaskRequest`：用户一次性请求。  
- `UserTaskInstance`：用户订阅任务实例（带归属）。  
- `ExecutionRun`：一次执行运行记录。  
- `ApprovalTicket`：待审单。  
- `A2AInteractionTrace`：跨 Agent 通讯链路。  
- `LocalActionReceipt`：本地连接器动作收据。

## 7.2 状态机（简化）

- 即时任务：`created -> routing -> running -> succeeded|failed|need_approval`  
- 订阅任务：`draft -> active -> paused -> archived`  
- 审核单：`pending -> approved|rejected|expired`

## 7.3 网络化协作会话模型（新增）

> 目标：把“人和 Agent 的多种协作关系”收敛为统一可治理的会话拓扑，而不是为每种场景单独造系统。

### 7.3.1 五类核心会话拓扑（提炼）

| 拓扑ID | 会话形态 | 典型结构 | 核心价值 | 核心风险 |
|---|---|---|---|---|
| `T1` | 单用户 + 单 Agent | 1 User + 1 Agent | 最低门槛，主流日常使用入口 | 误调用、结果黑箱 |
| `T2` | 单用户 + 多 Agent | 1 User + N Agent | 一人发起，多 Agent 分工协作 | 路由混乱、费用失控 |
| `T3` | 多用户 + 多 Agent | N User + M Agent | 群体协作，跨人跨 Agent 联动 | 权限越界、上下文泄露 |
| `T4` | 多用户 + 单 Agent | N User + 1 Agent | 多人共享一个能力中枢 | 责任归因不清、冲突输入 |
| `T5` | Agent 对 Agent（用户授权） | Agent A <-> Agent B..N | 自动化链路、低人工干预 | 越权代理、不可见执行 |

### 7.3.2 统一会话元数据（必须）

每个会话需包含：

- `conversation_topology`: `T1/T2/T3/T4/T5`  
- `human_participants`: 人类参与者集合  
- `agent_participants`: Agent 参与者集合  
- `authorization_mode`: `user_explicit / policy_based / delegated`  
- `visibility_mode`: `full / summarized / restricted`  
- `risk_level`: `low / medium / high / critical`

这样可保证同一套会话系统覆盖不同拓扑，而不是分叉实现。

### 7.3.3 会话权限与可见性规则（必须）

1. **人类优先可见原则**：默认用户可查看与自己相关的完整调用摘要。  
2. **最小可见原则**：跨用户场景下，仅展示本用户有权查看的输入/输出片段。  
3. **Agent 最小权限原则**：Agent 仅可读取完成当前任务所需上下文，不默认继承会话全量内容。  
4. **代理授权边界**：`T5` 场景必须有明确授权范围（能力、时间、数据域）。  
5. **高风险强制确认**：任意拓扑下高风险动作需确认或人审，不因“自动化链路”豁免。

### 7.3.4 T5（Agent-Only）专项治理（重点）

`T5` 是平台差异化能力，但也是最高风险场景，需专项约束：

- **显式授权票据**：用户授权生成 `delegation ticket`，包含有效期、能力范围、可调用对象。  
- **默认摘要回传**：Agent 间沟通默认向授权用户回传阶段摘要与关键决策。  
- **异常中断机制**：一旦命中策略违规或超预算，自动中断并通知授权用户。  
- **可撤销性**：用户可一键撤销代理授权，立即停止后续 Agent-to-Agent 调用。  
- **收据必达**：关键步骤必须生成可验签收据并关联到原始授权票据。

### 7.3.5 会话 UI 呈现策略（按拓扑自适应）

- `T1`：简洁聊天视图 + 结果卡片（默认）。  
- `T2`：聊天视图 + “协作分工面板”（显示每个 Agent 当前角色与进度）。  
- `T3/T4`：聊天视图 + “参与者权限提示” + “谁在代表谁操作”的标识。  
- `T5`：默认展示“代理执行时间线”，弱化普通聊天气泡，强化授权与审计信息。

### 7.3.6 会话切换与升级规则（应做）

- 会话可从 `T1` 升级到 `T2/T4`（邀请更多 Agent 或用户）。  
- 升级时必须提示新增权限影响与可见性变化。  
- 降级（移除 Agent/用户）后，新权限立即生效，历史数据按原可见规则保留。

### 7.3.7 会话级指标（新增）

- 各拓扑使用占比（T1-T5）。  
- 多 Agent 协作成功率（T2/T3/T5）。  
- 授权撤销率与撤销后违规调用率（T5）。  
- 多参与者会话冲突率（T3/T4）。  
- 各拓扑平均结果交付时间与费用可控率。

## 8. 进一步细化的用户故事（新增）

## 8.1 个人用户：通用调用体验

1. 作为个人用户，我希望只输入自然语言需求即可调用平台上的 Agent，而不用理解部署与协议细节。  
2. 作为个人用户，我希望系统在多 Agent 可选时自动路由最合适的能力，并告诉我选择依据。  
3. 作为个人用户，我希望一次调用失败时，系统自动给出可用替代 Agent 和重试选项。  
4. 作为个人用户，我希望调用前看到大致耗时和费用区间，避免“黑箱调用”。  
5. 作为个人用户，我希望调用完成后看到可复用的调用路径，便于下次一键复跑。  
6. 作为个人用户，我希望上传输入材料后，平台自动分发给合适 Agent，不要求我手动拆分任务。

## 8.2 个人用户：日常与临时任务并行

7. 作为个人用户，我希望同一个需求既可“立即执行一次”，也可“保存为周期任务”。  
8. 作为个人用户，我希望周期任务支持暂停、恢复、修改与删除，并保留历史执行记录。  
9. 作为个人用户，我希望周期任务异常时能第一时间收到通知并可一键修复。  
10. 作为个人用户，我希望不同类型任务在同一个任务中心统一管理，而不是分散在多个页面。

## 8.3 个人用户：可信调用与可解释

11. 作为个人用户，我希望看到本次调用涉及哪些 Agent、每步状态和最终结果来源。  
12. 作为个人用户，我希望高风险动作默认需要我确认，避免误调用。  
13. 作为个人用户，我希望调用结果可回放，便于核对争议。  
14. 作为个人用户，我希望能查看 Agent 的真实表现指标（成功率、稳定性、投诉率）再决定是否调用。  
15. 作为个人用户，我希望对结果不满意时能一键反馈并进入争议处理流程。

## 8.4 发布者（供给者）接入与运营

16. 作为发布者，我希望即使不是专业开发者，只要能独立部署 Agent，也能按标准接入平台供他人调用。  
17. 作为发布者，我希望接入流程清晰：身份校验、能力声明、健康检查、上架发布。  
18. 作为发布者，我希望看到自己 Agent 的调用量、成功率、投诉率和收益数据。  
19. 作为发布者，我希望在升级能力版本时，不影响既有用户任务，平台提供迁移路径。  
20. 作为发布者，我希望知道哪些行为会触发降权或下架，规则透明且可申诉。

## 8.5 本地能力与隐私控制

21. 作为个人用户，我希望本地能力默认最小权限，只开放我授权的目录/应用/动作。  
22. 作为个人用户，我希望高风险本地动作必须二次确认并可随时撤销授权。  
23. 作为个人用户，我希望本地连接器断线时，平台明确告诉我任务会如何处理（排队/取消/降级）。  
24. 作为个人用户，我希望在隐私模式下，本地原始内容不上传云端，只上传必要摘要。  
25. 作为个人用户，我希望每次本地执行都有可追溯记录，清楚知道“何时、何动作、何结果”。

## 9. MVP-Plus（下一阶段）功能切分

## 9.1 必做（P0）

- Ask 主路径：需求输入 -> 自动路由 -> 结果输出。  
- Agent 市场最小可用：搜索、详情、风险标签、快速试用。  
- 订阅任务最小闭环：创建、暂停、删除、历史。  
- 审核队列最小闭环：触发、审批、收据、通知。  
- A2A 可视化 L1 + L2。  
- 本地连接器最小能力（1-2 个高价值动作）。

## 9.2 应做（P1）

- A2A 可视化 L3（证据层）。  
- 订阅任务模板化与批量管理。  
- 审核策略可配置（空间级）。  
- 连接器权限分级与授权时效策略。

## 9.3 可后置（P2）

- 复杂可视化编排器。  
- 全量桌面自动化能力。  
- 多端原生客户端矩阵。  
- 高级治理报表与全自动风控编排。

## 10. 指标补充（针对本次细化）

- **用户结果效率**：首次结果时间（TTFR）、任务完成率。  
- **用户可控性**：订阅任务管理动作成功率（暂停/恢复/删除）。  
- **治理有效性**：高风险任务拦截准确率、审批 SLA 达成率。  
- **可解释性**：用户打开执行路径详情比例、路径详情满意度。  
- **本地安全性**：连接器越权拦截率、授权撤销成功率。

## 11. 风险与设计对策

1. **界面过重风险**：可视化太复杂影响普通用户。  
   - 对策：默认只展示 L1，逐层展开。  
2. **审批疲劳风险**：高频提示导致“无脑通过”。  
   - 对策：风险分级 + 批量规则 + 智能去重。  
3. **本地权限恐惧**：用户担心越权。  
   - 对策：权限最小化、一次一授权、强可撤销。  
4. **Agent 质量不稳定**：首批供给毁掉口碑。  
   - 对策：上架门槛 + 灰度流量 + 可替换回退。
