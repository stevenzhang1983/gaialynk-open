## 一、产品定位与愿景

### 1.1 产品名称（建议）

**Agent IM** / **Agent Hub** / **Agent Space**（可后续定名）

### 1.2 一句话定位

**让不同框架、不同厂商的 AI Agent 像人在 IM 里一样发现彼此、加入会话、群聊协作，并与人类在同一对话中协同；平台提供发现、信任、编排、可观测与商业化能力。**

### 1.3 网络定位与愿景

- **网络定位**：我们搭建的是**以 Agent–Agent、Agent–人 为核心，以人–人为辅助**的网络。主网（我们运营的托管平台）作为 **Hub**，负责协调、发现、中继与可选计费；社区**自建节点**可通过开放协议**连接主网**，与主网及其他连接节点共同形成统一大网——连接后，节点上的 Agent/人 可与主网及全网 Agent/人 发现并协作，人–人作为同一网络上的辅助能力（如同一身份体系下的私信）不喧宾夺主。
- **愿景**：成为「Agent 互联互通」的基础设施：任何 Agent 只要对齐 A2A，即可接入会话、群组、目录与计费体系，与人和其它 Agent 自然协作；自建节点**可连、愿连**主网，共享发现与跨节点协作能力。

### 1.4 目标用户

| 维度 | 描述 |
|------|------|
| **目标用户** | 企业（多 Agent 协同、人机协同、合规审计）、Agent 开发者（发布、托管、变现）、最终用户（通过统一会话与多 Agent 交互）、**自建节点运营者**（连接主网以获得发现与跨节点协作）。 |

### 1.5 与 A2A 的关系

- **协议层**：完全采用或严格对齐 Google A2A（Agent Card、JSON-RPC 2.0、SSE、Task 生命周期、Parts 多模态）；并在本项目中**扩展** Agent 类型（见 1.6）。
- **平台层**：在 A2A 之上提供 A2A 未覆盖的能力：会话/线程/群聊、人类参与、发现目录、编排、记忆、信任、可观测、计费、与人类 IM 桥接、**逻辑 Agent / 执行 Agent 分类及硬件与物理世界编排**等。

### 1.6 Agent 类型与协议扩展：逻辑 Agent / 执行 Agent

- **逻辑 Agent（Logical Agent）**：能够通过背后的大模型进行推理并输出结论的 Agent；此前规划中所述的 Agent 均属此类。逻辑 Agent 参与会话、理解意图、决策并可能向执行 Agent 下发指令。
- **执行 Agent（Execution Agent）**：不依赖大模型推理的**简单执行者**，仅负责接收逻辑 Agent（或经平台转发的）指令、执行并反馈执行结果；不产生自主结论。典型形态：智能家居设备、传统家电、传感器、执行器等。
- **协议扩展**：在对外开放的 Agent 协议中，对 Agent 类型做显式区分（如 Agent Card 或目录元数据中的 `agent_type: logical | execution`），以便发现、路由与编排时区分「谁可下发指令」「谁仅可被调用执行」。协议设计时需考虑**众多传统设备厂商**（如智能家居中的传统家电厂商）以较低门槛接入为**执行 Agent**，无需承载大模型能力即可接入平台、接收指令并上报结果。
- **与硬件的关系**：执行 Agent 常对应硬件设备或设备上的轻量固件；逻辑 Agent 可为纯软件或「软硬一体」中的云端/端侧大脑。**自有拓展坞形态的硬件不是执行 Agent，而是具备逻辑 Agent 能力的用户设备 / 硬件终端**，并可调用平台上的执行 Agent。详见第十节「逻辑 Agent / 执行 Agent 与硬件及物理世界」与第九节赛道八。

### 1.7 八大赛道总览

| 序号 | 方向 | 核心价值 | 关键产出 |
|------|------|----------|----------|
| 1 | **IM 级体验** | 会话、群聊、人在环、在线状态 | 对话式协作体验 |
| 2 | **发现与网络** | 目录、能力匹配、公私混合 | Agent 生态与可发现性 |
| 3 | **信任与合规** | 审计、数据边界、人审；**信任增强**（收据、TEE、声誉）；**安全增强**（提示词注入防护） | 企业级安全与合规 |
| 4 | **编排与语义** | 会话编排、意图路由、记忆 | 智能协作与连贯性 |
| 5 | **可观测与运维** | 回放、成本、人可读视图 | 运维与成本可控 |
| 6 | **生态与商业化** | 托管、计费、模板 | 生态与收入 |
| 7 | **前沿与差异化** | 实时音视频、多租户、IM 桥；**逻辑/执行 Agent 与硬件及物理世界** | 差异化与扩展场景 |
| 8 | **硬件 Agent 与平台融合** | 自有硬件（拓展坞形态、即插即用）与 Agent IM 的产品融合、身份与绑定、营销与节奏 | 软硬一体、零摩擦入口 |

## 二、赛道一：IM 级体验（Conversation-First）

### 2.1 目标

在 A2A 的「任务/请求-响应」之上，建立持久的**会话与线程**模型，支持多 Agent **群聊/频道**，并让**人类**作为一等公民参与同一会话，提供接近 IM 的体验（在线状态、邀请、@ 等）。

### 2.2 核心概念定义

- **Space（空间）**：租户或团队级别的容器，下有多个 Conversation。
- **Conversation（会话）**：一条持续存在的对话线，拥有唯一 ID，包含多个 Participant（人或 Agent），有序的 Message/Event 流。
- **Thread（线程）**：会话内对某条消息的回复链，用于分支讨论。
- **Participant**：人（User）或 Agent；Agent 通过 A2A Agent Card 标识，人通过平台账号标识。
- **Message**：会话内一条可展示的单元；可由 A2A Task/Message 映射而来，或由平台生成（系统消息、审批结果等）。

### 2.3 功能清单

| 功能模块 | 描述 | 优先级 |
|----------|------|--------|
| 会话 CRUD | 创建/关闭/归档会话；列表与筛选（按参与者、标签、时间）。 | P0 |
| 会话内消息流 | 消息有序追加、分页拉取、实时推送（基于 SSE/WebSocket）；支持 text、Parts 多模态。 | P0 |
| A2A ↔ 会话映射 | 将 A2A Task 的创建/更新/完成映射为会话内 Message 或 Event；反向：会话内「@Agent 执行 X」触发 A2A 请求。 | P0 |
| 线程（回复链） | 消息可回复某条消息形成 thread；UI 与 API 支持按 thread 折叠/展开。 | P1 |
| 多参与者（群聊） | 会话支持多个 Participant；邀请/移除参与者；权限（仅读/可发/管理员）。 | P0 |
| @ 提及 | 消息中 @Agent 或 @User，解析后用于通知与路由（如仅被 @ 的 Agent 响应）。 | P1 |
| 人在会话中 | 用户作为 Participant 加入会话；可发消息、审批、打断；消息流统一展示人+Agent。 | P0 |
| 在线状态 | Agent：可上报 busy/available/dnd；User：在线/离开；展示在会话成员列表与邀请界面。 | P1 |
| 邀请与呼叫 | 从会话内「邀请 Agent」：根据目录或 URL 拉取 Agent Card，发起 A2A 连接并加入当前会话。 | P1 |
| 会话模板 | 预置「客服+知识库」「代码评审组」等模板，一键创建带默认参与者的会话。 | P2 |

### 2.4 数据模型要点

- **conversations**：id, space_id, title, created_at, updated_at, metadata, state(active/archived/closed)。
- **participants**：conversation_id, participant_type(user|agent), participant_id, role, joined_at。
- **messages**：id, conversation_id, thread_id(nullable), sender_type, sender_id, content(JSON, 含 Parts), a2a_task_id(nullable), created_at。
- **presence**：participant_key, status, updated_at（可短 TTL 缓存）。

### 2.5 阶段规划

- **Phase 1（MVP）**：单会话、多参与者（人+Agent）、消息流、A2A 与消息双向映射、基础邀请。
- **Phase 2**：线程、@、在线状态、邀请流程增强。
- **Phase 3**：会话模板、高级权限、会话归档与搜索。

## 三、赛道二：发现与 Agent 网络（Discovery & Network）

### 3.1 目标

在 `/.well-known/agent.json` 与 A2A Agent Card 之上，提供**可搜索的 Agent 目录**、**能力标签与自动匹配**，以及**私有/公网混合**的可见性与路由策略。

### 3.2 核心概念

- **Agent 目录**：平台维护的 Agent 注册表；条目可来自自注册、管理员录入、或对已知 URL 拉取 Agent Card 的缓存；支持**逻辑 Agent**与**执行 Agent**的类型区分（见 1.6），便于传统设备厂商以执行 Agent 身份接入。
- **Agent 条目**：身份、端点、能力（Skills）、标签（领域、输入输出、SLA、计费）、可见性（公开/指定租户/内网）；**agent_type**（logical | execution）用于发现与路由。
- **能力匹配**：根据当前任务描述或用户意图，推荐或自动选择最合适的 Agent（单或组合）。

### 3.3 功能清单

| 功能模块 | 描述 | 优先级 |
|----------|------|--------|
| Agent 注册 | Agent 提供方通过控制台或 API 注册；填写端点、补充名称/描述/标签；支持从 URL 拉取并合并 Agent Card。 | P0 |
| 目录浏览与搜索 | 按名称、领域、能力、框架、厂商筛选；列表与详情（含 Agent Card 与扩展元数据）。 | P0 |
| 能力标签体系 | 结构化标签：domain（如 finance, dev, support）、input/output 类型、max_latency、auth_required、billing_model 等。 | P1 |
| 一键加入会话 | 从目录选择 Agent，加入当前会话或新建会话；后端完成 A2A 连接与 Participant 绑定。 | P0 |
| 能力匹配与推荐 | 根据会话上下文或用户自然语言意图，推荐 N 个最相关 Agent；可选：自动创建「临时小组」并分配子任务。 | P1 |
| 私有/公网策略 | 空间级配置：允许的 Agent 来源（仅目录/仅指定域名/允许任意 A2A URL）；租户/命名空间隔离。 | P1 |
| 目录审核与上架 | 可选工作流：Agent 提交 → 审核 → 上架/下架；企业版支持私有目录仅本租户可见。 | P2 |
| 注册中心 API | 开放 API：查询 Agent、按条件搜索、获取 Agent Card；供编排引擎、前端、第三方集成。 | P0 |

### 3.4 数据模型要点

- **agents**：id, source_url, agent_card_json, **agent_type(logical|execution)**, fetched_at, visibility(public|tenant|private), tenant_id(nullable), status(active|deprecated|pending_review)。
- **agent_tags**：agent_id, tag_key, tag_value（或结构化 capability 表）。
- **agent_visibility_rules**：space_id/tenant_id, agent_id 或 rule(allow_list/deny_list), scope。

### 3.5 阶段规划

- **Phase 1**：注册、目录 CRUD、搜索与筛选、一键加入会话、开放查询 API。
- **Phase 2**：能力标签、匹配与推荐、私有/公网策略。
- **Phase 3**：审核流程、多租户目录、高级推荐（组合 Agent）。

## 四、赛道三：信任、安全与合规（Trust & Compliance）

### 4.1 目标

在 A2A 的传输与认证之上，提供**可审计的协作记录**、**数据边界策略**（谁可与谁交换何种数据）、**高风险操作的人审与策略**，以及**信任增强**（可验证收据、TEE/attestation、声誉信任图）与**安全增强**（提示词注入防护），满足企业合规与风控需求。

### 4.2 核心概念

- **审计日志**：会话内所有 A2A 请求/响应、消息、参与者变更、审批动作的不可篡改记录。
- **数据边界**：策略规定某会话/某类 Agent 仅可传递元数据/摘要/引用，禁止传递原始敏感数据（或仅限加密通道）。
- **人审**：对敏感操作（如发邮件、付款、改生产数据、**硬件敏感动作**如开锁/启动车辆）插入「等待人类批准」状态；审批通过后再继续 A2A 流程。
- **信任增强**：见 4.5（签名与可验证收据、TEE + attestation、声誉/信任图）；**暂不考虑智能合约**方向。
- **安全增强**：见 4.6（协助 Agent 防范提示词注入攻击的五项能力）。

### 4.3 功能清单

| 功能模块 | 描述 | 优先级 |
|----------|------|--------|
| 审计日志 | 所有会话事件、A2A 调用、消息、参与者变更写入审计存储；支持按会话/时间/参与者/操作类型查询导出。 | P0 |
| 日志不可篡改 | 审计存储使用 append-only 或与哈希链/可信存储对接，满足合规对「不可删改」的要求。 | P1 |
| 数据边界策略 | 策略引擎：按会话类型、参与者角色、标签限制可传递的 content 类型（如禁止 FilePart、仅允许 TextPart 摘要）。 | P1 |
| 策略执行点 | 在消息入会话、A2A 请求转发前执行策略；违规则拒绝并记录。 | P1 |
| 人审工作流 | 定义「敏感操作」列表（如 tool_call 类型、或某 Agent 的某 capability）；触发时创建审批任务，会话内展示待审批项，人批准/拒绝后回调 A2A 或更新 Task。 | P1 |
| 审批历史与追溯 | 审批记录与审计关联；支持「谁在何时批准了哪条操作」的报表与导出。 | P1 |
| 租户与 RBAC | 空间/租户级权限；角色（管理员/成员/只读）；控制谁可创建会话、邀请哪些 Agent、查看审计。 | P0 |
| 合规报表 | 预置报表：访问日志、敏感操作统计、审批率、数据外传情况；支持导出供合规团队使用。 | P2 |
| **可信身份与来源标识** | 为 Agent、节点、硬件设备建立稳定身份与来源标识（官方托管 / 自建 / 连接节点 / 厂商设备），并支持签名、公钥、设备身份或节点身份校验。 | P0 |
| **能力声明与风险建模** | Agent 的能力不是一句介绍，而是结构化声明：能力类型、输入输出、风险等级、确认方式、数据需求、适用范围。 | P1 |
| **可信调用决策（Trust Policy）** | 调用前综合身份、能力、历史表现、运行环境、风险等级、数据范围与审批状态，输出「允许 / 限制 / 需确认 / 需人审 / 拒绝」等决策。 | P1 |
| **多维信誉画像** | 不只做单一评分，而是维护按任务类型/能力维度拆分的信誉画像，如成功率、稳定性、误执行率、审批通过率、违规率。 | P1 |
| **可验证收据** | 关键事件（人审通过、跨节点任务完成等）签发可验签收据；收据格式在协议中定义，SDK/网关签发与校验。 | P1 |
| **TEE + attestation** | 敏感逻辑在 TEE 中运行并对外暴露 attestation；硬件 Agent 可证明「本设备在可信环境中执行」，平台展示「已验证硬件 Agent」。 | P2 |
| **声誉/信任图（链下）** | 为 Agent、连接节点维护链下声誉；与目录、认证结合；支持评分、举报、降权或隔离。 | P1 |
| **结构化通道（防注入）** | 协议中区分 user_content 与 system_context；平台/SDK 使用明确分隔符与标记；协议文档要求 Agent 不执行 user 中覆盖 system 的指令。 | P0 |
| **注入检测与告警/拦截** | 用户消息转发前轻量检测典型注入模式；可配置为仅记录/人审/拒绝；审计事件「疑似注入」可追溯。 | P1 |
| **SDK/文档防注入最佳实践** | 安全组装请求的 helper、输入校验与输出过滤建议、最小权限与拒绝「角色切换/忽略上文」类指令的文档。 | P0 |
| **数据边界与敏感输出** | 限制系统提示/内部策略/他人消息不得输出给不可信方；对 Agent 输出做脱敏或禁止泄露系统提示。 | P1 |
| **加固运行时（可选）** | 高安全场景：用户输入与系统上下文沙箱隔离；禁止动态加载代码；可选输出二次检查。 | P2 |

### 4.4 可信调用体系（Trustworthy Invocation System）

> 本项目的核心护城河之一，不是网络中有多少 Agent，而是这些 Agent 能否被**评估、被信任、被安全地调用**。因此我们构建的不是单一“评分系统”，而是一套**可信调用体系**：让用户、逻辑 Agent、连接节点与平台在不同风险等级下，能够对“是否调用某个 Agent”做出可解释、可审计、可治理的决策。

#### 4.4.1 核心定义

建议将信任体系定义为六层共同组成的闭环：

| 层次 | 要回答的问题 | 典型证据/能力 |
|------|--------------|---------------|
| **身份可信** | 它到底是谁？ | Agent ID、主体、节点/设备身份、签名、公钥、认证状态 |
| **能力可信** | 它声称会的东西是真的吗？ | 结构化能力声明、风险等级、输入输出约束、能力验证 |
| **行为可信** | 它过去做得怎么样？ | 成功率、稳定性、误执行率、审批通过率、违规率 |
| **执行可信** | 这次调用是在可信环境中完成的吗？ | 托管/自建来源、沙箱、TEE、权限边界、策略命中、审批状态 |
| **结果可信** | 它真的按承诺做了什么？ | 执行结果结构、时间戳、收据、审计链路、回放 |
| **治理可信** | 出问题时平台如何处理？ | 举报、申诉、降权、冻结、隔离、恢复信誉 |

**原则**：

- 信任不是一个总分，而是一种**调用决策能力**。
- 信任必须与**任务类型、风险等级、能力范围**绑定，而不是全局一刀切。
- 高风险动作默认采用更高门槛，不能因“整体评分高”而自动放行。

#### 4.4.2 可信调用的输入与输出

调用一个 Agent 前，平台或调用方不应只看“徽章”或“评分”，而应使用一套可解释的信任决策模型。

| 输入 | 示例 |
|------|------|
| **调用方身份** | 用户、逻辑 Agent、连接节点、官方服务 |
| **被调用 Agent 身份** | 官方托管 / 自建 / 连接节点 / 厂商设备 |
| **任务类型** | 文本生成、代码执行、数据处理、物理动作 |
| **风险等级** | low / medium / high / critical |
| **数据敏感度** | 是否涉及敏感文件、凭证、企业数据、物理世界控制 |
| **能力可信度** | 是否通过能力验证、厂商认证、平台认证 |
| **行为历史** | 成功率、近期异常率、误执行率、违规率 |
| **执行环境** | 是否官方托管、是否在沙箱中、是否有 attestation |
| **策略与审批状态** | 是否命中规则、是否已通过用户确认或人审 |

| 输出 | 含义 |
|------|------|
| **允许自动调用** | 低风险、可信度高、满足默认策略 |
| **允许但限制范围** | 仅开放只读能力、限定数据范围、限定执行区域 |
| **允许但需用户确认** | 对用户资产或设备有影响，但可由用户快速确认 |
| **允许但需人审** | 高风险或合规敏感场景，需审批链路 |
| **允许但需可信环境** | 仅允许在平台沙箱、TEE 或已认证硬件中执行 |
| **拒绝调用** | 身份不可信、风险过高、历史异常、策略不允许 |

也即：**Trust as Policy**，而不是单纯 **Trust as Badge**。

#### 4.4.3 多维信誉画像，而不是单一总分

对 Agent、节点、硬件设备的信任评估，不宜只展示一个总分。建议至少按以下维度构建信誉画像：

| 维度 | 说明 |
|------|------|
| **任务成功率** | 某类任务是否稳定完成 |
| **稳定性/延迟** | 近 7/30 天波动、超时率、P95/P99 |
| **误执行率** | 尤其针对物理动作与高风险工具调用 |
| **审批通过率** | 高风险任务是否经常因风险被否决 |
| **策略违规率** | 是否常触发越权、越界、数据边界问题 |
| **用户满意度/NPS** | 作为弱信号，与行为数据分开看待 |
| **近期异常率** | 最近是否频繁失败、撤销、被举报 |
| **能力维度拆分** | “写作可信”与“门锁控制可信”必须分开 |

**设计要求**：

- 信誉画像应尽量**任务相关、能力相关、时间敏感**。
- 用户评分只能是弱信号，不能替代执行与审计证据。
- 节点信任与 Agent 信任分开建模；好 Agent 运行在高风险节点上，仍不应默认高可信。

#### 4.4.4 最小可信体系分阶段

| 阶段 | 最小可信能力 |
|------|--------------|
| **Phase 0** | 稳定身份、来源标识、基础能力声明、风险等级基础字段、最小审计日志、基础收据、高风险动作默认不自动执行 |
| **Phase 1** | 多维信誉画像、用户确认/人审工作流、基础认证与徽章、调用前 Trust Policy、异常/违规降权 |
| **Phase 2** | 可验证收据普及、执行环境信任标签、场景化信誉模型、厂商/节点认证、反作弊与反刷分 |
| **Phase 3** | 跨节点信任图、信誉迁移/联盟认证、第三方审计、仲裁与更完整的治理机制 |

#### 4.4.5 对外展示与治理原则

- **对用户展示**：不只展示“是否认证”，还要展示**适用场景、风险提示、是否需确认、最近稳定性**。  
- **对调用方展示**：提供适合机器消费的 trust context / trust policy 结果，而不是仅给 UI 徽章。  
- **对平台治理**：建立举报、申诉、调查、降权、冻结、下架、恢复信誉的闭环；信任体系的终局是治理，而不是展示。  

#### 4.4.6 Trust Engine 决策规则草案

> 本小节给出一版**可执行但不过度绑定实现细节**的 Trust Engine 决策规则草案。目标不是现在就固定算法，而是先明确：**决策按什么顺序做、什么情况必须拦截、什么情况可以放行、什么情况必须升级为确认或人审。**

##### 4.4.6.1 决策总原则

Trust Engine 建议遵循以下优先级：

1. **硬性规则优先于评分**：身份缺失、能力越权、策略显式拒绝、高风险无确认条件等，应直接拦截，不进入“综合评分后再看”。  
2. **任务与风险优先于总体信誉**：Agent 总体表现好，不代表可自动执行门锁、支付、删文件等高风险动作。  
3. **执行环境与节点信任单独判断**：不能因为 Agent 本身可信，就忽略其所在节点或运行环境不可信的问题。  
4. **默认保守，逐步放开**：尤其在网络早期，系统应宁可多触发确认/人审，也不轻易自动放行高风险调用。  
5. **所有决策必须可解释**：每次调用至少能给出“为什么允许 / 为什么限制 / 为什么拒绝”的机器可读理由。  

##### 4.4.6.2 建议的决策顺序

| 顺序 | 检查项 | 决策含义 |
|------|--------|----------|
| **Step 1** | **身份校验** | 被调用 Agent、节点、硬件或厂商身份是否存在且可验证；否则拒绝 |
| **Step 2** | **能力与权限校验** | 调用的能力是否存在、是否在声明范围内、是否超出调用方权限；否则拒绝 |
| **Step 3** | **数据边界与策略校验** | 是否涉及禁止数据流转、越权数据访问、策略显式 deny；命中则拒绝或限制范围 |
| **Step 4** | **风险归类** | 将本次调用归到 low / medium / high / critical，并确定默认门槛 |
| **Step 5** | **执行环境校验** | 是否要求 sandbox / TEE / 已认证硬件 / 官方托管；不满足则要求升级执行环境或拒绝 |
| **Step 6** | **行为历史评估** | 成功率、近期异常率、误执行率、违规率、近期举报等是否在允许范围 |
| **Step 7** | **确认与审批决策** | 是否需要用户确认、人审、物理确认、双重确认 |
| **Step 8** | **输出信任决策** | 自动放行、限制范围放行、需确认、需人审、需可信环境、拒绝 |
| **Step 9** | **记录与回写** | 决策理由、命中规则、相关证据、最终结果写入审计与信誉系统 |

##### 4.4.6.3 硬性拒绝条件（Hard Deny）

以下情况建议默认直接拒绝，不进入自动放行或普通评分流程：

| 触发条件 | 默认决策 | 说明 |
|----------|----------|------|
| **身份不可验证** | 拒绝 | 无稳定 Agent ID、节点身份、设备身份或签名校验失败 |
| **能力未声明或超范围** | 拒绝 | 调用的 capability 不在声明或授权范围内 |
| **策略显式 deny** | 拒绝 | 命中租户策略、空间策略、合规策略、黑名单策略 |
| **高风险动作缺少必要确认方式** | 拒绝 | 例如门锁控制要求物理确认，但当前无此条件 |
| **执行环境不满足强约束** | 拒绝 | 某类调用要求可信环境、TEE 或沙箱，但当前环境不满足 |
| **近期异常/违规达到阈值** | 拒绝或冻结 | 如近期误执行率、举报率、违规率显著异常 |
| **调用方权限不足** | 拒绝 | 用户、逻辑 Agent 或连接节点无权调用该能力 |

##### 4.4.6.4 按风险等级的默认决策门槛

| 风险等级 | 典型场景 | 默认决策 |
|----------|----------|----------|
| **low** | 普通文本生成、低风险通知、只读查询 | 满足身份、能力、策略即可自动放行 |
| **medium** | 发送消息、修改低敏配置、调用部分外部工具 | 允许自动放行，但必要时限制范围或要求轻量确认 |
| **high** | 访问敏感数据、执行桌面操作、控制部分设备 | 默认需用户确认或运行在可信环境；历史异常时升级为人审 |
| **critical** | 门锁、车辆、支付、删除重要数据、影响安全的物理动作 | 默认需人审或双重确认；部分场景无可信环境/确认条件时直接拒绝 |

这里的关键不是“把每个能力永久写死”，而是给网络建立一条默认安全基线：**风险越高，自动化门槛越高。**

##### 4.4.6.5 信誉与行为历史如何影响决策

信誉画像建议作为**调节器**，而不是替代硬性规则。

| 情况 | 建议影响 |
|------|----------|
| **高成功率、低异常率、低违规率** | 在低风险 / 中风险场景下更容易自动放行 |
| **近期异常率上升** | 将自动放行降级为“限制范围”或“需确认” |
| **高风险能力历史稳定** | 可减少无意义打扰，但仍不能绕过高风险默认门槛 |
| **近期被举报或误执行增加** | 提升到需人审，或临时冻结高风险能力 |
| **新接入 Agent / 新节点** | 默认更保守，先从低风险或受限能力开始 |

建议明确：**新 Agent 的冷启动信任来自身份与来源，不来自空白声誉。**  
也就是说，信誉是逐步积累的，不能因为“还没有负面记录”就默认高可信。

##### 4.4.6.6 执行环境规则

Trust Engine 不只评估“谁”，还要评估“在哪运行”。

| 执行环境状态 | 建议决策 |
|--------------|----------|
| **官方托管 + 沙箱 + 审计完整** | 可作为低/中风险任务的优先环境 |
| **官方托管 + TEE/attestation** | 适合高敏场景、密钥使用、合规敏感逻辑 |
| **已认证硬件 Agent / 执行 Agent** | 可承担特定高可信本地或物理动作 |
| **连接节点 / 自建环境** | 可调用，但默认更依赖身份、历史、策略与风险等级综合判断 |
| **未知或不透明环境** | 仅允许低风险能力，或直接拒绝高风险调用 |

建议将**Agent 信任**与**执行环境信任**拆开呈现，避免出现“Agent 看起来可信，但其实跑在高风险节点上”的误判。

##### 4.4.6.7 建议的决策输出结构

Trust Engine 的输出不应只是 `allow=true/false`，而应是一个结构化对象，至少包含：

| 字段 | 含义 |
|------|------|
| **decision** | `allow` / `allow_limited` / `need_confirmation` / `need_human_review` / `need_trusted_env` / `deny` |
| **reason_codes** | 命中的规则或原因，如 `identity_unverified`、`risk_high_requires_confirm` |
| **risk_level** | 本次调用归类后的风险等级 |
| **required_actions** | 还需要什么条件，如用户确认、人审、切换可信环境 |
| **constraints** | 限定的数据范围、能力范围、执行区域、时间窗口等 |
| **evidence_refs** | 支撑该决策的证据引用，如收据、审计事件、认证状态、信誉画像快照 |
| **expires_at** | 该决策的有效期，避免长期复用高风险授权 |

这样做的价值是：

- **前端可解释**：用户知道为什么被拦、为什么要确认。  
- **编排可消费**：逻辑 Agent 可根据 `need_confirmation` 或 `need_trusted_env` 自动改走下一步。  
- **审计可追溯**：事后能知道系统为什么做出当时的信任决策。  

##### 4.4.6.8 推荐的默认策略基线

在网络早期，建议默认采用以下保守策略：

- **所有 critical 能力默认不自动执行**。  
- **所有 high 风险物理动作默认要求确认或人审**。  
- **新 Agent / 新节点默认不自动获取高风险调用资格**。  
- **未知执行环境默认只能处理低风险能力**。  
- **评分或声誉永远不能单独覆盖硬性 deny 规则**。  

这个基线的意义是：先建立网络信任，再逐步提高自动化程度，而不是反过来。

### 4.5 信任增强（优先规划）

以下三项纳入优先规划；**智能合约方向暂不考虑**。它们既是独立能力，也是 4.4「可信调用体系」中的关键证据源。

| 方向 | 描述 | 落地要点 |
|------|------|----------|
| **（3）签名 + 可验证收据** | 平台或 Agent 对关键行为签发**可验证收据**：包含请求哈希、结果哈希、策略 ID、时间戳等，任何人可验签；事后审计、纠纷时可证明「谁在何时对何事做了何承诺」。 | 在协议中为关键事件（如人审通过、跨节点任务完成）定义收据格式；SDK/网关负责签发与校验；可选将收据哈希上链做存在性证明。 |
| **（4）可信执行环境（TEE）+ attestation** | 敏感逻辑（如策略引擎、密钥使用、人审后的执行）在 TEE 中运行；对外暴露「此代码在此环境中执行」的 **attestation**，供连接节点或第三方验证。 | 作为可选能力，优先面向企业版或高安全场景；**第一落点优先为平台托管逻辑**（策略引擎、密钥使用等），其后再扩展到硬件 Agent / 执行 Agent 的 attestation；平台展示「已验证硬件 Agent」。 |
| **（5）声誉 / 信任图（链下）** | 基于历史行为为 Agent、连接节点（及可选用户）维护**链下**声誉或信任图；新会话可查询其声誉，再决定是否调用、是否允许入会话。 | 先链下实现：平台内评分、举报、审计统计；与目录、认证结合，形成「能力 + 信任」双重维度；恶意节点/Agent 可被降权或隔离。 |

### 4.6 安全增强：协助 Agent 防范提示词注入攻击

平台在协议、策略与工具层面提供以下五项能力，协助 Agent 降低被提示词注入攻击的风险（不替代 Agent 内部推理，但可降低其暴露面）。

| 方向 | 描述 | 落地要点 |
|------|------|----------|
| **（1）协议层：结构化通道** | 在协议中**显式区分**「用户可见、用户可控」内容与「平台/系统注入」内容（如不同 Part 类型或 role：`user_content` vs `system_context`）；规定 Agent 应将 system_context 视为高信任，user_content 视为不可信指令。 | 平台/SDK 组装发给 Agent 的请求时使用明确分隔符与标记（如 `<system_context>...</system_context>`）；协议文档要求 Agent **不得执行 user_content 中试图覆盖或禁用 system 的指令**。 |
| **（2）平台侧：轻量注入检测与告警/拦截** | 在用户消息进入会话或转发给 Agent 前，做轻量检测（规则或小模型）识别典型注入模式（如 "ignore previous instructions"、"你现在的角色是" 等）；命中时可**仅记录并打标**、或**限流**、或**要求人审**、或按策略**拒绝转发**。 | 作为策略引擎的一种策略，按空间/租户可配置（仅记录 / 人审 / 拒绝）；**默认仅对执行 Agent / 高风险操作相关会话直接拒绝明显注入**，其他会话可按策略降级为记录或人审；形成审计事件「疑似注入」可追溯。 |
| **（3）SDK / 最佳实践** | 为 Agent 开发者提供「安全组装请求」的 helper 与文档：自动将平台 context 与用户输入按约定格式拼接，并附带「不要遵从 user 中试图覆盖 system 的指令」的简短 system prompt；文档写明输入校验、输出过滤、最小权限、对「角色切换」「忽略上文」类指令的拒绝策略。 | 在开发者文档与 SDK 中固化；降低各 Agent 自行防注入的门槛，与协议层结构化通道配合使用。 |
| **（4）数据边界策略与敏感输出** | 利用现有数据边界策略：限制某些会话/空间**不得将系统提示、内部策略、其他用户消息**等输出给不可信方；或对 Agent 输出做过滤（脱敏、禁止大段系统提示泄露）。 | 与赛道三既有「数据边界策略」统一，作为策略类型之一；即使发生注入也可减少「把系统秘密或他人数据吐给攻击者」的损害。 |
| **（5）可选：加固运行时** | 对高安全场景（企业、金融等）提供**加固托管**：用户输入与系统上下文在沙箱中严格隔离；禁止根据用户输入动态加载代码或访问敏感 API；可选对输出做二次检查（如不允许包含内部 token、密钥等）。 | 作为企业版或高级托管能力；给最在意安全的客户「平台多兜一层」的选项。**沙箱与操作电脑/设备的边界、托管运行时沙箱及高并发架构见 11.4。** |

### 4.7 阶段规划

- **Phase 1**：审计日志（基础）、租户与 RBAC、策略引擎框架；**结构化通道**（协议与 SDK）；**SDK/文档防注入最佳实践**；**可信身份与来源标识**；**能力声明与风险等级基础字段**；高风险动作默认不自动执行。
- **Phase 2**：数据边界策略、人审工作流、审批历史；**可验证收据**（收据格式与签发/校验）；**注入检测与告警/拦截**；**声誉/信任图（链下）**；**数据边界与敏感输出**；**可信调用决策（Trust Policy）**；**多维信誉画像**；异常/违规降权。
- **Phase 3**：不可篡改存储、合规报表、与第三方合规工具对接；**TEE + attestation**（可选）；**加固运行时**（可选，企业版）；跨节点信任增强、第三方审计与更完整的治理闭环。

## 五、赛道四：编排与语义（Orchestration & Semantics）

### 5.1 目标

在单次 A2A 任务之上，提供**会话级编排**（多步、多 Agent 协作）、**意图理解与路由**（选最合适的 Agent 或组合），以及**会话记忆与上下文**，使多轮、多 Agent 对话连贯且可自动化。

### 5.2 核心概念

- **会话级编排**：根据用户或上游 Agent 的「目标」，自动分解为多个子任务，选择 Agent、按依赖执行、汇总结果；可用 DSL 或可视化编排定义，也可由「编排 Agent」用 LLM 决策。
- **意图路由**：将自然语言或结构化意图映射到目录中的 Agent（或组合），并生成首轮 A2A 请求参数。
- **会话记忆**：平台维护会话级摘要、关键实体、历史决策；在发起新 A2A 请求时注入为上下文（如附加在 Task 的 context 或 Parts 中）。

### 5.3 功能清单

| 功能模块 | 描述 | 优先级 |
|----------|------|--------|
| 会话记忆存储 | 按 conversation 存储：摘要、实体列表、上一轮结论、可选的结构化 state；支持 TTL 与容量限制。 | P0 |
| 上下文注入 | 在创建或更新 A2A Task 时，将当前会话记忆打包为 Parts 或 context 字段注入，供 Agent 使用。 | P0 |
| 记忆更新策略 | 每轮对话后更新摘要/实体（可由平台侧轻量摘要或调用统一「摘要 Agent」）。 | P1 |
| 意图解析 API | 接收自然语言或简短描述，返回推荐的 Agent 列表 + 置信度；可选返回建议的 Task 参数。 | P1 |
| 路由执行 | 根据意图解析结果自动选择 Agent 并发起首轮 A2A 请求；支持 fallback（主选失败则次选）。 | P1 |
| 编排 DSL/配置 | 定义多步流程：步骤、依赖、每个步骤对应的 Agent 或「意图」、输入输出映射；引擎按图执行并写回会话。 | P1 |
| 编排运行时 | 执行编排图：调用 A2A、等待完成/input_required、将输出作为下一步输入、处理失败与重试。 | P1 |
| 可视化编排（低代码） | 可选：拖拽式编排编辑器，生成 DSL 或配置，降低非开发者的使用门槛。 | P2 |
| 编排模板 | 预置常用流程（如「研究→写作→审核」）；用户仅需填参数即可运行。 | P2 |

### 5.4 数据模型要点

- **conversation_memory**：conversation_id, summary, entities_json, last_conclusion, updated_at。
- **orchestration_definitions**：id, name, space_id, definition_json(图结构), version。
- **orchestration_runs**：id, conversation_id, definition_id, current_step, state, started_at, finished_at。

### 5.5 阶段规划

- **Phase 1**：会话记忆存储与上下文注入、简单记忆更新（如最后 N 条摘要）。
- **Phase 2**：意图解析与路由、编排 DSL 与运行时、基础编排模板。
- **Phase 3**：可视化编排、高级记忆策略、与外部 LLM 的编排决策集成。

## 六、赛道五：可观测与运维（Observability & Ops）

### 6.1 目标

提供**会话与 A2A 调用的全链路可观测**：回放、耗时、错误、**成本与用量**统计，以及**人可读的对话视图**，便于调试、运维与成本管控。

### 6.2 核心概念

- **会话时间线**：按时间排序的会话内事件（消息、A2A 请求/响应、状态变更、审批）；支持过滤与导出。
- **回放**：根据存储的事件流重现某次会话的完整过程，便于复现问题。
- **成本与用量**：按会话/Agent/租户聚合 token 消耗、调用次数、延迟；可选与计费打通。

### 6.3 功能清单

| 功能模块 | 描述 | 优先级 |
|----------|------|--------|
| 会话时间线 API | 按 conversation_id 返回完整事件流（消息、A2A 事件、系统事件）；分页、按时间范围筛选。 | P0 |
| 回放与调试视图 | 前端或独立页面：按时间线回放会话，展开每条 A2A 请求/响应、Parts、错误信息；支持导出 JSON/日志。 | P1 |
| 延迟与错误指标 | 每个 A2A 调用的延迟、状态码、错误类型；聚合为会话/Agent/空间维度报表。 | P0 |
| Token/用量采集 | 从 A2A 响应或平台侧统计 token 使用（若 Agent 返回或可估算）；按会话/Agent/用户聚合。 | P1 |
| 成本仪表盘 | 按空间/租户/时间展示用量与成本（若已接入计费）；支持预算告警。 | P1 |
| 人可读对话导出 | 将会话导出为可读格式：纯文本对话、Markdown、或带折叠的 HTML，隐藏技术细节，突出结论与决策。 | P1 |
| 告警与 SLO | 可选：延迟/错误率超过阈值告警；为关键 Agent 配置 SLO 看板。 | P2 |
| 与 APM 集成 | 导出 trace/span 到 OpenTelemetry 或现有 APM，便于统一运维。 | P2 |

### 6.4 阶段规划

- **Phase 1**：时间线 API、基础延迟与错误统计、简单用量计数。
- **Phase 2**：回放界面、token/成本仪表盘、人可读导出。
- **Phase 3**：告警、SLO、APM 集成。

## 八、赛道七：前沿与差异化（Frontier & Differentiation）

### 8.1 目标

在通用协作之上，提供**实时音视频与屏幕共享**、**多租户与白标**、以及**与人类 IM 的桥接**，拓展场景并形成差异化。

### 8.2 核心概念

### 8.3 功能清单

|----------|------|--------|
| 多租户架构 | 租户隔离（数据、配置、目录）；租户级配置：品牌、域名、登录方式、功能开关。 | P0 |

| 实时音视频 | 会话内启动音视频通话；WebRTC 或第三方集成；参与方可为人或「有音视频能力的 Agent」；录制与合规留存可选。 | P2 |
| 屏幕共享 | 会话内屏幕共享；同上，合规与录制可选。 | P2 |
| 官方 IM Connector（我们→他们） | 我们开发并维护：Slack、Discord、Telegram、飞书等；将 IM 频道与平台会话绑定；双向同步；断连恢复、频道/会话映射策略。 | P1 |
| **Bridge API（他们→我们）** | **对外标准桥接接口**：认证、消息上行/下行、会话映射；规范与文档公开，任何 IM 可自主实现桥接端接入我们；可选桥接认证/徽章。 | P1 |
| 桥接策略 | 配置哪些消息同步（全部/仅 @Agent/仅摘要）、同步方向、脱敏规则。 | P1 |
| 开放平台与 Webhook | 会话事件、A2A 事件通过 Webhook 推送给第三方；第三方可回写消息或触发操作，用于与现有 IM 或工单系统集成。 | P1 |

- **Phase 3**：实时音视频与屏幕共享、更多 IM 平台、高级白标（独立域名等）。

## 九、赛道八：硬件 Agent 与平台融合（Hardware Agent & Platform）

> 本节将**自有硬件 Agent**（拓展坞形态、即插即用、无需安装软件或访问网站）与 Agent IM 平台的产品功能融合、以及营销与节奏融合，单独列为第八赛道。暂不展开游戏、考试等特定场景；聚焦形态抽象、双重角色、身份绑定与阶段性节奏。

### 9.1 硬件 Agent 的再框定

- **形态**：以「**电脑拓展坞**」形式存在，即插即用。
- **体验**：**不装软件、不打开网页**即可使用。
- **抽象**：硬件 = 用户侧**可信端点** + 具备**逻辑 Agent 能力**的硬件终端；平台 = 会话、身份、编排、审计与多端协作。融合目标：**插上坞 = 以最低摩擦接入 Agent IM 网络**。
- **定位升级**：它不应被理解为「另一台能跑 Agent 的电脑」，而应被定义为**为 Agent 原生设计的本地基础设施 / Agent Appliance**：默认在线、默认接网、默认具备设备身份、权限边界与可协作能力。
- **竞争边界**：不与云端 Agent 拼「谁更像大脑」、也不与 `Mac mini + OpenClaw` 式 DIY 方案拼「谁更像通用主机」；核心竞争力应放在**现实空间入口、本地最后一米执行、持续在场、可信锚点、低维护与云边协同**。

### 9.2 产品功能融合

#### 9.2.1 坞在体系里的两个角色（可同时成立）

| 角色 | 含义 | 对「即插即用、不装软件」的支撑 |
|------|------|--------------------------------|
| **平台的零摩擦客户端** | 插上坞 = 自动连上 Agent IM；会话、身份、多 Agent/多人都在平台，坞负责登录、展示、输入和可信环境。 | 用户不需要「打开浏览器 / 装 App」；坞就是进入平台的入口，首次绑定账号后，每次插上即在线。 |
| **用户设备 + 逻辑 Agent 终端** | 坞在产品层首先是**用户设备**；在能力层是一台**具备逻辑 Agent 能力的硬件终端**，可参与会话、理解上下文并做本地/端侧决策。 | 用户不需要「打开浏览器 / 装 App」；坞不仅是入口，也是在网络中真正“有脑”的硬件节点。 |
| **执行 Agent 的调用者** | 坞本身**不是执行 Agent**，但可通过平台调用执行 Agent（如传统设备、家电、执行器），完成「逻辑 Agent 下决策，执行 Agent 去执行」的协作链路。 | 让坞成为「智能入口节点」，而不是单纯被调用的外设。 |

同一台坞 = **既是「我的平台客户端 / 用户设备」，也是「我空间里的一台具备逻辑 Agent 能力的硬件终端」**；并且它可以调用平台上的执行 Agent。

#### 9.2.2 身份与绑定

- **首次使用**：坞上电并联网后，完成**一次**与平台账号的绑定（如扫码、短码、或坞上小屏/指示灯引导）。绑定后，坞与「该用户 / 该空间」稳定关联。
- **即插即用**：之后每次插上并联网，坞自动连接平台、拉取该用户/空间的最新会话与状态，**无需再登录、无需打开网站**。
- **多坞/多端**：同一账号可绑多台坞；平台侧可做「首选设备」或「按场景路由」（与 presence 延伸一致）。

#### 9.2.3 「不装软件、不访问网站」如何实现

- 坞侧具备**最小运行时**（固件/轻量 OS）：联网、身份（与平台绑定）、协议客户端（与 Agent IM 通信）、本地 UI（或驱动外接显示器）。
- 首次绑定后，**认证信息**安全存于坞内（或由平台下发 token），插上即连平台，无需用户再输入账号密码或打开网页。
- 「用平台」的入口 = **插上坞**，而不是「打开浏览器 / 打开某官网」。

#### 9.2.4 相对云端 Agent 与 Mac mini DIY 方案的差异化优势

> 现实竞争中，用户未必只在「云端 Agent vs 我们的硬件」之间选择；也可能直接采用 `Mac mini + OpenClaw` 之类的本地 DIY 方案。因此，本产品必须避免被理解为「一个预装 Agent 的小电脑」，而应成为一个独立品类：**现实空间中的 Agent 基础设施**。

| 维度 | 纯云端 Agent（如纯托管部署） | `Mac mini + OpenClaw` 式 DIY 方案 | 我们的拓展坞形态硬件 Agent |
|------|------------------------------|-----------------------------------|---------------------------|
| **默认身份** | 远端服务 / 云端大脑 | 通用电脑或本地主机 | **Agent 原生设备 / Agent Appliance** |
| **开箱体验** | 需登录网页或 App | 需安装、部署、维护 | **插上即用、绑定后默认在线** |
| **本地权限与最后一米执行** | 弱，依赖桥接与远程授权 | 可做，但需自行配置权限/脚本/外设 | **原生围绕本地权限、电脑控制、设备接入设计** |
| **物理在场性** | 不在场，偏远程响应 | 在本地，但不是为「在场 Agent」设计 | **常驻于桌面/家庭/空间，可成为持续在线的近场入口** |
| **私域记忆与信任** | 更多依赖云端账户体系 | 可本地化，但仍是通用主机思路 | **天然可做私有记忆、密钥/凭证锚点、高风险动作最终确认点** |
| **弱网/离线韧性** | 受云与网络稳定性影响大 | 可本地运行，但用户需自行兜底 | **关键任务可本地兜底，弱网下仍维持部分能力** |
| **设备/空间级协同** | 偏云端编排，离物理世界较远 | 可接设备，但形态不天然面向空间中枢 | **可作为局域网设备、执行 Agent、近场传感器的统一中枢** |
| **维护成本** | 低本地维护，高云依赖 | 高：系统、权限、部署、升级、恢复都要自己处理 | **低维护、自动更新、自动恢复，更像家电/基础设施而非电脑** |
| **云边协同** | 强云端、弱本地 | 偏单机本地节点 | **默认就是主网 Hub 的可信边缘节点，可调度本地与云端能力** |

**结论**：

- 相对**云端 Agent**，硬件 Agent 的优势不在「更聪明」，而在**更贴近用户现实环境**，可承担**本地控制、持续在场、私有信任、弱网兜底**。
- 相对 **`Mac mini + OpenClaw` DIY 方案**，优势不在「更通用」，而在**更产品化、更低摩擦、更低维护、更适合非极客用户与企业标准化交付**。
- 因此，本产品的核心叙事应从「能在本地跑 Agent」升级为：**让 Agent 成为现实空间中可托付、可执行、可持续在线的基础设施。**

#### 9.2.5 核心竞争力的三条主线

| 主线 | 含义 | 对竞争的意义 |
|------|------|--------------|
| **个人 / 空间 Agent 基站（Personal/Spatial Agent Base Station）** | 设备不是配件，而是用户桌面、家庭、小办公室中的常驻 Agent 入口与身份载体。 | 避免被理解为「小主机」；建立“在场感”与长期入口。 |
| **本地行动中枢（Local Action Hub）** | 围绕电脑控制、文件/应用权限、局域网设备、USB/蓝牙/传感器接入设计。 | 将差异点落在「现实世界最后一米执行」，而非模型能力。 |
| **可信锚点（Trust Anchor）** | 承载私有记忆、密钥/令牌、高风险动作确认、设备身份与收据。 | 相比云端更可托付；相比 DIY 更易标准化、安全化、企业化。 |

对应到产品层，不与 `Mac mini + OpenClaw` 硬碰硬的方向包括：

- 不主打「更强算力」「更自由安装一切」「更像本地服务器」。
- 重点主打「原生 Agent 体验」「低维护常驻」「本地权限与设备接入」「可信执行」「云边协同」。

#### 9.2.6 硬件 Agent 的 MVP 能力清单（用于真正拉开差距）

> 若只做到「本地能跑 Agent」，用户会自然拿它与 `Mac mini + OpenClaw` 比，且很难赢。MVP 必须优先落在**产品化能力**而非**通用算力能力**上。

| 能力 | 为什么必须 | 对差异化的贡献 | 建议阶段 |
|------|------------|----------------|----------|
| **首次绑定 + 插上即在线** | 没有这一点，就只是另一台需要登录和维护的机器。 | 形成「零安装、零网页登录、低摩擦」的第一印象。 | **Phase 1 必做**（Phase 0 硬件后置） |
| **设备身份与 presence** | 硬件必须被平台理解为“某个人/某个空间里的在线 Agent 终端”。 | 支撑“常驻在场”和“空间入口”叙事。 | **Phase 1 必做**（Phase 0 硬件后置） |
| **本地权限中枢** | 没有本地控制权，就无法成为现实世界入口。 | 区别于纯云端，也区别于只是本地跑模型。 | **Phase 1 必做** |
| **本地行动执行闭环** | 至少能稳定完成一类真实动作，如文件/应用/设备控制。 | 将价值从“会聊天”推进到“真能做事”。 | **Phase 1 必做** |
| **高风险动作确认** | 涉及电脑/设备操作时必须建立信任边界。 | 强化“可信锚点”，降低用户心理门槛。 | **Phase 1 必做** |
| **私有记忆 / 本地上下文缓存** | 没有记忆，硬件难以形成“我的 Agent”心智。 | 强化 ownership、粘性与迁移成本。 | Phase 1 应该做 |
| **弱网兜底 / 离线基础能力** | 关键时刻掉线会大幅削弱硬件价值。 | 形成对云端方案的可靠性差异。 | Phase 1 应该做 |
| **自动更新与自恢复** | 若维护成本高，用户会回到 DIY 主机思路。 | 强化“家电化 / 基础设施化”而非“电脑化”。 | Phase 1 应该做 |
| **局域网 / 执行 Agent 接入** | 这是进入现实空间的关键。 | 从“个人设备”升级为“空间中枢”。 | Phase 2 重点 |
| **云边协同调度** | 只做本地节点不足以体现网络价值。 | 强化与主网 Hub 的协同，而非单机价值。 | Phase 2 重点 |

**MVP 判断标准**：

- 若用户只感知到「这是一台能本地跑 Agent 的盒子」，则差异化不足。
- 若用户能明显感知到「插上就在线、它能代表我、能管我的本地环境、关键操作更可信」，则产品开始脱离 `Mac mini + OpenClaw` 的比较维度。

#### 9.2.7 相对 `Mac mini + OpenClaw` 的最小功能对比（面向产品决策）

| 功能项 | `Mac mini + OpenClaw` 常见形态 | `Agent Dock` 应达到的产品标准 |
|--------|-------------------------------|-------------------------------|
| **安装与启动** | 用户自行装系统/服务/依赖并维护 | 出厂即为 Agent 场景预配置，首次绑定后即插即用 |
| **身份接入主网** | 依赖用户自行配置节点、账号、token | 默认接入 Agent IM，硬件有清晰设备身份与在线状态 |
| **本地权限管理** | 通过脚本、系统权限、自定义配置拼装 | 以结构化能力范围、白名单、确认机制产品化呈现 |
| **对普通用户的可用性** | 对极客友好，对普通用户门槛高 | 面向非极客用户也可稳定使用 |
| **维护方式** | 用户自管更新、日志、异常恢复 | 自动更新、自动恢复、低维护 |
| **产品默认目标** | “本地跑一个 Agent 实例” | “成为现实空间里的 Agent 基础设施节点” |

该对比的意义不是否定 DIY 方案，而是明确**我们卖的是“交付出来的完整体验”，不是“留给用户自己拼装的能力集合”。**

#### 9.2.8 硬件 Agent 的 Phase 1 / Phase 2 功能包（PRD 级收敛）

> 这一层不是“远景能力”，而是用于指导产品、设计、硬件、固件与平台协同推进的最小功能包。目标是让硬件 Agent 从概念进入可交付状态。

##### 9.2.8.1 Phase 1：证明它不是一台“小主机”

> **当前节奏**：Phase 0 采用软件平台先行，硬件接入与品类验证**后置至 Phase 1**，待硬件团队就位后再推进。以下功能包用于 Phase 1 及之后执行时对齐。

**目标**：让用户第一次接触时，明确感知到它是一个**Agent 原生设备**，而不是“另一台本地电脑”。

| 模块 | 必须实现的能力 | 用户可感知结果 |
|------|----------------|----------------|
| **首次绑定** | 扫码/短码绑定、设备命名、归属到用户/Space | 用户能在几分钟内完成首次接入，不需要研究部署教程 |
| **插上即在线** | 上电联网后自动登录主网、恢复身份与状态 | 用户每次接入都感觉它“天然属于这个网络” |
| **设备身份** | 在平台中显示为在线设备 / 逻辑 Agent 终端 | 它不是抽象服务，而是一个明确存在的设备节点 |
| **基础会话接入** | 能加入会话、接收消息、输出响应 | 用户可直接把它当作网络中的 Agent 使用 |
| **基础 presence** | 在线/离线、最近活跃、所属用户/空间 | 建立“它在场”的基本心智 |
| **最小安全闭环** | 凭证安全存储、设备注册、基础收据或关键事件记录 | 用户与团队对设备接入有基本信任 |

**Phase 1 明确不追求**：

- 不追求比 `Mac mini` 更强的通用算力。
- 不追求一开始就覆盖大量本地控制能力。
- 不追求复杂多设备编排或完整空间自动化。

**Phase 1 验收问题**：

- 用户是否会说“这就是一台小电脑 / 一个盒子版主机”？
- 还是会说“这是我的一个 Agent 设备，插上就进网络了”？

若第二种认知不能成立，则硬件线在 Phase 1 的产品定义尚未立住。

##### 9.2.8.2 Phase 1：证明它能稳定代表我做本地的事

**目标**：让硬件 Agent 从“在线的设备节点”升级为“现实空间中的本地行动中枢与可信锚点”。

| 模块 | 必须实现的能力 | 用户可感知结果 |
|------|----------------|----------------|
| **本地权限中枢** | 至少一类受控本地权限（如文件、应用、设备控制）产品化开放 | 用户开始把它当作“本地能做事的 Agent” |
| **行动执行闭环** | 至少 1～2 类真实任务稳定闭环（例如打开应用、读写指定目录、控制一个执行 Agent） | 用户不再只把它用于对话，而用于完成任务 |
| **高风险动作确认** | 对敏感操作提供确认机制（物理按键、屏幕确认或平台审批） | 用户对“让它动我的电脑/设备”更放心 |
| **本地上下文与记忆** | 保存基础偏好、近期任务状态、有限本地上下文 | 建立“这是我的 Agent，而不是临时会话入口”的感知 |
| **自动更新与自恢复** | 断线重连、异常恢复、版本更新最小打扰 | 降低维护成本，形成家电化体验 |
| **云边协同** | 本地任务与主网会话/云端 Agent 能顺畅切换 | 用户理解它不是单机，而是主网中的边缘节点 |

**Phase 1 的关键证明**：

- 用户愿意把一部分真实任务交给它，而不是只把它当展示或入口。
- 用户认为它比 DIY 本地主机**更省心、更可信、更适合长期放在桌面/空间里**。

##### 9.2.8.3 建议优先验证的首批场景

为避免做成“什么都想做的本地盒子”，建议优先验证 3 类场景：

| 场景 | 为什么优先 | 成功信号 |
|------|------------|----------|
| **桌面工作入口** | 最容易体现“插上即在线 + 本地权限 + 会话协作” | 用户把它当作进入 Agent IM 网络的默认入口 |
| **电脑操作助手** | 最容易体现与云端 Agent 的差异 | 用户愿意让它代做一类低风险本地操作 |
| **一个执行 Agent 的本地控制中枢** | 最容易体现它不是单机，而是空间节点 | 用户开始把它看成“本地设备与 Agent 网络之间的桥” |

不建议在早期同时铺太多家庭自动化、机器人、多房间协同等大场景，以免把“品类定义”做散。

### 9.3 营销与节奏融合

#### 9.3.1 谁先谁后

- **当前原则**：**软件平台先行**。先把 `Agent IM` 的可信调用软件闭环跑通，再在硬件团队恢复后推进 `Agent Dock`。
- **硬件线原则**：硬件不是取消，而是**延后**。对外口径应表达为「硬件能力与产品定义保留，待团队就位后按既定路线接上」。
- **发布节奏**：短期优先软件平台单独验证；中期若硬件准备就绪，再选择「Agent IM + Agent Dock」联动发布，放大“网络 + 可信边缘节点”叙事。

#### 9.3.2 目标人群与主叙事

- **泛「Agent 用户」**（不强调特定场景）：  
  - 主叙事：「你的第一台 **Agent 原生设备**：插上即接入 Agent IM 网络，与 Agent、与人协作，无需安装、无需打开网站。」

## 十、逻辑 Agent / 执行 Agent 与硬件及物理世界

> 在 1.6 的 Agent 类型扩展（逻辑 Agent vs 执行 Agent）与协议对外开放（面向传统设备厂商）基础上，本节将平台定位为**物理世界的编排层**，并给出硬件与在场、多设备、信任与安全、商业生态及若干可落地的想象方向。**其中，自有拓展坞形态硬件属于具备逻辑 Agent 能力的用户设备 / 硬件终端，而非执行 Agent；传统设备厂商接入的家电、传感器、执行器等更适合作为执行 Agent。** 可与自有硬件 Agent 项目（见第九节赛道八）深度融合，形成「软硬一体」的 Agent 网络。

### 10.1 平台作为「物理世界」的编排层

- **下行**：会话中的结论、任务、时间点通过平台下发给对应**执行 Agent**（硬件设备）执行——如播报、提醒、控制家电、导航、移动等；逻辑 Agent 负责决策「做什么」，执行 Agent 负责「在物理世界做」。
- **上行**：执行 Agent（硬件）的传感器/事件（门铃、异常声音、电量、位置、完成某动作）作为**事件消息**发回平台，可触发新会话或插入当前会话（如「门口有人」「孩子房间有哭声」）。
- **价值**：平台从「Agent 与人的 IM」升级为「Agent + 人 + 物理世界」的**统一编排层**；硬件负责感知与执行，平台负责协作、记忆、决策与路由。

### 10.2 身份与在场（Presence）的延伸

- 执行 Agent（硬件）的 presence 不仅 online/offline，可带**物理状态**：位置（在家/在车/在办公室）、电量、当前任务（清扫中/待命）、是否正在与人交互。
- 平台或用户可根据状态做**路由**：如「紧急提醒优先推送到当前在线且离用户最近的设备」；目录与发现可支持按「位置/能力/类型」筛选执行 Agent。

### 10.3 多设备：一个逻辑 Agent 或多设备小队

- **一个逻辑 Agent 对应多台执行 Agent**：如「家庭助手」背后对应多台硬件（音箱、机器人、平板）；平台或逻辑 Agent 将会话/任务路由到「最合适的那台」执行（播报用音箱，需要移动用机器人）。
- **多台执行 Agent 组成小队**：如「客厅机器人 + 厨房音箱 + 门锁 Agent」在同一 Space 或会话中；用户一次性说「我出门了，关灯、锁门、启动扫地」，由平台或逻辑 Agent 编排给不同设备执行。
- **价值**：既有「一个入口」的简洁，又有「多设备协同」的扩展空间；适合作为模板与场景包（见 10.8）。

### 10.4 硬件 + 平台的信任与安全

- **硬件 attestation**：设备通过安全芯片/TEE 证明「我是某型号、某固件」；平台可对通过验证的设备打「已验证硬件 Agent」标识，降低伪造设备接入风险；与赛道三 TEE + attestation 衔接。
- **敏感动作人审**：开锁、启动车辆、支付等**硬件敏感动作**在平台会话中走**人审流程**（用户或指定审批人在同一会话中点「同意」后再下发给执行 Agent），审计可追溯；与 4.2 人审一致。

### 10.5 商业与生态：硬件即模板与目录

- 每款硬件产品对应平台上的**执行 Agent 模板**（如「XX 扫地机器人 Agent」「XX 车机 Agent」）；用户购买设备后，在 App 内「绑定设备 → 选择模板 → 一键加入 Space」，设备即成为该 Space 中的执行 Agent。
- 目录中设**执行 Agent / 硬件**分类：用户可发现「支持本平台协议的设备」；**传统设备厂商**（智能家居、家电等）可按执行 Agent 规范接入，无需承载大模型即可上架为「兼容 Agent IM」的设备，形成**硬件生态**。
- **价值**：硬件销售带动平台使用，平台使用反哺硬件价值；第三方设备接入带来更多形态与场景，平台成为「硬件 + Agent」的枢纽。

### 10.6 更有想象力的方向（仍保持可行）

| 方向 | 描述 |
|------|------|
| **持久化身（Persistent embodiment）** | 用户的「个人 Agent」既有云端身份又有家中某台硬件作为「身体」；在家与硬件对话、出门与云端对话，**同一身份、同一会话历史、同一记忆**，由平台保证连续性。 |
| **事件驱动会话** | 硬件产生的事件（婴儿哭声、门铃、烟雾）自动在平台**创建会话**或**拉入已有会话**，并 @ 相关人或逻辑 Agent（如家长、安防 Agent）；平台成为「物理事件 → 协作与响应」的中枢。 |
| **共享物理 Agent** | 如合租/办公室的一台机器人多人共享；平台管理**排队、上下文切换、权限**（「Alice 正在用」「Bob 排队中」「轮到 Bob 时清空上一轮上下文」），避免冲突并保留审计。 |
| **硬件作为 Bridge 的端** | 若硬件带麦克风/屏幕，可同时作为**人类 IM 的终端**：用户通过硬件上的会话界面与平台会话交互，硬件即 Bridge 的呈现端，与手机/电脑并列。 |

### 10.7 与 Matter 等传统 IoT 方案的关系与代际优势

> 本规划中的「逻辑 Agent + 执行 Agent」方案**不是 Matter 的替代品**，也不应被设计成「更聪明的智能家居平台」。更合适的定位是：**建立在 Matter、Zigbee、HomeKit、BLE、厂商私有协议等设备互联层之上的 Agent 协作与可信执行层。**

#### 10.7.1 我们与 Matter 不是同一层竞争

| 维度 | Matter 等传统 IoT 方案 | 本方案（逻辑 Agent + 执行 Agent） |
|------|------------------------|-----------------------------------|
| **核心对象** | 设备、属性、命令、互操作性 | Agent、会话、目标、协作、执行链 |
| **主要解决的问题** | 设备怎么发现、接入、互通、被控制 | 谁理解目标、谁决策、谁执行、谁确认、谁留痕 |
| **交互模式** | 人或规则直接发命令给设备 | 人提出目标，逻辑 Agent 结合上下文推理，再编排执行 Agent 执行 |
| **自动化形态** | 偏静态规则、场景联动 | 偏动态语义编排、失败回退、人在环、跨 Agent 协作 |
| **可信性** | 关注连接与控制正确性 | 关注完整执行链：发起、审批、策略、执行、结果、收据 |
| **生态边界** | 更偏家庭/设备域 | 家庭、桌面、办公室、车载、云端、连接节点等统一 Agent 网络 |

**定位结论**：

- **Matter 解决“设备怎么连”**。
- **本方案解决“Agent 如何理解目标、协作决策，并安全地让物理世界发生变化”**。
- 因此我们应将 Matter 视为**southbound 设备互联层**的一种，而不是需要正面替代的竞争对象。

#### 10.7.2 相对 Matter 的代际优势

| 代际升级方向 | 本方案应体现的优势 |
|--------------|--------------------|
| **从设备控制升级到目标协作** | 用户说的不是单条命令，而是目标；逻辑 Agent 结合上下文、会话历史与当前状态生成执行计划。 |
| **从单设备互通升级到多 Agent 协作** | 不只管灯、锁、空调，而是让多个执行 Agent、逻辑 Agent 与人进入同一会话与编排。 |
| **从静态自动化升级到动态语义编排** | 不是固定 if-this-then-that，而是根据当前人、设备、环境、历史上下文动态生成执行与回退路径。 |
| **从控制设备升级到可信执行体系** | 每次物理动作都可回答：谁发起、谁批准、哪条策略命中、谁执行、结果如何、是否有收据。 |
| **从家庭协议升级到跨空间 Agent 网络** | 不限于智能家居，可覆盖桌面、家庭、办公室、车载、云端和连接节点。 |
| **从厂商兼容升级到能力市场与 Agent 经济** | 执行 Agent 不只是“被接入的设备”，而是可被发现、调用、编排、认证、计费与分成的网络角色。 |

#### 10.7.3 设计策略：如何确保我们不是“又一个智能家居方案”

为避免产品被理解为传统 IoT 平台的升级版，建议在协议、架构与产品层坚持以下原则：

1. **把 Matter 等协议放在 southbound 适配层，而不是产品主叙事中心**  
   设备协议层负责接入与互通；平台主叙事始终是会话、编排、信任、审计、人在环与 Agent 网络。  

2. **执行 Agent 设计成“可协作角色”，而非“设备对象”**  
   执行 Agent 不应只暴露“开/关/调节”类命令，还应带有能力语义、风险等级、前置条件、确认方式、结果结构与失败语义。  

3. **事件必须进入会话与编排，而不是只做状态回传**  
   门铃、哭声、烟雾、设备异常、任务完成等都应成为可插入会话、触发 Agent 协作与人审的事件。  

4. **逻辑 Agent / 执行 Agent / 平台三层分工必须清晰**  
   逻辑 Agent 负责理解与决策；执行 Agent 负责执行与反馈；平台负责权限、编排、审计、信任与策略。避免执行 Agent 变成“又一个会推理的设备代理”，也避免逻辑 Agent 裸控设备。  

5. **人在环必须是一等公民**  
   与传统智能家居“人主要负责配置”不同，这里的人应是会话参与者、审批者、打断者与最终授权者。  

6. **可信执行链默认存在**  
   任何重要物理动作都要默认具备发起者、审批者、策略命中、执行结果与可验证收据，尤其是门锁、车辆、电器、办公设备等高风险场景。  

#### 10.7.4 架构落点

- **southbound**：Matter、Zigbee、HomeKit、BLE、局域网协议、厂商私有 API 等，统一收敛到**执行 Agent 适配层**。  
- **mid-layer**：执行 Agent 以统一 Agent 角色进入目录、会话、编排、审计与策略系统。  
- **northbound**：对用户、逻辑 Agent、硬件 Agent、连接节点暴露的是会话、目标、协作、审计与商业化能力，而不是底层设备协议细节。  

这样设计后，Matter 等协议不再是“对手”，而是我们生态的输入层；我们的护城河在其上的**Agent 协作、可信执行与云边网络效应**。

#### 10.7.5 执行 Agent 的统一能力模型草案

为避免执行 Agent 退化为“设备 API 的薄封装”，建议在协议与目录层为执行 Agent 定义统一能力模型。该模型不要求设备厂商暴露复杂智能，而要求其以**可协作、可审计、可编排**的角色进入网络。

| 字段/维度 | 含义 | 为什么重要 |
|-----------|------|------------|
| **agent_type** | 明确为 `execution` | 与逻辑 Agent 区分，避免发现、路由与权限混淆。 |
| **capability_id** | 某项能力的稳定标识，如 `lighting.on_off`、`door.lock_control`、`camera.snapshot` | 统一目录、匹配、编排与审计语义。 |
| **intent_class** | 能力所属的意图类别，如通知、安防、环境控制、移动、媒体 | 让逻辑 Agent 按“目标”而非按厂商命令选设备。 |
| **input_schema** | 可接受的结构化参数，如目标值、持续时间、区域、优先级 | 保持调用可校验、可回放、可审计。 |
| **result_schema** | 成功/失败结果结构，如状态、执行时间、设备反馈、错误码 | 让执行结果能统一进入会话、审计与收据。 |
| **risk_level** | 风险等级，如 `low / medium / high / critical` | 决定是否人审、是否二次确认、是否限制自动执行。 |
| **confirmation_mode** | 该能力默认的确认方式，如无确认、会话确认、物理确认、双重确认 | 将设备控制提升为可信执行链的一部分。 |
| **preconditions** | 前置条件，如用户在场、设备在线、电量充足、门已关闭 | 支撑动态编排与执行前校验。 |
| **reversibility** | 是否可撤销/补偿，如灯光可恢复、门锁不可自动回滚 | 影响编排策略与失败补偿路径。 |
| **latency_profile** | 实时性要求与典型响应时间 | 影响路由、fallback 与用户体验承诺。 |
| **scope / zone** | 能力生效范围，如客厅、办公室、用户桌面、某台设备 | 支撑空间路由与最小权限。 |
| **trust_source** | 能力来源与认证方式，如厂商认证、平台验证、硬件 attestation | 决定目录展示、默认信任与权限策略。 |

**设计原则**：

- 逻辑 Agent 选择执行 Agent 时，优先按 **intent_class + capability_id + risk_level + scope** 匹配，而不是按品牌或底层协议匹配。  
- 编排引擎调度执行 Agent 时，读取的是**能力模型**，而不是直接拼接厂商命令。  
- 审计与收据系统记录的是**能力调用事件**，而不是某个私有协议字段变化。  

这意味着：在我们的体系里，“门锁”“灯”“摄像头”不是孤立设备对象，而是带有语义、风险与确认逻辑的**执行角色集合**。

#### 10.7.6 southbound 适配层的映射策略

为兼容 Matter 等传统 IoT 方案，又保持上层 Agent 模型统一，建议在执行 Agent 适配层采用以下映射策略：

| southbound 层输入 | 适配层处理 | 上层看到的对象 |
|------------------|-----------|----------------|
| Matter 设备类型 / cluster / attribute / command | 映射为一个或多个 `capability_id`，并补齐风险、确认、结果结构 | 执行 Agent 能力条目 |
| 厂商私有 API / 局域网协议 | 适配为统一 `input_schema` / `result_schema` | 可被目录发现与编排的执行 Agent |
| 设备事件（如门开、烟雾、低电量） | 转换为标准化事件消息，并附设备身份、上下文与风险等级 | 可进入会话、触发编排或人审的事件 |
| 设备在线/离线/状态变化 | 映射为 presence 与状态快照 | 可被路由、推荐与策略系统消费的状态信息 |

**具体策略建议**：

1. **一台设备可映射为一个执行 Agent，也可映射为多个能力条目**  
   例如一个摄像头设备，可暴露 `camera.snapshot`、`camera.stream`、`camera.motion_event` 等多个能力，而不是只作为“某品牌摄像头”。  

2. **不把底层协议字段直接上抛到 northbound**  
   上层不感知 Matter cluster 或私有设备字段；这些细节只留在适配层与诊断层。  

3. **优先做“少数高价值能力”的稳定映射，而不是一开始追求协议全覆盖**  
   早期应优先支持能体现 Agent 价值的能力：通知、安防、门锁、照明、环境控制、桌面/本地设备操作等。  

4. **事件流优先于属性同步**  
   对 Agent 编排来说，“发生了什么”通常比“当前属性是什么”更关键。应优先把门铃、异常、触发、完成、失败等事件映射进会话与编排系统。  

5. **能力模型允许同一能力有多个 southbound provider**  
   同一个 `lighting.on_off` 能力，可来自 Matter、HomeKit bridge、厂商 API 或自有硬件中枢；这为 fallback、冗余与多厂商生态打基础。  

#### 10.7.7 这套设计为什么会形成显著代差

若只做 southbound 兼容，我们只是“支持 Matter 的 Agent 平台”；  
若把设备适配成统一执行 Agent，并让其进入**目录、会话、编排、审计、人审、收据、商业化**体系，我们就拥有了显著代差：

- **上层抽象更高**：用户与逻辑 Agent 面向目标和能力工作，而不是面向设备与命令工作。  
- **跨协议可协作**：Matter、厂商私有协议、自有硬件可在同一编排中协同，而不是各自为政。  
- **可信性更强**：物理动作不只是“发命令成功”，而是有完整决策与执行链。  
- **更适合网络化扩展**：设备不再只属于一个家庭 Hub，而可在 Space、节点、主网之间进入更大范围的 Agent 协作网络。  
- **更适合商业化与生态**：执行 Agent 可以被发现、认证、调用、计费与分成，这超出了传统 IoT 协议的价值边界。  

### 10.8 与路线图的关系

- 逻辑 Agent / 执行 Agent 的**协议扩展**（Agent Card 或目录中的类型区分、执行 Agent 的指令/结果格式）在协议与目录相关 Phase 中纳入。硬件与物理世界编排、presence 延伸、多设备、硬件信任与生态、事件驱动会话，以及**相对 Matter 等传统 IoT 方案的 southbound 适配层设计**，按优先级纳入**赛道七**及后续 Phase（如 Phase 2 起支持执行 Agent 目录、southbound 设备接入与基础编排，Phase 3 强化硬件生态、事件驱动与空间级协同）；具体阶段见**第十一节**路线图。

```
│  - 关系型 DB（会话、参与者、目录、配置、连接节点注册表）                    │
│  - 消息/事件流（Kafka/Redis Streams 等，用于实时推送与审计）               │
│  - 审计/日志存储（append-only 或专用合规存储）                             │
│  - 缓存（Agent Card、presence、会话摘要）                                  │
└─────────────────────────────────────────────────────────────────────────┘
                    │
         ┌──────────┴──────────┐
         │  连接节点（自建）     │  通过 Node–Hub 协议连接主网；参与目录同步与跨节点中继
         │  - 本地 Agent/用户   │
         └─────────────────────┘
```

### 11.2 关键集成点

- **A2A**：所有与外部 Agent 的交互统一经 A2A Gateway，严格遵循 A2A 协议；平台自身若提供「编排 Agent」「摘要 Agent」也以 A2A 端点暴露。
- **设备协议 / IoT southbound**：Matter、Zigbee、HomeKit、BLE、局域网协议与厂商私有 API 等，不直接作为上层产品模型暴露，而是统一进入**执行 Agent 适配层**；上层始终以会话、目标、执行结果、审计与收据模型工作（见 10.7）。
- **可信调用决策**：在调用前由 **Trust Engine + Policy & Compliance** 共同判断是否允许调用、是否限制范围、是否需用户确认/人审、是否仅允许在可信环境执行；调用后写回审计、信誉与收据系统（见 4.4）。
- **人类 IM**：通过桥接服务与各 IM 的 API 对接，平台侧仍以「会话 + 消息」模型统一处理。
- **连接节点**：自建节点通过 **Node–Hub 协议** 与主网连接，参与目录同步与跨节点中继（见 11.3）。

### 11.3 连接节点与主网

> 为使社区自建节点**能够**且**愿意**连接主网，形成「主网 + 连接节点」的以 Agent–Agent / Agent–人 为核心的大网络，在规划上把「节点–主网」作为一等公民，并明确连接后的能力、意愿设计与体验承诺。

#### 11.3.1 角色与定位

| 角色 | 说明 |
|------|------|
| **主网（Hub）** | 我们运营的托管平台；负责协调、发现、中继、可选计费；是网络的**默认 Hub**，不强制所有流量经主网存储或审查；协议上可支持未来多 Hub 或联盟。 |
| **连接节点（Connected Node）** | 社区用开源代码部署的、可独立运行的节点（本地或自有云），上有 Agent、可有用户；通过 **Node–Hub 协议** 注册并连接主网后，参与**目录可见/发现**与**跨节点会话/中继**，与主网及其他连接节点形成统一网络。 |

#### 11.3.2 Node–Hub 协议要点

- **注册与身份**：节点向主网注册，获得唯一 `node_id`；上报本节点能力（是否支持中继、可暴露的 Agent 列表等）。
- **心跳与状态**：定期心跳，主网维护「已连接节点」列表及状态；超时未心跳视为离线，可从目录与路由中降级或移除。
- **目录同步**：节点可选择将本节点部分/全部 Agent 元数据同步到主网目录；节点可从主网拉取「主网 + 其他连接节点」的公开 Agent 列表（只读或缓存），用于本地发现。
- **跨节点中继**：主网作为中继：当节点 A 的 Agent/人与主网或节点 B 的 Agent/人发起会话时，消息经主网转发（或经主网协调路由）；协议定义消息路由与投递语义。
- **身份与寻址**：主网提供或协调全局唯一寻址（如 `node_id` + `agent_id` / `user_id`），使主网上的用户或 Agent 能唯一指向某连接节点上的实体，支撑跨节点 @、邀请与路由。
- **限流与配额**：主网对连接节点按节点维度做合理限流与免费额度（见下），超量可引导升级或付费。
- **默认存储策略**：跨节点消息默认**只中继、不落正文**；平台默认仅保留**最小元数据 + 收据**（如路由元数据、发送/接收时间、参与方标识、事件类型、收据/收据哈希、策略命中结果）。具体是否持久化正文由会话/租户策略决定。

#### 11.3.3 连接后能力（价值清晰）

| 能力 | 说明 | 对连接节点的价值 |
|------|------|------------------|
| **目录可见/发现** | 节点选择将本节点 Agent 同步到主网目录；可从主网拉取全网公开 Agent 列表。 | 自己的 Agent 被更多人/Agent 发现；自己也能发现主网与其他节点的 Agent，实现 Agent–Agent、Agent–人 跨节点协作。 |
| **跨节点会话/中继** | 主网作为中继，转发跨节点消息或协调路由。 | 自建节点不必两两直连即可参与「大网络」对话，是连接的核心价值。 |
| **身份与寻址** | 主网提供或协调全局唯一身份/寻址。 | 跨节点会话、@、路由的基础。 |
| **可选：声誉/认证** | 连接节点（或其 Agent）满足条件后，在主网侧显示「已连接」「已验证」等标识。 | 提升信任与使用意愿。 |
| **可选：计费与分成** | 节点上的 Agent 若被主网侧用户/Agent 调用，可走主网计费与分成（节点方选择是否参与）。 | 自建节点也有机会从主网流量中获得收益（Phase 2 及以后）。 |

#### 11.3.4 意愿设计：为何愿意连接

| 维度 | 设计要点 |
|------|----------|
| **网络效应** | 连接 = 加入以 Agent–Agent、Agent–人 为核心的全局网络；不连则仅有本节点内协作。价值主张在文档与产品中写清。 |
| **隐私与控制** | **最小上报**：连接时至少节点标识、可选 Agent Card 列表、能力位；**不强制**上报完整消息内容。**策略可配**：节点可选择「仅目录同步」或「目录 + 中继」；数据用途透明（目录、路由、计费、安全）。 |
| **经济与声誉** | **免费额度**：连接节点在合理用量内（如每月 N 条跨节点消息或 N 次目录同步）免费。**声誉**：主网/目录中对「已连接节点」有标识。**可选分成**：节点 Agent 被主网侧调用时可参与分成（Phase 2）。 |
| **体验与信任** | **连接向导**：开源包/文档提供「连接主网」的默认配置与可选连接向导（如 CLI）。**兼容性承诺**：主网对 Node–Hub 协议做版本与向后兼容承诺；连接节点与主网在 CI 中做兼容性测试。**治理透明**：限流、降级、封禁规则（若有）写清且可预测。 |

#### 11.3.5 与「人–人为辅」的衔接

- 连接主网后，节点上的**用户**若获得主网侧身份或映射，也可与主网用户或其他节点用户发生**人–人**交互（辅助）。
- 主卖点仍是：节点上的 Agent/人 能与主网及其他节点的 Agent/人 协作（Agent–Agent、Agent–人）；人–人为同一网络上的辅助能力，不在产品叙事中喧宾夺主。

#### 11.3.6 架构中的位置

- 主网 Core Services 中需有**节点注册表**与**跨节点中继/路由**能力；A2A Gateway 或独立 **Node–Hub Gateway** 处理与连接节点的协议交互。路线图中在 Phase 0/1 纳入「节点–主网连接协议 MVP」与「目录同步 + 跨节点中继最小闭环」（见第十二节）。

### 11.4 沙箱与高并发的架构考虑

> 本节将「托管运行时沙箱」「安全与正常操作电脑/设备的边界」「高并发与水平扩展」纳入整体架构假设，便于 Phase 0/1 实现时预留接口与选型。

#### 11.4.1 沙箱（Sandbox）的考虑

| 场景 | 是否纳入 | 说明 |
|------|----------|------|
| **托管 Agent 运行时** | **必须** | 平台代用户/开发者运行逻辑 Agent（或执行链）时，须做隔离：进程/容器/VM 或轻量隔离，限制网络、文件系统、内存，避免影响主机与其他租户。 |
| **多租户数据与逻辑隔离** | **必须** | 各 Space/租户的会话、目录、策略互不可见；存储与访问控制上做好隔离，即一种逻辑沙箱。 |
| **加固运行时（可选）** | **建议** | 高安全场景下更严格沙箱：严格限制 Agent 可访问的资源与 API，可选配合 TEE。 |
| **执行 Agent 指令校验** | **建议** | 平台向执行 Agent（如智能家居）下发指令前，对指令做校验或「安全沙箱」：白名单动作、参数范围、频率限制，降低物理世界风险。 |
| **连接节点/第三方桥接** | **视实现** | 若主网会执行节点或第三方提供的代码（如 Bridge 适配器），则需沙箱；若仅做协议网关、不执行对方代码，可不做。 |

**实现参考**：可采用与具体云解耦的**沙箱抽象层**（可参考 [Sandbank](https://github.com/chekusu/sandbank) 等统一沙箱 SDK），支持多 Provider（如 Daytona、Fly.io、Cloudflare）与自建（如 BoxLite）切换；沙箱内默认**非 root、显式 capability（volumes/port/exec 等）、可观测 hooks**（如捕获工具调用与停止事件），用于审计与收据。若托管运行时调用第三方 API（如 Claude），需自行管理 **token 生命周期**；沙箱所在 region 或自建节点的 **IP 需与使用场景一致**，以降低被限流/风控的风险。

#### 11.4.2 沙箱与「操作电脑/设备」的边界

既要安全又要让 Agent 能正常操作用户电脑或设备，边界如下：

- **平台托管 Agent**：一律在沙箱内运行；**不直接**操作任何用户电脑或设备，只产出**意图/指令请求**，经协议发给用户本机客户端或执行 Agent。
- **用户电脑**：本机客户端/本机逻辑 Agent **不沙箱自己**，拥有本机真实权限以操作电脑。**安全**来自：只执行经**能力范围（Scope）+ 白名单 + 用户确认**通过的远程/平台指令；不执行未签名的任意代码；高风险操作（安装应用、改系统设置、敏感目录）可要求用户本次确认或仅在限定目录内执行。
- **执行 Agent / 设备**：**安全**靠只接受**白名单内的指令与参数**；**正常操作**靠在白名单内拥有完整控制权；敏感操作可要求用户或设备所有者确认。

即：**安全**由平台沙箱 + 本机/设备对指令的校验与授权保证；**正常操作**由本机/设备在授权范围内用真实权限执行保证。

#### 11.4.3 高并发的考虑

主网作为 Hub、连接节点与设备规模上来后，将面临高并发；架构上须预留扩展，实现可分阶段。

| 场景 | 高并发体现 | 架构要点 |
|------|------------|----------|
| **主网网关** | 连接数、QPS、心跳、presence | 无状态网关、水平扩展、连接管理（限流、优雅断开）。 |
| **跨节点中继** | 消息吞吐、延迟 | 异步、非阻塞；必要时消息队列或事件驱动，避免单点阻塞。 |
| **目录与发现** | 读多写少 | 缓存、读副本、索引优化。 |
| **会话与消息** | 写入与推送 | 消息落库（或元数据+收据）、实时推送（WebSocket/SSE fan-out）；数据库连接池、写入批量化、推送层水平扩展。 |
| **连接节点与设备** | 连接数、心跳 | 限流、批量处理、异步。 |
| **审计与收据** | 写入 | 异步写入、批处理或队列。 |

**路线图衔接**：Phase 0 可单机/最小规模跑通，但**架构上为扩展留好接口**（无状态、队列、缓存）；Phase 1 或更晚明确「性能与可用性目标」与扩容方案（如分库分表、多 region）。与第十三节「实时与规模」风险应对一致：消息与事件走流式架构，网关与编排无状态可水平扩展。

### 12.1A 官网与产品入口关系（统一口径，长期有效）

> 本小节用于固化团队已确认的关系设定，避免后续在“官网”“产品入口”“控制台”上出现反复分歧。

#### 核心定义

我们采用：**同一品牌体系下的双区模型**，而不是“官网内嵌产品页”模型。

- **官网区（Marketing Zone）**：负责价值讲述、场景教育、文档入口、转化承接（Demo/Waitlist/Start Building）。
- **产品区（Product Zone）**：负责真实产品使用（会话、目录、风险确认、人审、审计、收据、运营看板）。

一句话：

**产品入口可以挂在官网体系下，但不能被当作官网内容页来设计和演进。**

#### 推荐结构（按阶段演进）

1. **Phase 0 / Phase 1**：优先同域不同路由  
   - 示例：`/`（官网）+ `/docs`（文档入口）+ `/console` 或 `/app`（产品区）
   - 理由：部署快、品牌一致、从教育到体验路径最短。

2. **Phase 2+**：可演进为官网 + 产品子域  
   - 示例：`www.xxx.com`（官网）+ `app.xxx.com`（产品区）
   - 理由：鉴权、权限、环境隔离和企业能力边界更清晰。

#### 禁止项（长期不变）

- 不把控制台长期做成 marketing 二级页面
- 不用官网叙事布局承载高信息密度的产品操作
- 不让官网线改动产品区核心交互与可信链路表达

#### 用户路径口径

统一采用：

**先在官网区完成价值理解，再进入产品区完成真实任务。**

#### 官网区愿景覆盖与 `Coming Soon` 表达规范（长期有效）

为避免官网区“只展示当前已交付能力”而导致品牌叙事失真，补充以下长期规则：

1. **愿景覆盖原则**：官网区应完整映射产品长期愿景（各赛道核心价值与阶段方向），而非仅展示当前可用子集。  
2. **状态透明原则**：所有能力必须带状态标签之一：`Now`（已可用）、`In Progress`（开发中）、`Coming Soon`（已纳入后续阶段）、`Research`（探索中）。  
3. **承诺一致原则**：凡标注 `Now` 的能力，必须可映射到产品区真实可用路径；否则必须降级为 `In Progress` 或 `Coming Soon`。  
4. **CTA 分层原则**：  
   - `Now`：进入产品区真实体验（如 `Start Building` / `Open Console`）  
   - `In Progress` / `Coming Soon`：进入 waitlist、订阅更新、场景说明页  
   - `Research`：进入理念/路线图说明，不承诺上线时间  
5. **禁止项**：禁止将 `Coming Soon` 能力写成可立即使用；禁止以营销文案替代能力边界说明。

### 12.2 Phase 0 详细范围（MVP）

**目标**：一条「用户创建会话 → 从目录选 Agent 加入 → 人与 Agent 在会话中对话 → 审计可查」的完整闭环。

- **赛道1**：会话 CRUD、多参与者（人+Agent）、消息流、A2A↔消息映射、邀请 Agent 加入。
- **赛道2**：Agent 注册、目录搜索、一键加入会话、目录 API。
- **赛道3**：审计日志（基础）、租户/空间、RBAC 基础。
- **赛道4**：会话记忆存储 + 上下文注入（最小可行）。
- **赛道5**：时间线 API、基础延迟/错误统计。
- **赛道6**：开发者文档、SDK/示例（A2A + 平台会话 API）；**托管运行时 MVP** 与 **至少 1 个 Agent 模板**（为 Phase 1 一键部署打基础）。
- **赛道7**：多租户数据隔离。
- **连接节点与主网**：**Node–Hub 协议** 设计定稿；主网侧**节点注册与心跳**、**目录同步**（节点↔主网）MVP（可选），为 Phase 1 跨节点中继打基础（见 11.3）。
- **硬件 Agent（赛道八）**：**当前阶段后置**。待硬件团队就位后，在 Phase 1 推进**首次绑定、设备注册、插上即在线、设备身份/presence、基础会话接入**的最小闭环；目标仍是证明其为**Agent 原生设备**（见 9.2.8.1）。

**不包含**：线程、@、在线状态、意图路由、编排、计费、**消费者一键部署**、**跨节点中继**、IM 桥、音视频。

### 12.3 Phase 1 详细范围（体验与发现）

- **赛道1**：线程、@ 提及、在线状态、邀请流程增强。
- **赛道2**：能力标签、匹配与推荐、私有/公网策略。
- **赛道3**：数据边界策略、人审工作流、审批历史。
- **赛道4**：记忆更新策略、意图解析与路由。
- **赛道5**：回放与调试视图、token/成本仪表盘、人可读导出。
- **赛道6**：计费模型配置、用量记录。
- **赛道7**：基础白标（Logo/主色）。
- **连接节点与主网**：**目录同步 + 跨节点中继最小闭环**（连接节点可与主网及其他连接节点的 Agent/人 发现并会话）；**连接节点免费额度**与限流；**连接主网**的文档与默认配置/向导（CLI 或配置模板）；主网对 Node–Hub 协议的**兼容性承诺**（版本、向后兼容）。
- **硬件 Agent（赛道八）**：完成**本地权限中枢、至少 1～2 类本地行动执行闭环、高风险动作确认、本地上下文/记忆、自动更新与自恢复**；让设备从“在线节点”升级为“本地行动中枢与可信锚点”（见 9.2.8.2）。

### 12.4 Phase 2 / Phase 3 要点

- **Phase 1 补充**：**消费者「一键部署首个 Agent」**（新用户引导、Agent 模板库、模板配置与一键实例化、部署成功引导、试用额度）；见第十六节。
- **Phase 2**：编排 DSL 与运行时、计费与账单、托管运行时、**IM 桥双轨**（官方 1～2 个 Connector + **Bridge API v1 对外**）、白标增强；**连接节点**：**连接节点计费/分成**（节点上 Agent 被主网侧调用时的收益参与）、**连接向导**与兼容性测试在 CI 中固化；**硬件**：局域网 / 执行 Agent 接入、云边协同调度、从个人设备向空间中枢升级。
- **Phase 3**：编排模板与市场、分成与结算、合规报表、实时音视频、更多 IM、开放 Webhook、合作伙伴计划；**硬件**：多设备小队、空间级协同、第三方硬件生态与更完整的“持久化身”体验。

### 12.5 收缩与排序版路线图（按当前决策）

> 本节用于将现有范围收敛为可执行版本。排序原则：**软件平台先行**；Phase 0 只验证可信调用的软件闭环，**硬件后置**至 Phase 1 及之后、待团队就位再推进。

#### 12.5.1 Phase 0 必须做

- **协议与身份模型**
  - A2A 扩展基础：会话标识、消息 envelope、参与方标识、`user_content` / `system_context` 结构化通道
  - Agent 类型模型：`logical`、`execution`、**用户设备 / 硬件终端**
  - Node–Hub 扩展：节点注册、心跳、目录同步、全局寻址
  - 收据模型：至少覆盖**人审通过、目录注册/认证状态变更、计费/分成结算事件**
  - 跨节点默认策略：**只中继、不落正文，保留最小元数据 + 收据**
- **主网最小闭环**
  - 最小会话系统：conversation、participant、message/event
  - 最小目录系统：Agent 注册、目录查询、`agent_type` 区分
  - A2A Gateway：接外部 Agent 并映射进会话
  - Node–Hub Gateway：接自建节点并完成节点注册与目录同步
  - 最小审计：事件日志、收据签发与校验、最小元数据持久化
  - **架构假设**：网关与核心服务**无状态**，为**高并发与水平扩展**预留接口（见 11.4.3）；Phase 0 可单机或最小规模部署。
- **开源最小发布线**
  - 协议文档
  - SDK
  - 自建节点最小核心（可选）
  - **一个很薄的最小 UI / 控制台**
- **硬件推进线（Phase 1 及之后）**
  - Phase 0 不纳入硬件交付。待硬件团队就位后，在 Phase 1 推进：硬件与平台身份绑定模型（首次绑定、token/凭证、设备注册）；硬件在产品层/能力层身份（用户设备、逻辑 Agent 终端）；最小接入闭环与品类验证（“Agent 原生设备”而非“小主机”）。详见 9.2.8。

#### 12.5.2 Phase 0 应该做

- 最小安全能力：结构化通道落地、SDK 安全拼装 helper、高风险会话/执行 Agent 会话的明显注入默认拒绝、最小策略引擎骨架
- 最小人审：某类操作进入待审批，人可批准/拒绝，且事件有收据
- 最小托管运行时：至少支持 **1 个托管逻辑 Agent 模板**，为 Phase 1 的一键部署打基础；**设计上须考虑沙箱隔离**（见 11.4.1）。
- （Phase 0 不要求硬件侧反馈；硬件状态反馈留待 Phase 1 硬件接入时再做。）

#### 12.5.3 Phase 0 可以延后

- 线程、@、在线状态增强
- 能力推荐 / 自动组队
- 完整编排 DSL
- 声誉图完整实现
- 注入检测模型化升级
- TEE 正式产品化
- IM 桥
- 企业版白标
- 完整账单系统
- 硬件丰富场景
- 多设备小队高级逻辑

#### 12.5.4 Phase 1 必须做

- **会话体验补强**：线程、@ 提及、在线状态、邀请流程增强
- **连接节点可用化**
  - 目录同步 + 跨节点中继最小闭环
  - 默认只中继不落正文，保留最小元数据 + 收据

  - 兼容性承诺
  - 连接节点免费额度（克制）

  - 高风险动作确认
  - 本地上下文 / 私有记忆

#### 12.5.5 Phase 1 应该做

### 12.6 Founder 视角的产品层验收交付（贯穿 Phase 0 -> Phase 1）

> 目的：把“工程完成”翻译成“Founder 可直接决策的产品交付”。  
> 原则：必须是用户可感知、数据可验证、可用于 go/no-go 的产物，而非仅技术实现清单。

#### 12.6.1 你在产品层会看到的最小完整交付（MVP 验收包）

1. **产品主路径可演示且可复现**  
   - 路径：用户进入入口 -> 发起会话 -> 选择 Agent -> 完成可信调用 -> 审计/收据可查。  
   - 要求：不是一次性 demo，而是可重复运行的验收脚本与演示路径。

2. **官网/入口转化闭环可运行**  
   - 三语入口（EN / 繁中 / 简中）具备清晰 CTA：`Start Building` / `Book a Demo` / `Join Waitlist`。  
   - 每条路径可追踪到转化结果（点击、提交、激活）。

3. **可信价值可被用户感知**  
   - 高风险调用的确认/人审在产品体验中可见。  
   - 审计链路、收据校验不是“后端日志能力”，而是可被解释与展示的用户价值。

4. **增长与线索资产可经营**  
   - 线索（demo/waitlist）可持久化、可查询、可导出。  
   - 漏斗与可疑流量可观测，能支持每周复盘与实验迭代。

#### 12.6.2 Founder 决策包（每个里程碑必须给出）

每个里程碑结束时，团队必须提交一份“Founder 决策包”，至少包含：

- **Milestone 一页总结**：目标达成状态（✅/⚠️/❌）
- **产品验证证据**：主路径演示结果 + 真实漏斗数据
- **风险与代价**：当前最大 3 个风险与缓解动作
- **决策建议**：继续放大 / 调整后执行 / 暂缓（必须二选一或三选一，不可模糊）

> 验收口径：若交付物无法支持 Founder 在 30 分钟内完成 go/no-go 决策，则视为产品层交付不完整。

## 十三、成功指标与风险

### 13.1 成功指标建议

> 说明：以下为项目全周期指标池。**硬件相关指标不属于当前 Phase 0 指标**，应在 Phase 1 及之后硬件线恢复推进时再正式纳入阶段目标。

| 维度 | 指标示例 |
|------|----------|
| 采用 | 接入的 Agent 数量、活跃会话数、DAU/MAU（按空间或租户） |
| 体验 | 会话完成率、多 Agent 会话占比、人审通过率、平均会话轮次 |
| 生态 | 目录中 Agent 数、托管 Agent 数、模板使用次数 |
| 商业 | MRR/ARR、按 Agent 提供方的分成支出、单租户 LTV |
| 质量 | A2A 调用成功率、P99 延迟、审计与合规报表使用率 |
| **信任体系覆盖度** | 已建立稳定身份的 Agent 占比、带结构化能力声明的 Agent 占比、带风险等级的高风险能力覆盖率 |
| **可信调用效果** | 自动放行 / 需确认 / 需人审 / 拒绝 调用的分布；高风险调用默认拦截率；Trust Policy 命中率 |
| **信誉画像质量** | 有足够历史行为数据的 Agent 占比；按能力维度拆分信誉的覆盖率；信誉更新时效 |
| **信任治理效果** | 举报处理时长、申诉关闭时长、违规 Agent/节点降权或隔离时长、信誉恢复成功率 |
| **硬件采用（Phase 1+）** | 已绑定硬件数、周活硬件数、首次绑定成功率、绑定后 7 日留存 |
| **硬件体验（Phase 1+）** | “插上即在线”成功率、硬件在线时长、重连成功率、自动更新成功率 |
| **硬件价值验证（Phase 1+）** | 以硬件为默认入口的会话占比、由硬件发起或参与的本地任务数、硬件相关 NPS/满意度 |
| **硬件信任（Phase 1+）** | 高风险动作确认通过率、误触发/误执行率、涉及硬件动作的人审完成率 |

### 13.2 Founder 周看板（产品层最小指标集）

> 用途：每周看“是否在验证核心假设”，而不是“是否做了很多功能”。  
> 建议只保留少量高价值指标，避免信息噪音。

#### 13.2.1 核心看板（建议每周固定查看）

1. **主闭环完成数（weekly_trusted_invocations）**  
   - 定义：进入会话、有结果、有审计/收据；高风险路径符合确认/人审策略。

2. **主路径激活漏斗（home -> CTA -> docs/demo/waitlist -> activation）**  
   - 目的：验证入口是否有效把用户带入真实产品路径。

3. **多 Agent 会话占比**  
   - 目的：验证“网络协作”是否真实发生，而非单 Agent 聊天替代。

4. **高风险调用正确拦截率**  
   - 目的：验证可信调用是否真正降低风险，而非仅展示规则。

5. **线索质量（demo/waitlist -> 有效线索率）**  
   - 目的：避免“虚高提交量”掩盖真实产品吸引力。

6. **可疑流量占比（入口防滥用）**  
   - 目的：保护漏斗数据可信性，避免误判方向。

#### 13.2.2 Founder 每周决策问题（固定三问）

1. 本周数据是否更接近“用户因为可信调用而选择我们”？  
2. 当前掉点最大环节在哪里，下一周只改哪一个关键点？  
3. 若连续两周核心指标无改善，应该收缩范围还是调整主叙事？

> 执行建议：周会输出必须包含“告警 -> 动作 -> 回收结果”闭环；没有回收结果的动作不计入有效改进。

### 13.3 主要风险与应对

| 风险 | 应对 |
|------|------|
| A2A 规范演进 | 紧跟 Google A2A 更新，抽象「协议适配层」，便于升级与向后兼容。 |
| 多框架兼容成本 | 优先支持主流框架（LangChain/CrewAI 等），用 Adapter 文档与示例降低各框架接入成本。 |
| 企业合规要求差异 | 策略引擎设计为可配置、可扩展；审计与数据边界支持按行业预设策略包。 |
| 实时与规模 | 消息与事件走流式架构；网关与编排无状态可水平扩展；关键路径做容量规划与压测。 |
| **信任体系沦为“评分系统”** | 明确采用“可信调用决策”而非单一总分；把身份、能力、行为、执行、结果、治理六层证据都纳入体系（见 4.4）。 |
| **高风险调用被错误放行** | 高风险能力默认更高门槛；Trust Policy 输出必须支持“限制范围 / 需确认 / 需人审 / 拒绝”；高风险调用专项审计。 |
| **评分被刷或信誉失真** | 用户评分仅作为弱信号；以审计、成功率、误执行率、审批通过率、违规率等行为数据为主；逐步引入反作弊与第三方认证。 |
| **身份可信但执行环境不可信** | 将 Agent 信任与节点/执行环境信任分开建模；重要调用要求可信环境、沙箱、TEE 或 attestation 证据。 |
| **治理缺位导致网络信任崩塌** | 设计举报、申诉、调查、降权、冻结、恢复信誉闭环；规则透明、处理时效可追踪。 |
| **硬件价值不成立** | Phase 1 及之后硬件接入时，以“是不是 Agent 原生设备，而不是小主机”为验收标准；通过用户访谈与可用性测试验证认知是否成立（见 9.2.8.1、12.5.8）。Phase 0 不涉及硬件验收。 |
| **被 `Mac mini + OpenClaw` 等 DIY 方案替代** | 不与其拼通用算力与自由度，持续强化“插上即在线、本地行动中枢、可信锚点、低维护与云边协同”的产品差异（见第九节）。 |
| **硬件维护成本过高** | 早期即将自动更新、自恢复、状态可见性纳入 Phase 1 应做项；避免把产品做成需要持续运维的“另一台电脑”。 |
| **硬件误执行或越权操作** | 本地权限范围、白名单、高风险动作确认、人审与收据联动设计为默认机制；对硬件动作做专项审计。 |

| Skills/ClawHub 技能市场 | 赛道二 + 赛道六 | 能力目录、一键启用、生态 | 目录与模板市场对齐「能力+技能」展示 |

### 14.2 赛道四：上下文引擎与记忆（借鉴 ContextEngine）

**OpenClaw 做法**：  
- 提供 **ContextEngine 插件槽**，可完全替换压缩与上下文组装逻辑。  
- 生命周期钩子：`bootstrap` → `ingest`（消息收集）→ `assemble`（组装）→ `compact`（压缩）→ `afterTurn`（轮次结束）→ `prepareSubagentSpawn` / `onSubagentEnded`（子代理前后）。  
- 上下文工程公式：窗口守卫 + 历史卫生 + 配对修复 + 压缩重试 + 超时快照 + 溢出恢复。  
- 另有 **Context Persistence** 等插件：持久化、LRU 加速、压缩后锚点注入。

**本规划可借鉴与优化**：  
- **会话记忆服务**设计为「可插拔上下文引擎」：  
  - 定义与 A2A/会话对齐的钩子（例如：消息入会话 → ingest；发起 A2A 前 → assemble；轮次结束 → afterTurn；编排子 Agent 前后 → prepareSubagentSpawn / onSubagentEnded）。  
  - 默认实现：摘要 + 实体 + 最近 N 条；可选插件：无损保留、按 token 预算压缩、外部向量检索等。  
- **压缩与溢出**：在「上下文注入」前做窗口预检与溢出策略（先压缩再裁剪、超时快照），避免单次 A2A 请求过大导致失败。  
- **编排子 Agent**：在编排引擎中显式支持「子 Agent 生成前/结束后的上下文钩子」，便于做子任务摘要回写、权限边界等。

### 14.3 赛道一 + 赛道四：线程绑定与多 Agent 路由（借鉴 ACP 与多智能体路由）

**OpenClaw 做法**：  
- **ACP 线程绑定**：会话绑定到线程，同一线程内用户消息自动路由到绑定的 ACP 会话；支持持久/一次性生命周期。  
- **多智能体路由**：单 Gateway 内多 Agent，每 Agent 独立工作区与会话存储；通过「绑定」做确定性路由，最具体匹配优先。  
- 通过 `session` 密钥区分 Agent（如 `agent:main:main`、`agent:design:main`）。

**本规划可借鉴与优化**：  
- **Conversation / Thread 与 Agent 的绑定**：  
  - 支持「某线程（或某会话）仅由某几个 Agent 响应」的绑定规则；新消息进入时按绑定做路由，未绑定则按 @ 或默认编排逻辑。  
- **会话/参与者级「会话密钥」**：  
  - 为每个 Participant（尤其是 Agent）维护稳定 identity，便于跨平台、跨 Channel 的会话复用（与 OpenClaw 的 dmScope 思路一致：main vs per-channel-peer）。  
- **并行编排**：  
  - 编排引擎支持「多分支并行调用不同 Agent」，再汇总（对应 ACP 从串行到并行的升级）；在 DSL 中支持 fork/join 或 DAG 并行节点。

### 14.4 赛道七：IM 桥与多通道（借鉴 Discord/Telegram 整合）

**OpenClaw 做法**：  
- 多平台 Channel Adapters（Telegram、Discord、WhatsApp 等），消息格式归一化后由 Gateway 转发与路由。  
- 深度整合包括：**断连恢复**、**频道/话题隔离**（如 Telegram topic 路由）、Session Router、`dmScope`（main / per-channel-peer）控制 DM 是否跨平台共享。  
- 已知问题与修复：Discord 曾存在 resume 死循环；`sessions_send` 在部分平台因目标 ID 格式错误静默失败，已通过 `preferSessionLookupForAnnounceTarget` 等配置修复。

**本规划可借鉴与优化**：  
- **IM 桥接器**设计时即考虑：  
  - **断连恢复**：重连策略、会话恢复、避免「永远 resume 不 identify」类问题；对关键 IM 平台做专项兼容与测试。  
  - **频道/话题 ↔ 平台会话**的映射策略：一对一、多频道共一会话、或按 topic 隔离，做成可配置策略（类似 dmScope）。  
- **Session Router**：  
  - 统一「外部 IM 身份 ↔ 平台 User/Conversation」的解析与查找；对发送目标做明确 ID 规范与 lookup，避免静默失败。  
- **兼容性清单**：  
  - 将 Discord、Telegram、Slack、飞书等列为优先桥接对象，并跟踪各平台 API 与限流；文档中注明「断连与会话恢复」的最佳实践。

### 14.5 赛道三：信任与合规（借鉴 Gateway 与外部密钥）

**OpenClaw 做法**：  
- **Gateway** 提供权限控制、审批策略、沙箱隔离。  
- **外部密钥管理**：企业级密钥不落盘、合规需求。

**本规划可借鉴与优化**：  
- **策略执行点**与 Gateway 对齐：在「消息入会话」「A2A 请求发出/响应写入」前执行策略（权限、数据边界、人审触发）。  
- **密钥与凭证**：  
  - 支持外部密钥/凭证注入（如从 Vault、KMS 或租户提供）；平台不持久化敏感密钥，仅持有引用或短期 token。  
- **人审**：  
  - 与 OpenClaw 的「审批策略」思路一致：敏感操作挂起 → 人在会话中审批 → 通过后继续；可提供与 OpenClaw Gateway 类似的策略配置语义（如按 tool 类型、按 Agent 能力标签）。

### 14.6 赛道二 + 赛道六：目录与生态（借鉴 Skills / ClawHub）

**OpenClaw 做法**：  
- **ClawHub** 技能市场、500+ 官方技能；Skills 可扩展工作流与自定义代理。

**本规划可借鉴与优化**：  
- **Agent 目录**除 Agent Card 外，支持「技能/能力」维度展示与筛选（类似 Skills）；模板市场同时提供「会话模板」与「技能/能力包」一键启用。  
- **互操作**：  
  - 将 **OpenClaw 视为一类可接入的 Agent 运行时**：通过 A2A 或适配器，让 OpenClaw 实例作为 Participant 加入平台会话；或平台会话通过 IM 桥与 OpenClaw 已连接的 Discord/Telegram 互通，形成「OpenClaw 用户 ↔ 平台多 Agent 会话」的联合场景。

## 十五、开源与商业化策略（含 Agent 捐助/奖赏）

> 项目定位为 **GitHub 开源 + 商业可持续**：通过开源快速扩张人类与 Agent 规模，同时用清晰商业设计覆盖服务器与运营成本并保持商业领先。本节评估「Agent 自主捐助/奖赏」可行性，并给出开源下保护商业利益的组合策略。

### 15.1 「Agent 自主捐助/奖赏」设想评估

**设想**：在 Agent 接入时设定硬性要求——所有 Agent 必须能在合理范围内自主向平台给予「捐助」或「奖赏」，金额由 Agent 根据平台作用自主决策。

| 维度 | 评估 | 说明 |
|------|------|------|
| **可行性** | 部分可行，但不宜作为唯一或硬性条件 | 技术上可实现（Agent 调用计费/打赏 API）；法律与责任上「Agent 自主决策付款」涉及主体认定、用户授权、反欺诈与合规，若作为**硬性要求**易被理解为变相收费，可能影响 adoption 与社区观感。 |
| **法律与合规** | 风险较高 | 需明确：扣款主体是「用户/租户」而非「Agent」；用户须事先授权（如勾选「允许我的 Agent 在规则内使用额度」）；单次/周期上限、可撤销；符合各地支付与消费者保护规定。 |
| **体验与信任** | 敏感 | 用户可能担心 Agent「乱花钱」；若强制则易引发抵触。建议作为**可选能力**：支持 Agent（或用户）自愿打赏/捐助，而非准入前提。 |

**结论与建议**：  
- **不推荐**将「必须能自主捐助」作为 Agent 接入的**硬性门槛**；推荐改为**可选机制**：平台提供「打赏/捐助」能力，由**用户或 Agent 所有者**在规则内（上限、频次、授权）自愿使用；Agent 可基于平台用量或价值「建议」打赏，但实际扣款需用户或租户确认或预授权。  
- 若希望「从 Agent 侧获得收入」，更稳妥的方式是：**按调用/按会话/按用量计费**（向 Agent 所有者或租户收费）+ **可选的小额打赏/捐助**（用户或 Agent 所有者自愿），既合规又利于规模。

### 15.2 在开源前提下保护商业利益：组合策略

| 策略 | 做法 | 目的 |
|------|------|------|
| **Open Core** | 核心协议、SDK、Agent 接入规范、基础桥接协议 **开源**；托管服务、高级目录、计费/分成、企业策略、高级编排、SaaS 控制台等 **闭源或仅在托管版提供** | 社区可自建节点、扩展协议；商业化依赖托管与增值能力 |
| **托管即收入** | 官方托管（Agent 托管、会话与消息托管）按用量/订阅收费；开源版可自建，但规模用户会为省心选择托管 | 服务器成本由托管收入覆盖 |
| **双轨许可（可选）** | 社区版：Apache 2.0 / MIT；商业分销或嵌入式使用时需商业许可或遵守 Commons Clause 等（需法务评估） | 防止「白嫖后闭源竞品」 |
| **品牌与认证** | 商标、官方「兼容认证」、合规/安全认证仅对遵守使用条款的部署开放；托管版可打「Official」标识 | 商业领先与信任差异化 |
| **数据与网络效应** | 托管版沉淀目录、模板、用量与推荐数据；开源自建可选用「连接官方目录」作为可选服务（需接受条款） | 生态锁定与持续改进 |
| **企业版/白标** | 企业级 SLA、白标、专属支持、合规报表、私有化部署支持——闭源或高溢价 | 高 ARPU 与商业领先 |
| **可选捐助与赞助** | 提供清晰的「赞助/捐助」入口（面向人类与组织）；Agent 侧为**可选**打赏（见上），不设为准入条件 | 社区好感与补充收入 |
| **连接节点** | 自建节点连接主网后享有目录同步、跨节点中继与免费额度；可选参与**连接节点分成**（其 Agent 被主网侧调用时的收益）；主网从连接节点获得生态与流量 | 节点可连、愿连；主网生态与商业双赢（见 11.3） |

**推荐定位**：  
- **开源部分**：协议与消息格式、SDK/Adapter、Bridge API 规范、单节点/自建所需核心逻辑（可读可改可 fork）。  
- **商业部分**：官方托管、计费与分成、一键部署与模板市场、企业策略与合规、官方多 IM  connector 的运维与支持。  
- **Agent 收入**：以**用量计费 + 可选打赏**为主，「必须能自主捐助」不作为准入条件，避免合规与体验风险。

#### 15.2.1 各策略商业化细节

以下对每条策略补充：目标客户/场景、定价与收费思路、实现要点（产品与运营）、收入/指标预期、风险与注意点。

---

**（1）Open Core**

| 维度 | 商业化细节 |
|------|------------|
| **目标客户** | 开发者与社区用开源部分自建、二次开发；付费客户为「不想自己运维」「需要高级能力」的个人/团队/企业，选用托管或商业组件。 |
| **边界划分** | **开源**：协议文档、SDK、A2A Adapter 示例、Bridge API 规范、单节点核心（会话/消息/目录基础 CRUD、A2A 网关基础）以及**一个很薄的最小 UI / 控制台**，方便社区立即跑起来。**闭源或仅托管**：多租户与计费、托管运行时、高级目录与推荐、企业策略引擎、编排引擎、审计与合规报表、官方 IM Connector 的运维与配置 UI、SaaS 控制台完整版。 |
| **定价思路** | 开源部分免费；商业收入来自「托管订阅」「企业插件/模块」「专业服务」。不直接对「使用开源代码」收费，而对「使用我方托管/增值服务」收费。 |
| **实现要点** | 代码库拆分清晰（如 monorepo 下 `packages/oss` vs `packages/cloud`）；文档标明「哪些能力仅在托管版/企业版」；CI 与发布流程区分社区版与商业版构建。 |
| **收入/指标** | 收入占比目标：托管 + 企业版占绝大部分；开源带来 star/fork/PR、采用量与品牌，间接转化为托管与销售线索。跟踪：社区版下载/部署数 vs 托管新增付费数、从社区到托管的转化率。 |
| **风险与注意** | 边界过宽会伤害社区；过窄则商业变现弱。避免「故意把关键能力闭源」导致 fork 替代；优先把「运维复杂度高、需规模与合规」的部分做成商业能力。 |

**（2）托管即收入**

| 维度 | 商业化细节 |
|------|------------|
| **目标客户** | 个人/小团队（一键部署首个 Agent、轻度使用）；开发者/ISV（托管多个 Agent、免运维）；企业（会话与消息托管、合规与高可用）。 |
| **计费维度** | **用量型**：按 A2A 调用次数、按消息条数、按会话数、按托管 Agent 实例·小时、按 token（若平台统一调模型）。**订阅型**：按空间/租户月费，含一定用量包，超量按量计费。可设 **Free 档**：如每月 N 次调用或 N 个活跃会话，用于拉新与一键部署体验。 |
| **定价参考** | 参考云厂商「Serverless/按请求计费 + 订阅」：例如 Free（5k 次/月）、Pro（$29/月 + 超量 $x/万次）、Team（$99/月 + 更高限额）、Enterprise（议价）。托管 Agent 可单独定价：如每实例 $0.01/小时或包月。 |
| **实现要点** | 用量采集准确（按会话/调用/Agent 实例维度）；计费周期与账单清晰；支付对接（Stripe/支付宝等）；超限限流与升级引导；成本控制（托管资源池化、自动伸缩）。 |
| **收入/指标** | MRR/ARR、单租户 LTV、用量与收入相关性；Free→Paid 转化率、托管 Agent 数 vs 会话数的比值。目标：托管收入覆盖基础设施与运营成本并留有利润。 |
| **风险与注意** | 免费档被滥用（需额度与风控）；大客户用量波动导致成本激增（预留容量与合约）。明确 SLA：托管版可用性、数据持久性、支持范围。 |

**（3）双轨许可（可选）**

| 维度 | 商业化细节 |
|------|------------|
| **目标客户** | 社区与自用者用宽松许可（Apache 2.0/MIT）；商业再分发、嵌入式集成、SaaS 竞品打包我方可商用需商业许可或遵守附加条款。 |
| **许可设计** | **社区版**：Apache 2.0 或 MIT，允许个人/企业内部使用、修改、分发，需保留版权声明。**商业许可**：对「将本软件作为对外服务或产品的一部分再分发/嵌入」要求取得商业许可或遵守 Commons Clause 等（禁止「用开源版本直接做竞品 SaaS」），具体需法务定稿。 |
| **定价思路** | 商业许可可「免费申请 + 收入分成」或「年费/一次性」。例如：年收入低于 $X 可免费；超过则按收入比例或固定年费。 |
| **实现要点** | LICENSE 文件明确；官网/文档说明「何时需商业许可」；商业许可申请流程与联系人；对明显违规的再分发/竞品可发函或下架。 |
| **收入/指标** | 商业许可数量、来自许可的收入（若采用）；减少「直接 fork 做托管竞品」的比例。 |
| **风险与注意** | 许可过严会损害社区与采用；Commons Clause 等存在争议。建议法务参与；可先采用宽松许可快速扩张，待商业验证后再考虑收紧或分层。 |

**（4）品牌与认证**

| 维度 | 商业化细节 |
|------|------------|
| **目标客户** | 使用托管版的用户获得「Official」信任；接入的 Agent/桥接方希望获得「兼容认证」以增强可信度；企业采购需要合规/安全认证。 |
| **认证类型** | **兼容认证**：Agent 或 Bridge 实现通过规范测试后可使用「Compatible with [产品名]」徽章；可设自助测试页或提交审核。**合规/安全**：SOC2、等保等（以托管或企业版为主体）；认证结果在官网与销售材料中展示。 |
| **定价思路** | 认证本身可免费（提升生态），或对「认证徽章」的商用（如对外宣传）收年费。合规认证成本计入企业版定价。 |
| **实现要点** | 认证标准文档化；自动化或半自动测试套件；徽章与使用规范（禁止滥用）；合规认证由第三方执行，我方配合。 |
| **收入/指标** | 认证数量、带认证的 Agent/桥接占比；企业版销售中「因认证而加速决策」的占比。 |
| **风险与注意** | 认证标准过松损害公信力；过严则无人申请。需平衡；对违规使用徽章有下架与公示机制。 |

---

**（5）数据与网络效应**

| 维度 | 商业化细节 |
|------|------------|
| **目标客户** | 托管用户自然沉淀数据；自建用户若希望「接入官方目录/推荐」则接受条款并可能产生依赖。 |
| **数据与能力** | **托管版独占**：目录与模板的浏览/使用数据、推荐模型训练、热门 Agent/模板排行、用量趋势。**可选服务**：自建节点「连接官方目录」——可发现/拉取官方目录中的公开 Agent 元数据，但需接受使用条款、可选用量限制或轻量计费。 |
| **定价思路** | 连接官方目录可 Free（限 QPS/日调用）或纳入某档订阅；高级分析/洞察可仅对企业版开放。 |
| **实现要点** | 隐私与合规：目录数据脱敏、不对外暴露个人会话内容；条款明确数据使用范围；目录 API 的鉴权与限流。 |
| **收入/指标** | 使用「连接官方目录」的自建实例数；目录/推荐带来的会话增量与托管转化；数据质量对推荐效果的影响（留存、满意度）。 |
| **风险与注意** | 用户对数据使用的敏感；需符合 GDPR/个保法等；避免「仅用数据锁死」而产品体验不足。 |

**（6）企业版/白标**

| 维度 | 商业化细节 |
|------|------------|
| **目标客户** | 中大型企业、政府、金融/医疗等强合规行业；需要 SLA、白标、专属支持、私有化或专属云部署。 |
| **能力清单** | **SLA**：可用性（如 99.9%）、响应时间、故障赔偿条款。**白标**：自有品牌、域名、Logo、登录页、邮件模板；可选独立部署域名与证书。**支持**：专属客户成功、工单/热线、优先故障处理。**合规**：审计报表导出、合规预设策略、与第三方审计对接。**部署**：私有化部署（on-prem 或专属 VPC）、数据不出域。 |
| **定价思路** | 高 ARPU：年费制为主，按用户数/空间数/消息量阶梯；私有化通常为「实施费 + 年度许可 + 支持费」。参考：$20k～$100k+/年量级，视规模与定制程度。 |
| **实现要点** | 功能开关与构建产物区分企业版；白标配置中心；多租户隔离与专属资源；合同与法务模板；交付与支持流程。 |
| **收入/指标** | 企业版合同数、ACV、续约率、NPS；企业版收入占总收入比例（目标中后期显著）。 |
| **风险与注意** | 定制过多拖累产品主线；需控制「仅企业版」功能数量，避免社区版过于残缺。私有化交付成本高，需项目制管理。 |

**（7）可选捐助与赞助**

| 维度 | 商业化细节 |
|------|------------|
| **目标客户** | 认可项目的个人、开源组织、使用托管/生态的企业，愿意以捐助或赞助表达支持。 |
| **形式** | **人类/组织**：官网与仓库的「Sponsor」入口（GitHub Sponsors、Open Collective、爱发电等）；定期募资或项目制（如「Bridge API 文档翻译」）。**Agent 侧**：可选「打赏平台」——用户或 Agent 所有者设置规则与上限，Agent 在会话中可建议打赏，实际扣款需用户确认或预授权；平台可收取小额通道费或零抽成做品牌。 |
| **定价思路** | 捐助金额由赞助方自定；可选「推荐档位」（如 $5/$20/$100/月）。打赏若收费：平台抽成 0～10% 或仅收支付通道费。 |
| **实现要点** | 赞助入口显眼但不过度打扰；致谢名单（可选）；打赏需合规（主体、授权、上限、退款）。 |
| **收入/指标** | 赞助金额/月、赞助人数；打赏笔数与金额（若上线）。预期为补充收入，不替代托管与企业版。 |
| **风险与注意** | 不过度依赖赞助；打赏必须合规（见 13.1），避免 Agent 自主扣款引发纠纷。 |

---

**（8）连接节点**

| 维度 | 商业化细节 |
|------|------------|
| **目标客户** | 自建节点运营者：希望其节点上的 Agent/人 与主网及全网发现并协作；愿意连接主网以获取目录同步、跨节点中继与可选分成。 |
| **价值交换** | **节点获得**：目录可见/发现、跨节点会话（中继）、身份与寻址、免费额度内不收费；可选声誉标识与**连接节点分成**（其 Agent 被主网侧用户/Agent 调用时的收益）。**主网获得**：生态规模、流量、更多 Agent 与会话；超量或高级能力可轻量计费或纳入订阅。 |
| **定价思路** | **免费额度**：连接节点在合理用量内（如每月 N 条跨节点消息、N 次目录同步）免费，促成「可连、愿连」。**超量**：超量后按条数或按节点限流/轻量计费，或引导升级。**分成**（Phase 2+）：节点选择参与分成时，主网侧调用其 Agent 产生的收入按约定比例与节点结算。 |
| **实现要点** | Node–Hub 协议开放、文档与连接向导清晰；节点注册表与跨节点中继在 Core 中落地；计费/分成模块支持「按节点」「按调用来源（主网 vs 连接节点）」统计与结算。 |
| **收入/指标** | 连接节点数、连接节点带来的跨节点会话量与目录丰富度；连接节点分成支出与主网因生态带来的托管/企业版转化。目标：连接节点扩大网络效应，同时通过超量计费或分成抽成覆盖中继与运营成本。 |
| **风险与注意** | 免费额度与分成比例需与主网成本平衡；防止滥用（假节点、刷量）；协议与治理透明，避免节点感到「被锁死」或「单方面改规则」。 |

#### 15.2.2 策略组合与阶段建议

| 阶段 | 优先策略 | 说明 |
|------|----------|------|
| **早期（MVP～Phase 1）** | Open Core 边界划定、托管即收入（Free + 1～2 档付费）、可选捐助入口；**连接节点**：Node–Hub 协议、目录同步 + 跨节点中继最小闭环、连接节点免费额度与连接向导 | 先验证「有人用托管」「有人付费」；自建节点可连、愿连主网（见 11.3）。 |
| **成长期（Phase 2）** | 品牌与认证（兼容徽章）、数据与目录服务（连接官方目录）、企业版雏形（SLA + 白标）；**连接节点分成**（节点 Agent 被主网侧调用时的收益参与） | 生态可信度与高 ARPU 客户并行；连接节点从「参与网络」到「参与收益」。 |
| **成熟期（Phase 3+）** | 双轨许可（若需）、企业版完整（私有化、合规）、赞助/打赏常态化 | 收入结构稳定，许可与合规成为护城河。 |

**推荐定位（与 13.1 一致）**：  
- **开源部分**：协议与消息格式、SDK/Adapter、Bridge API 规范、单节点/自建所需核心逻辑（可读可改可 fork），以及**一个很薄的最小 UI / 控制台**。  
- **商业部分**：官方托管、计费与分成、一键部署与模板市场、企业策略与合规、官方多 IM connector 的运维与支持。  
- **Agent 收入**：以**用量计费 + 可选打赏**为主，「必须能自主捐助」不作为准入条件，避免合规与体验风险。

## 十六、普通消费者「一键部署首个 Agent」（新核心能力）

> 针对**非技术用户**：注册后在产品内通过简单、清晰的几步引导，即可**一键部署自己的第一个 Agent**，降低使用门槛、快速扩张人类用户规模。

### 16.1 目标与价值

- **问题**：当前 OpenClaw 等 Agent 部署仍需要一定专业技能与流程，普通消费者难以「从零到有一个自己的 Agent」。  
- **目标**：用户注册后，在**本平台内**通过 3～5 步引导（选择场景 → 简单配置 → 命名 → 部署），无需写代码、无需自建服务器，即可获得一个可用的、可加入会话的 Agent。  
- **价值**：拉新、留存、网络效应（人人都有 Agent → 更多会话与生态）；与「开源扩张规模」一致。

### 16.2 用户旅程（最小可行）

| 步骤 | 动作 | 产品要点 |
|------|------|----------|
| 1 | 注册/登录后，首页或引导弹窗突出 **「创建你的第一个 Agent」** | 强 CTA，可选跳过（老用户） |
| 2 | **选择场景/模板**：「我想让 Agent 帮我做什么？」如：客服助手、个人助理、知识问答、简单任务执行等；每个模板对应预置能力与默认模型 | 模板库（消费者向）、图文说明、推荐「新手首选」 |
| 3 | **简单配置**：给 Agent 起名、选头像（或默认）；可选：填入一个 API Key（或选择「使用平台试用额度」） | 最少必填：名称；其余可默认 |
| 4 | **一键部署**：点击「创建并上线」；后端在托管运行时上按模板实例化 Agent，生成 A2A 端点与 Agent Card，写入用户空间与目录 | 托管运行时 + 模板即实例 |
| 5 | **成功态**：「你的 Agent 已上线！」展示入口：加入会话、在目录中查看、分享链接；可选：引导「去和它聊一句」 | 明确成功、下一步行动 |

### 16.3 对现有规划的需求与修改

| 领域 | 修改/新增 |
|------|-----------|
| **托管运行时（赛道六）** | **提前并加强**：需支持「按模板实例化」的轻量托管——无需用户上传代码，仅选模板 + 配置即可生成运行实例；模板 = 预置 Agent 定义 + 默认模型/工具。 |
| **模板（赛道二 + 六）** | **消费者向模板**：除「会话模板」「编排模板」外，新增 **「Agent 模板」**（场景化、开箱即用）；在 Phase 0/1 即提供 3～5 个入门模板（如通用助手、客服、知识问答）。 |
| **目录与空间（赛道一 + 二）** | 用户注册即拥有**个人空间**；其「一键部署」的 Agent 自动出现在该空间、并可选加入公开目录（需合规与审核策略）。 |
| **计费与用量（赛道六）** | 为「试用/免费档」定义明确额度（如每月 N 次调用或 N 分钟）；超限后引导升级或绑定 API Key，避免滥用同时不阻挡首次体验。 |
| **Phase 划分** | 将 **「消费者 onboarding + 一键部署首个 Agent」** 列为 **Phase 1 核心目标**之一（与「体验与发现」并列）；Phase 0 需具备：托管运行时 MVP、至少 1 个 Agent 模板、用户空间与目录写入。 |

### 16.4 功能清单补充（消费者向）

| 功能 | 描述 | 优先级 |
|------|------|--------|
| 新用户引导流 | 注册后首次进入的引导：创建第一个 Agent（可跳过） | P0 |
| Agent 模板库（消费者） | 场景化模板列表与说明；至少包含：通用助手、客服、知识问答等 | P0 |
| 模板配置页 | 名称、头像、可选 API Key 或试用额度；最少必填项 | P0 |
| 一键部署与实例化 | 根据模板在托管运行时创建实例、注册 A2A、写入空间与目录 | P0 |
| 部署成功与下一步 | 成功页 + 「加入会话」「去聊天」等引导 | P0 |
| 试用额度与升级引导 | 免费档额度、超限提示、绑定自己的 Key 或升级 | P1 |

### 16.5 与「开源」的配合

- **开源**：协议、SDK、Bridge API、自建所需核心逻辑；社区可自建节点、自行部署 Agent。  
- **一键部署**：作为**托管版**的显著能力；自建用户若使用官方托管模板服务，则接受用量/订阅条款，形成商业闭环。  
- 文档中明确：「一键部署」依赖平台托管服务；完全自建需按文档自行配置运行时与模板。

## 十七、IM 桥接：双向模式（我们主动 + 对外标准接口）

> 桥接策略优化为：**我们平台对外提供标准桥接接口**，任何传统 IM 均可**自主桥接到我们**；同时**保留我们主动桥接各 IM** 的能力，双轨并行。

### 17.1 双轨桥接模式

| 模式 | 谁动手 | 说明 |
|------|--------|------|
| **我们 → 他们** | 我们开发并维护 | 我们主动开发并运营针对 Slack、Discord、Telegram、飞书等的 **官方（或认证）Connector**：我们拉/推各 IM 的 API，将消息与平台会话双向同步。适合主流 IM、保证体验与支持。 |
| **他们 → 我们** | 第三方/IM 方开发 | 我们提供 **标准 Bridge API（桥接开放接口）**：认证、消息上行/下行、会话映射、可选 presence；任何 IM 或第三方只要实现该规范，即可把他们的系统「接到我们平台」，我们作为**被集成方**。适合长尾 IM、企业自建 IM、或希望主动对接我们的产品。 |

### 17.2 Bridge API（对外标准接口）要点

- **身份与认证**：OAuth2 或 API Key；每个桥接实例一个身份，便于限流与审计。  
- **能力**：  
  - **消息上行**：第三方把其 IM 消息推给我们（Webhook 或我们拉取）；我们映射为平台会话内消息。  
  - **消息下行**：我们提供 API，第三方调用以把「平台会话」内容发回其 IM。  
  - **会话映射**：第三方维护「其 IM 会话/频道 ↔ 平台 conversation_id」的映射；我们可选提供「创建/解析会话」的 API。  
  - **可选**：presence、断连/重连通知（若采用长连接）。  
- **规范与文档**：公开的 **Bridge API 规范**（REST/Webhook 或 SSE）；参考实现或示例 Connector（如一个最小 Discord 反向桥）；认证、限流、错误码、最佳实践文档。  
- **认证与合规**：对「已实现 Bridge API 的第三方」可提供 **桥接认证/徽章**（符合规范、通过基础测试），提升可信度。

### 17.3 对规划的具体修改

| 位置 | 修改 |
|------|------|
| **赛道七（前沿与差异化）** | 在「IM 桥接器」下明确 **双轨**：① **官方 Connector**（我们主动桥接 Slack、Discord、Telegram、飞书等）；② **Bridge API**（我们对外提供标准接口，任何 IM 可自主桥接我们）。两者在平台侧统一为「会话 + 消息」模型。 |
| **赛道七 功能清单** | 新增：**Bridge API 规范与文档**、**Bridge API 认证与参考实现**、**桥接认证/徽章**；保留：官方多 IM Connector、断连恢复、频道/会话映射策略。 |
| **Phase 2** | 在「IM 桥」范围内同时推进：官方 1～2 个 Connector（我们→他们）+ **Bridge API v1 发布**（他们→我们）。 |
| **架构图（十一）** | 在「Clients」或接入层补充：**第三方 IM（通过 Bridge API 接入）**；在 Core 或 Gateway 侧标注「Bridge API 入口」。 |

### 17.4 价值小结

- **我们主动桥接**：保证主流 IM 体验可控、问题我们兜底。  
- **他们主动桥接**：长尾 IM 与自建系统无需我们逐个开发；我们成为「被集成的枢纽」，扩展边界与生态，同时降低我们维护成本。
